## 五、技术基建

### 5.1 lib/ai-capabilities.ts 能力注册表（V1 核心基建）

统一管理所有 AI 能力入口。结构草案：

```
export type AiCapabilityName =  | 'matting'              // 抠图  | 'outpaint'             // 扩图  | 'upscale'              // 高清放大  | 'text2img'             // 文生图  | 'img2img'              // 图生图  | 'text-rewrite'         // AI 文案改写  | 'background-replace';  // 换背景
export interface AiCapability {  name: AiCapabilityName;  label: string;           // 运营看得懂的中文名  status: 'stub' | 'live';  description: string;  run?: (input: unknown) => Promise<unknown>;}
export const AI_CAPABILITIES: Record<AiCapabilityName, AiCapability> = { /* ... */ };
export async function callAiCapability(name: AiCapabilityName, input: unknown) {  const cap = AI_CAPABILITIES[name];  if (cap.status === 'stub') {    // 返回 mock 或抛出统一的"功能建设中"标记    throw new AiCapabilityStubError(cap.label);  }  return cap.run!(input);}
```

**好处**：

- 所有未接入能力**集中登记一份**，后期接 API 时一眼看清待办
- UI 层统一 catch `AiCapabilityStubError` 弹 toast
- 能力名称、文案运营侧口径在一处管理，不散落到组件里

### 5.2 Schema 层面

**V1.0 不动 schema**。`templates` 和 `psd_layers` 现状够用。

- 场景分类：用 TS 常量 `const SCENES = [...]`，不建 `template_categories` 表
- 尺寸白名单：同样 TS 常量
- 等 V1.1 真有多业务线/大量场景时再考虑建字典表

### 5.3 AI 能力的"假数据"来源

Stub 阶段需要假数据让 UI 看着真实：

- 抠图：返回原图 + 一个预设的透明 PNG（放 /public/mock/matted.png）
- 扩图：返回原图 + 一张预生成的扩图 mock
- 文案改写：返回固定 3 条 mock 文案

建议在 `public/mock/` 存这些素材。
