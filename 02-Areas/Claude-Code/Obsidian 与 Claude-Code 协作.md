## 一、Harness 基础配件

不管有没有代码，这些是你 harness 的组成部分。首次接入时按需配置，后续在转向循环中迭代。

### CLAUDE.md

项目级持久指令。让 Claude 生成草稿，你负责审查精简（< 200 行）：

```
读一下这个项目的代码结构、技术栈、开发规范，生成 CLAUDE.md
```

或用交互式初始化（会主动问你要设置哪些内容）：

```bash
CLAUDE_CODE_NEW_INIT=1 claude
/init
```

层级体系、写作规范和迭代方法见 [[Harness构建指南]]。

### Skills（按需加载的知识）

CLAUDE.md 只放普遍适用的内容。特定场景的知识放 `.claude/skills/`，Claude 在相关任务中自动加载：

```
帮我在 .claude/skills/testing/ 里创建 SKILL.md，把我们项目的测试规范写进去：用 Vitest、优先单元测试、mock 外部依赖不 mock 数据库
```

用 `/plugin` 浏览社区插件市场，插件会打包 Skills、Hooks 和 MCP 服务。

### Hooks（确定性执行）

CLAUDE.md 规则约 80% 遵守率，Hooks 是 100%。必须每次执行的事用 Hook。

四种类型：`command`（shell 命令）、`http`（POST 到 URL）、`prompt`（单轮 LLM 评估）、`agent`（多轮子代理）。

常用示例——改完文件自动格式化：

```json
{
  "hooks": {
    "PostToolUse": [{
      "matcher": "Edit|Write",
      "hooks": [{ "type": "command", "command": "npx prettier --write $FILE" }]
    }]
  }
}
```

用 `/hooks` 查看当前配置。

### Subagents（隔离的专用代理）

在 `.claude/agents/` 下定义，可指定模型、工具集和隔离方式：

```
帮我在 .claude/agents/ 里创建一个 reviewer 子代理，用 Opus 模型，只允许读取文件和搜索，专门做 code review
```

用 `/agents` 查看可用的子代理。加 `isolation: worktree` 让它在独立 git worktree 中工作。

## 二、无代码：从想法到第一行代码

适用于：脑子里有个想法，还没创建代码仓库。

### 2.1 在 Obsidian 建项目笔记

想法开始占用超过一次对话的精力时就该建。不需要等代码出现：

```
帮我在 01-Projects 里建个项目笔记，项目叫 XX，目前还在想法阶段。我想做的是……（简述目标和动机）
```

笔记应包含：

- 这东西是什么、解决什么问题
- 初步技术方向
- 还没想清楚的问题

然后把项目记录到 vault 根目录：

```
在 ~/Brain/CLAUDE.md 的"当前项目"里加上这个项目
```

### 2.2 调研与方案对比

让 Claude 帮你做技术选型，结论沉淀到笔记里：

```
我想做一个 XX，帮我对比一下 A 方案和 B 方案的优劣，考虑性能、开发效率和社区生态
```

调研完毕后：

```
把刚才的方案对比结论更新到 ~/Brain/01-Projects/xx.md 里
```

### 2.3 规划架构

方向确定后，进 Plan Mode（`Shift+Tab`）让 Claude 出架构方案：

```
基于我们选定的方案，帮我规划项目结构、核心模块划分、技术栈和第一步要做什么
```

- 按 `Ctrl+G` 在编辑器里直接修改方案
- 满意后把架构决策同步到项目笔记

### 2.4 初始化项目骨架

切回执行模式（`Shift+Tab`），让 Claude 创建仓库和基础结构：

```
按照刚才的架构方案，帮我初始化项目：mkdir + git init + 创建基础目录结构和配置文件
```

项目创建后，立刻搭建 harness：

```
读一下项目结构，生成 CLAUDE.md，包含构建命令、代码风格和架构说明
```

根据需要配置 Skills、Hooks（见第一节）。至此进入正常开发循环（→ 第四节）。

## 三、有代码：接入既有仓库

适用于：代码仓库已经存在，首次用 Claude Code 接入。

### 3.1 让 Claude 读懂项目

```bash
cd ~/path/to/project
claude
```

先让它探索代码库，不急着改：

```
读一下整个项目的代码结构、技术栈和开发规范，给我一个概览
```

### 3.2 生成 CLAUDE.md

```
基于你刚才的分析，生成 CLAUDE.md，包含构建命令、代码风格、核心架构和不能动的文件
```

生成后**必须审查精简**——对每一行问"去掉它 Claude 会犯错吗？"不会就删。

### 3.3 在 Obsidian 建项目笔记

```
读一下代码结构，帮我在 ~/Brain/01-Projects 里创建项目笔记，包含项目目标、技术栈、核心架构和已知限制
```

然后登记到 vault：

```
在 ~/Brain/CLAUDE.md 的"当前项目"里加上这个项目的路径
```

### 3.4 补全 harness

根据项目情况配置 Skills、Hooks、Subagents（见第一节）。经验法则：

- 先只搭 CLAUDE.md，跑几轮对话
- 发现 Claude 反复犯同一个错 → 补规则或升级 Hook
- 发现某类任务需要大段专业知识 → 拆成 Skill

至此进入正常开发循环（→ 第四节）。

## 四、日常开发循环

不管项目从哪条路径进来，日常工作流是相同的。

### 4.1 恢复上下文

每次新会话第一步：

```
看一下 git status、最近的 git log 和 TODO，告诉我今天该干什么
```

如果 Obsidian 里有相关笔记：

```
看一下 @~/Brain/01-Projects/my-project.md 里记录的待办事项，结合 git status 告诉我从哪里继续
```

### 4.2 研究与规划

非简单任务（3 步以上或涉及架构决策），先进 Plan Mode（`Shift+Tab`）：

```
帮我规划一下怎么实现用户认证功能，需要改哪些文件、分几步做、有什么风险
```

- 按 `Ctrl+G` 在编辑器里修改方案
- 满意后说"按这个方案执行"
- 方向不对就**立刻停下来重新规划**

**什么时候跳过规划：** 如果你能用一句话描述要改的 diff，直接动手。

给 Claude 看参考实现比让它凭空设计效果好得多：

```
参考 @src/auth/login.ts 的实现模式，帮我给注册流程写一个类似的
```

### 4.3 执行与验证

切回执行模式（`Shift+Tab`），**给 Claude 验证手段**——这是提升产出质量最有效的一件事：

```
改完后跑 npm test 确认没有破坏现有测试，再跑 npx tsc --noEmit 确认类型正确
```

UI 相关改动可以用 Claude in Chrome 扩展让它自己截图验证。

### 4.4 上下文管理

长对话中 Claude 的表现会随上下文膨胀而下降：

| 操作 | 时机 | 怎么做 |
|------|------|--------|
| 压缩上下文 | 上下文用到 60% 左右 | `/compact 保留 API 变更相关的内容` |
| 查看上下文占用 | 不确定还剩多少空间 | `/context` |
| 清空重开 | 切换到不相关的任务 | `/clear` |
| 侧边提问 | 小问题，不想污染主对话 | `/btw 这个函数的参数类型是什么` |
| 回退到检查点 | Claude 走偏了 | `Esc+Esc` 或 `/rewind` |
| 委托子代理 | 需要大量探索不想撑满上下文 | "用子代理去调查一下 XX 的实现方式" |

**反模式**：混合不相关任务（厨房水槽会话）、反复纠正超过 2 次（`/clear` 重来）、CLAUDE.md 太长导致规则被忽略。

### 4.5 勤提交

每完成一个有意义的步骤就提交：

```
帮我提交当前的改动，commit message 写清楚做了什么
```

用 `claude --worktree feature-name` 在独立 worktree 中工作，主分支不受影响。

### 4.6 随手记录

遇到值得记的东西，五分钟的记录能省下一次对话 20 分钟的上下文铺垫：

```
把刚才这个踩坑经验帮我写到 ~/Brain/01-Projects/my-project.md 的"已知问题"里
```

架构级变化（技术栈变更、模块拆分、新的硬约束）**立刻同步**项目笔记，否则下次对话会基于过时信息给方案。日常迭代（加组件、修 bug）留给收工时统一更新。

## 五、收工：沉淀与迭代

### 5.1 收工总结

```
看一下今天的 git log，帮我在 ~/Brain/05-Daily 里写个今日总结，文件名用今天的日期
```

### 5.2 更新 Harness

Claude 犯的每一个错都应该变成一条规则：

```
把刚才这个教训加到项目的 CLAUDE.md 里，防止以后再犯
```

同样重要的是定期清理：

```
读一下项目的 CLAUDE.md，有哪些规则是从代码本身就能推断出来的？帮我精简掉
```

### 5.3 同步项目笔记

```
看一下最近一周的 git log，更新 ~/Brain/01-Projects/my-project.md 里的架构说明和进度
```

### 5.4 归档 Inbox

```
看一下 ~/Brain/00-Inbox 里的笔记，帮我分类移到合适的文件夹
```

## 权限模式速查

按 `Shift+Tab` 循环切换前三种，其余通过 CLI 参数或配置启用：

| 模式 | 自动放行 | 适用场景 | 启用方式 |
|------|---------|---------|---------|
| Default | 仅读取 | 入门、敏感操作 | Shift+Tab |
| Auto-accept edits | 读取 + 文件编辑 | 日常迭代 | Shift+Tab |
| Plan mode | 仅读取（禁止编辑） | 研究探索 | Shift+Tab |
| Auto mode | 全部，后台分类器审查 | 长任务 | Team/Enterprise |
| bypassPermissions | 除受保护路径外全部 | 隔离容器/VM | 配置文件 |

日常推荐：简单任务用 Auto-accept edits，复杂任务先 Plan mode 出方案再切回 Default 执行。

| 操作 | 按键 / 命令 |

| 在编辑器里改方案 | `Ctrl+G` |

| 压缩上下文 | `/compact <保留什么>` |
| 清空上下文 | `/clear` |
