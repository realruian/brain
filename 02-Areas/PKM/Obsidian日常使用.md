# Obsidian 日常使用

## 笔记怎么写

不用追求完美，几个关键词比空白页有用一万倍。格式参考：

```markdown
---
tags:
  - 某个标签
date: 2026-04-14
---

# 标题

正文内容
```

判断标准：**这件事将来可能还会遇到吗？** 是就记，不是就别记。

## 笔记放哪里

| 内容类型               | 放哪里            | 举例                          |
| ------------------ | -------------- | --------------------------- |
| 随手一记，还没想好分类        | `00-Inbox`     | "今天看到一个不错的 Canvas API 用法"   |
| 跟某个项目直接相关，有明确目标和终点 | `01-Projects`  | "design-studio 登录功能开发"      |
| 长期持续关注的领域，没有终点     | `02-Areas`     | "前端性能优化"、"AI 工具链"           |
| 参考资料、教程、备查知识       | `03-Resources` | "Next.js Server Actions 用法" |
| 完成或不再需要            | `04-Archive`   | 已结束项目的笔记                    |
| 每日记录               | `05-Daily`     | "2026-04-14"                |

**Projects vs Areas 的区别**：Projects 有明确的截止条件（"完成 XX 功能"），Areas 是持续维护的责任（"保持代码质量"）。架构决策、技术备忘这类没有终点的知识应放 Areas 或 Resources，不是 Projects。

**拿不准放哪？先扔 `00-Inbox`。** 定期让 Claudian 帮你归档：

```
看一下 00-Inbox 里的笔记，帮我分类移到合适的文件夹
```

## 笔记怎么连接

### 三种组织方式的适用场景

| 方式 | 用途 | 举例 |
|------|------|------|
| **文件夹** | 宏观分类，决定"放在哪里" | `02-Areas/前端/` |
| **Tags** | 跨文件夹的横向属性 | `#参考`、`#待整理` |
| **Wikilink** | 两篇笔记之间有内容关联 | `[[React Hooks 笔记]]` |

三者不互斥，但不要重复：同一层意思只用一种表达。

### Wikilink 常用写法

```markdown
[[笔记名]]               # 基础链接
[[笔记名|显示文字]]       # 自定义链接文字
[[笔记名#某个标题]]        # 链接到具体章节
[[笔记名#^块ID]]          # 链接到具体段落（在目标段落末尾加 ^ 生成 ID）
```

> **注意**：`[[笔记名]]` 是单向链接。被链接的笔记会在「反向链接」面板（Backlinks）自动收到通知，但不会自动回链。真正的双向需要两边各写一个链接。

### 反向链接面板（Backlinks）

打开右侧面板 → Backlinks，可以看到：
- **已链接**：明确写了 `[[当前笔记]]` 的笔记
- **未链接提及**：提到了当前笔记名但还没加链接的笔记 — 这里经常藏着意想不到的关联

### Tags 使用规范

```markdown
tags:
  - PKM              # 一级标签
  - area/frontend    # 嵌套标签，用于分层分类
```

搜索语法：`tag:#PKM`，或在搜索栏用 `tag:PKM` 过滤。

**什么时候用 tag 而不是 wikilink**：tag 适合描述笔记的「属性」（类型、状态），wikilink 适合描述「关系」（这篇笔记和另一篇有内容关联）。

## Daily Notes 怎么用

`05-Daily` 下每天一篇，文件名格式 `YYYY-MM-DD`。推荐内容：

```markdown
---
tags:
  - daily
date: 2026-04-14
---

# 2026-04-14

## 今日工作
-

## 想法 / 收获
-

## 待跟进
-
```

建议用**模板插件**（核心插件 → Templates）或 Templater 自动生成，不要每天手动写。

## Dataview：把笔记变成数据库

Dataview 是社区插件，可以用 SQL 风格查询你的笔记。几个常用场景：

````markdown
```dataview
TABLE date, tags FROM "00-Inbox"
SORT date DESC
```
````

→ 列出 Inbox 里的所有笔记及日期，帮助定期清理。

````markdown
```dataview
LIST FROM "05-Daily"
WHERE date >= date(today) - dur(7 days)
```
````

→ 显示最近 7 天的 Daily Notes，方便生成周总结。

## 常用 Claudian 指令

Claudian 是把 AI Agent（Claude Code、Codex、Gemini CLI 等）嵌入 Obsidian 侧边栏的插件，支持直接读写 Vault 文件、执行多步任务。用 `@文件名` 引用笔记上下文，`/` 触发快捷指令。

| 场景 | 指令 |
|------|------|
| 整理 Inbox | 看一下 00-Inbox 里的笔记，帮我分类移到合适的文件夹 |
| 搜索笔记 | 我之前记过关于 XX 的内容，帮我找一下 |
| 生成周总结 | 看一下 05-Daily 里这周的笔记，帮我生成周总结 |
| 创建参考笔记 | 帮我在 03-Resources 里创建一篇关于 XX 的参考笔记 |
| 阅读代码写笔记 | 读一下 ~/dev/xxx 的代码结构，帮我创建一篇项目笔记 |
| 连接相关笔记 | 看一下这篇笔记，帮我找 Vault 里有关联的笔记并加上链接 |
| 清理冗余 | 检查 Vault 里有没有重复内容或可以合并的笔记 |

## 核心原则

1. **别让记笔记变成负担** — 能交给 Claudian 的就交给它
2. **先扔 Inbox，每周清一次** — 别纠结分类，但要定期处理
3. **只记将来有用的** — 不是所有事都值得记录
4. **笔记不求完美** — 先写下来，以后再补充
5. **链接优于整理** — 与其花时间移动文件，不如多建 `[[链接]]`
