# **AI Coding 怎么上手？**

> 问卷显示 66.67% 的设计师反馈"AI 生成的代码看不懂、改不动"，这是目前最普遍的困惑。但 AI Coding 并不要求设计师成为开发，不同交付类型的难度差异很大，可以从交付产品 Demo 开始逐步跑通。

# **研发不愿意配合怎么办，**担心交付的代码研发不用怎么办，出了bug谁负责？

> 这是问卷中反馈最集中的痛点——75% 的设计师表示协作流程不清晰，58.33% 担心研发不愿意接受设计师写的代码。

**1.研发不愿意配合怎么办？**

- 先拿出咱优美的“Demo作品集”，表示诚意。（亲测有用，demo足够好的话，一般研发会大为震惊）
    
- 从0-1的需求开始合作，比如活动页H5等。不涉及主仓库的变动。
    
- 从时间比较宽松的需求合作。逐步构建信任基础。
    

**2.担心交付的代码研发不用怎么办？--如何提升采纳率**

- 确保在同一个仓库工作。
    
- 确保配置好了编码规范和 Rules //Skills配置（研发帮忙），让 AI 生成的代码天然符合要求。
    
- 了解代码仓库的目录结构是怎样的，哪些公共文件或公共样式是绝对不能改的。可指定AI只改某个文件夹的代码，避免它乱发挥。
    

**3.研发不愿意配合、兜底，出问题了责任在谁？**

责任归属取决于交付形式和上线链路：

- 如果是交付产品 Demo（不上线、不进仓库），代码不进入生产环境，出问题的风险极低，责任也不涉及线上故障。
    
- 如果是交付 0-1 需求（独立部署上线），设计师作为主要开发者，需要对代码质量负责，研发做 Review 后共同承担。
    
- 如果是交付 1-N 需求（进入存量仓库、走 PR 流程），设计师提交代码，研发 Review 并合并，合并后的代码责任由研发侧承担，这和正常研发流程一致。设计师的责任边界是：提交符合规范的代码、做好自查 diff、如实说明改动范围。
    

**核心原则：不绕过 Review，不跳过流程**

设计师 AI Coding 不是偷偷改代码上线，而是通过正规的 PR 流程参与开发。只要遵守这个原则，责任边界就是清晰的。研发 Review 通过并合并，意味着他对这段代码的质量做了背书；如果研发拒绝合并，代码就不会上线，风险也不会发生。所以，出问题的责任不会单独落在设计师身上，关键是要走完整的协作流程，不要绕过研发自行上线。

# 看不懂代码怎么办？

**你不需要"看懂所有代码"，你需要的是"看懂关键部分"。**

**心态调整：**1）研发也不是每行代码都看懂的，他们也天天查文档、问 AI；2）你的目标不是成为程序员，而是能"读懂大意、改对地方"。3）先从完全看不懂，vibe coding 开始慢慢了解。

**实用方法："直接问AI😀"**
