境外工程Harness 运营 SOP：让 AI 上下文保持鲜活

> Harness 不是装修完就锁门的房子，是每天都要住的地方。  
> 这份 SOP 告诉你：**什么时候该更新、更新哪里、怎么更新**。

---

## 🚀 使用姿势：把我丢给你的 CC，生成维护规则

> **这是整份 SOP 最重要的一步。** 与其靠人记得更新，不如让 CC 主动提醒、主动起草、随 PR 一并提交。

**做一次，永久生效。** 把下面这段话直接丢给你工程的 CC：

```
请读取这份 SOP wiki：https://km.sankuai.com/collabpage/2758323328
根据其中的五条触发路径和七个 Harness 文件，在当前工程的 CLAUDE.md 末尾追加一个
「Harness 更新检查」section，要求：
1. 每次完成需求开发或 bug 修复后，CC 主动对照五条路径判断是否需要更新
2. 需要更新时，CC 起草好 md 内容，询问研发确认
3. 研发确认后，CC 将 md 变更随业务 PR 一并提交
```

CC 会自动读取 wiki、理解规则、写入 CLAUDE.md。之后每次在这个工程里工作，CC 都会在任务结束时自动 check，不再依赖人的自觉。

**效果是什么样的：**

- 修完一个 bug，CC 说：「这次修复了 distribution 模块的慢查询，需要追加到 issues.md 吗？我已起草好，你确认一下」
    
- 新需求开发完，CC 说：「接口有变更，需要更新 docs/specs/ 和 dependencies.md 吗？」
    
- 研发点确认，CC 提交 PR，md 和代码变更在同一个 PR 里，review 时一眼可见
    

**为什么随业务 PR 一并提交，而不是单独开 PR：** md 和代码是同一次改动的上下文，放在一起 review 时能互相印证；单独开 PR 反而容易被跳过不 merge。

## 一、Harness 文件体系（7 个文件）

每个工程建立以下 7 个文件，各司其职：

|文件|职责|上限|
|---|---|---|
|`CLAUDE.md`|工程地图：工程简介 / 技术栈 / 核心约束 / 禁止事项 / 已知高频风险|≤ 80 行|
|`.claude/hooks-template.json`|编译自动检查 Hook，每次 AI 改 Java 文件后自动跑 `mvn compile`|—|
|`docs/knowledge/issues.md`|历史线上问题知识库：现象 / 根因 / 修复 PR / 预防规则|持续追加|
|`docs/tests/cases.md`|核心回归测试场景，AI 生成测试时的参考基准|持续追加|
|`docs/specs/`|需求设计文档目录，每个需求一个文件|持续新增|
|`docs/dependencies.md`|上下游依赖全景：上游调用方 / 下游依赖 / 跨工程共享 DB 表|按需更新|
|`docs/<module>-mq.md` / `-db.md`|MQ topic 详情 / DB 数据源详情（topic > 3 或数据源 > 1 时从 CLAUDE.md 拆出）|按需新增|

## 二、全景：五条触发路径 × 七个文件

```
触发路径                         →  需要更新的文件
─────────────────────────────────────────────────────────────────────
① 新产品需求                     →  docs/specs/（必须）
                                    docs/dependencies.md（新增上下游时）
                                    docs/tests/cases.md（新增回归场景）
                                    docs/<module>-mq.md / -db.md（新增 topic/数据源时）

② 服务质量运维报告（每周）       →  docs/knowledge/issues.md（必须）
                                    CLAUDE.md（高频问题升级为约束时）
                                    docs/tests/cases.md（补充修复后的回归场景）

③ Raptor 告警                    →  docs/knowledge/issues.md（必须）
                                    CLAUDE.md（反复出现的告警加约束）

④ 坐席反馈 → 研发排查确认       →  docs/knowledge/issues.md（改了代码时必须）
                                    CLAUDE.md（发现边界行为约束时）
                                    docs/tests/cases.md（补充修复后的回归场景）

⑤ CR 评审报告 & PR 回复         →  docs/knowledge/issues.md（工程特有坏味道）
                                    CLAUDE.md（编码约束 / 设计决策解释）
                                    .claude/hooks-template.json（发现编译约束时）
```

五条路，殊途同归：**最终都要沉淀到 Harness 文件里**，让下一次 AI 协作不用从零开始交代背景。

## 三、路径详解

### 路径一：新产品需求来了

**触发时机**：产品提了需求、排了迭代，研发开始设计阶段。

**要做什么**：

1. 在工程的 `docs/specs/` 下新建一个设计文档，用 `TEMPLATE.md` 为模板，写入需求背景、接口变更、DB 变更、MQ 变更
    
2. 如果引入了新的上下游依赖，同步更新 `docs/dependencies.md`
    
3. 如果新增了 MQ topic 或 DB 数据源，追加到 `docs/<module>-mq.md` 或 `-db.md`
    
4. 需求上线后，在 `docs/tests/cases.md` 补充这个需求的核心回归场景
    

**一句话判断**：这个需求上线后，AI 如果不知道这个背景，会不会误操作？会 → 必须写。

**示例**：

```
需求：新增"超时工单自动关闭"能力，接入 Mafka topic: csc.ticket.timeout
→ 新建 docs/specs/2026Q2-auto-close.md
→ 在 docs/sepdds-mq.md 追加这个 Consumer
→ 在 CLAUDE.md 的"核心约束"加一行：超时关闭消费幂等，不得重复关单
→ 在 docs/tests/cases.md 追加：工单超时后自动关闭，重复触发不重复关单
```

**不需要写的**：纯前端改动、配置调整、文案修改——不影响 AI 理解代码行为的，不用写。

---

### 路径二：每周服务质量运维报告

**触发时机**：每周运维报告出来，里面有 case 需要研发跟进修复。

**流程**：

```
1. 研发认领 case，排查根因
2. 修复上线
3. 修复完成后，5 分钟内追加到 issues.md ← 这步最容易被跳过！
4. 如果是高频问题（同类出现 2 次以上），同步在 CLAUDE.md 加约束
5. 补充 docs/tests/cases.md 中对应的回归场景
```

**issues.md 追加格式**（复制粘贴就能用）：

```
## YYYY-MM-DD - <一句话描述问题>

**现象**：<坐席/系统看到了什么>
**根因**：<哪行代码/哪个逻辑导致的>
**修复**：<改了什么，PR 链接>
**预防规则**：<下次 AI 写代码时应该注意什么>
```

**为什么要写"预防规则"**：这是最有价值的一行。它会被 AI 在下次代码评审时自动参考，等于把这个 case 变成了永久的检查项。

**真实示例**：

```
## 2026-04-15 - 分单延迟飙升，因 distribution 模块 DB 连接池耗尽

**现象**：工单分配 P99 延迟从 180ms 升至 3s，持续 20 分钟
**根因**：慢查询导致连接长时间占用，连接池（maxActive=20）耗尽
**修复**：加索引 + 连接池扩容至 50，PR #1234
**预防规则**：distribution 模块新增 DB 查询，必须先用 explain 验证走索引
```

---

### 路径三：收到 Raptor 告警

**触发时机**：研发收到 Raptor 推送的告警通知，同步给 CC 排查。

**处理流程**：

```
收到告警
  ↓
打开 Claude Code，进入对应工程目录
  ↓
研发把告警内容同步给 CC，CC 粘贴给 AI："收到这个告警，帮我排查"
  ↓
AI 结合 Harness 上下文（CLAUDE.md + dependencies.md）给出排查路径
  ↓
修复上线
  ↓
追加到 issues.md（同路径二的格式）
  ↓
反复出现的同类告警 → 在 CLAUDE.md 加"已知高频告警"约束
  ↓
补充 docs/tests/cases.md 中对应的回归场景
```

**告警修复后必须沉淀的判断标准**：

- 同一类告警第一次出现 → **必须写 issues.md**
    
- 同一类告警反复出现 → **必须写 issues.md，并在 CLAUDE.md 加约束**
    
- 修复后有对应回归场景 → **补充 docs/tests/cases.md**
    
- 误告警（配置问题）→ 写一句说明，防止下次又被误导
    

**CLAUDE.md 加约束的示例**：

```
## 已知高频告警
- `buffer 模块 queue_size > 1000`：通常是上游流量突增，先看 access 模块入口 QPS，不要直接扩容
```

### 路径四：坐席反馈 → 研发排查确认

**触发时机**：坐席通过 TT 或群反馈了问题，研发排查后确认是 bug 或需要优化。

**这条路径最容易被忽略**，因为没有系统化的触发机制，全靠人工意识。

**判断是否需要沉淀**：

|情况|是否沉淀|沉淀位置|
|---|---|---|
|研发改了代码修复|**必须**|`issues.md`|
|研发优化了逻辑/配置|**必须**|`issues.md`|
|修复后有新的回归场景|**必须**|`docs/tests/cases.md`|
|坐席操作问题，代码没问题|不需要|—|
|数据问题，走数据修复流程|不需要|—|

**示例**：

```
## 2026-04-18 - 坐席反馈：工单转接后状态显示异常

**现象**：坐席 A 转接给坐席 B 后，B 的工作台工单状态仍显示"待接入"
**根因**：workbench-bff 聚合接口缓存未及时失效，TTL=30s
**修复**：转接成功后主动 evict 对应坐席的工单列表缓存，PR #5678
**预防规则**：workbench-bff 所有写操作（转接/关单/备注）必须同步清除相关坐席缓存
```

---

### 路径五：CR 评审报告 & PR 回复

**触发时机**：通过 cr-checklist 完成代码评审后，或研发在 PR 上回复了评审意见。

**为什么这条路径最有价值**：CR 评审发现的问题，往往是这个工程的"特有坏味道"——同一类问题在同一个工程里反复出现。把它沉淀下来，下次 AI 评审时会自动带着这个记忆去检查。

**流程**：

```
CR 评审完成（cr-checklist 输出报告）
  ↓
识别：哪些问题是"这个工程特有的模式"？
  ↓
沉淀到 docs/knowledge/issues.md 的"已知高频风险"章节
  ↓
如果是约束性规则（不该这么写）→ 同步更新 CLAUDE.md
  ↓
如果涉及编译/构建约束 → 检查是否需要更新 .claude/hooks-template.json
```

**判断是否值得沉淀**：

|情况|沉淀位置|
|---|---|
|发现一个设计缺陷，研发确认修复|`issues.md`|
|发现一个编码习惯问题（如未设超时）|`CLAUDE.md` 的"核心约束"或"禁止事项"|
|研发在 PR 回复里解释了一个非显而易见的设计决策|`CLAUDE.md` 加注释，或 `issues.md` 加背景|
|评审发现误报（AI 判断有误）|不沉淀，但可以在 CLAUDE.md 加例外说明|

**示例**：

评审报告里发现：

```
高风险：distribution 模块 3 处 DB 查询无超时设置
```

研发修复后，沉淀到 `distribution/CLAUDE.md`：

```
## 核心约束
- 所有 Zebra DB 查询必须设 queryTimeout，默认 2000ms（历史多次因慢查询触发连接池耗尽）
```

## 四、维护节奏

|频率|动作|涉及文件|
|---|---|---|
|**每次修复线上问题后**|追加问题记录（5 分钟，不要拖）|`issues.md`，高频问题同步 `CLAUDE.md`|
|**每次新需求设计时**|新建设计文档，同步依赖和测试场景|`specs/`、`dependencies.md`、`cases.md`、`mq/db.md`|
|**每次 CR 评审后**|沉淀工程特有坏味道和设计决策|`issues.md`、`CLAUDE.md`、`hooks-template.json`|
|**每月一次**|清理低频约束，保持精简|`CLAUDE.md`（≤ 80 行）|
|**每季度一次**|检查上下游是否有变更|`dependencies.md`|

## 五、issues.md 分层治理：防止知识库膨胀

`issues.md` 是积累最快的文件，也是最容易失控的文件。**不加治理的话，半年后它会变成一个没人愿意读的垃圾堆。**

### 分层信号：什么时候该拆

|信号|建议动作|
|---|---|
|issues.md 超过 **100 条**记录|开始考虑按领域拆分|
|某个领域（如缓存、MQ、DB）的问题占比超过 **30%**|该领域独立成文件|
|新人说"issues.md 太长，不知道从哪看"|立刻拆，不要等|

### 拆分方式：主文件留主链路，领域文件管细节

拆分后的结构：

```
docs/knowledge/
├── issues.md              ← 主文件：只保留主链路问题 + 各子模块文件的指针
├── issues-distribution.md ← 分单模块相关问题
├── issues-workbench.md    ← 坐席工作台相关问题
├── issues-im.md           ← 即时通讯相关问题
└── issues-<xxx>.md        ← 按子模块/子领域命名，视工程实际情况拆
```

主文件 `issues.md` 的顶部加指针区：

```
## 子模块知识库索引
- 分单模块 → [issues-distribution.md](issues-distribution.md)（分单延迟 / 规则引擎 / 连接池）
- 坐席工作台 → [issues-workbench.md](issues-workbench.md)（缓存失效 / 状态同步 / 转接异常）
- 即时通讯 → [issues-im.md](issues-im.md)（消息丢失 / 会话状态 / 推送延迟）
- <xxx 子模块> → [issues-xxx.md](issues-xxx.md)（按工程实际子模块命名）

## 主链路问题（跨子模块 / 架构级 / 高频告警）
...（只保留不属于单一子模块的问题）
```

### 拆分原则

- **主文件只留主链路**：跨领域的问题、架构级的约束、高频告警留在 `issues.md`；单一领域的细节下沉到领域文件
    
- **指针必须写清楚**：主文件里的领域指针要说明该文件管什么，让 AI 知道去哪找
    
- **领域文件格式不变**：仍然是"现象 / 根因 / 修复 / 预防规则"四段式，AI 读起来一致
    
- **不要过早拆**：条数不多时强行拆反而增加维护负担，50 条是合理的触发线

```
帮我把这个 case 沉淀到 issues.md：
[把根因和修复简单描述一下]
```
