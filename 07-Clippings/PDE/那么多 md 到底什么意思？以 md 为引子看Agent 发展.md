# 那么多 md 到底什么意思？以 md 为引子看Agent 发展


本文做三件事
---
把一堆 .md 文件的来龙去脉梳理清楚、从演化中抽出几个对组织有用的启发、最后落到一个具体问题：业务组织如果想认真构建 AI 新范式的环境，该怎么动手，Ground 该写什么。

引子：为什么会是 markdown

在 AI 编程工具铺开的两年里，出现了一件不太被注意但很反常的事——一个 1991 年就被发明、本意是给人快速写文档的格式，markdown，正在变成 AI 时代最重要的接口规范。

`CLAUDE.md`、`AGENTS.md`、`SKILL.md`、`MEMORY.md`、`ADR`、`PLAYBOOK`……每一个工具厂商、每一个 agent 框架，几乎不约而同地选了 .md 作为载体。

这不是巧合。LLM 在海量文本上训练，对 markdown 这种"半结构化、人类可读"的格式有最低的语义损耗。比起 JSON 的严苛、XML 的冗余、YAML 的歧义，markdown 是 AI 和人类的最大公约数。**当所有人都没商量好用什么格式时，语言模型自己选择了它**。

这个细节本身就是 AI 范式转移的隐喻：**新的标准不是被规定出来的，是被使用催生出来的**。


一、一堆 md 的考古学

按时间线大致是这样：

2011 · ADR 出现

**Architecture Decision Records**。Michael Nygard 在一篇博客里提出，出发点很朴素：代码能告诉你"是什么"，告诉不了你"为什么不选另一个"。

十多年里它一直是软件工程的小众实践，直到 AI 进场——AI 不读 ADR 就会反复建议"为什么不用 X 方案"，**ADR 第一次有了硬刚需**。

2014 前后 · README.md 成为事实标准

GitHub 让 README 变成项目的门面。重要的是，它确立了一个心智模型：

项目根目录的某个特定文件，是"读这个项目"的入口。

今天所有 CLAUDE.md / AGENTS.md 的位置约定，都是从这里继承的。

2022 · ChatGPT，长 prompt 时代

所有指令都塞进一个对话框，system prompt 和 user prompt 混在一起，没有"持久化"的概念。

2023 · MemGPT 提出 context 管理

Letta 团队把 context 视为一种有限资源，引入"长期记忆"和"短期记忆"分层。**这是 MEMORY.md 这类文件的思想起源**。

2023-2024 · 各家 IDE 各自立旗

这是一段碎片化的时期，每家都有自己的私有约定：

工具
私有约定文件
Cursor
`.cursorrules`
Windsurf
`.windsurfrules`
GitHub Copilot
`.github/copilot-instructions.md`
Aider
`CONVENTIONS.md`
Anthropic Claude Code
`CLAUDE.md`

2025 上半年 · AGENTS.md 标准化尝试

OpenAI 联合若干厂商推动一个跨工具的通用文件名 `AGENTS.md`，目标是终结上面的碎片化。今天它处在"被多数工具兼容、但还没完全替代各家私有文件"的过渡态。

2025 中 · Anthropic Skills 发布

这是个关键转折点。它把"长 prompt"演化成"按需加载的能力模块"：每个 SKILL.md 自带 frontmatter 描述触发条件，agent 自己判断什么时候加载。

关键跃迁
---
接口从"一次性注入"升级为"运行时调度"。这是 md 从"文档"变成"组件"的转折点。

2025 下 · Plugins、Subagent、Hooks 等机制铺开

Claude Code、Cursor 等都引入插件市场、子 agent 协作、生命周期钩子。md 文件不再是孤立文档，而是装配进一个运行时的"组件"。

到 2026 · md 作为 AI 接口规范的事实地位确立

几乎所有正经在用 AI 协作的团队，都在用某种形式的 md 文件分层管理 context。

几个不太主流但值得一提的名字

**SOUL.md / PERSONA.md / CHARTER.md** —— 不是来自编程工具，而是来自 Character.AI 风格的角色扮演 chatbot 和 Letta 这类 agent 框架的 "persona" 概念。**在产品化 chatbot 里是必要的，但在严肃的工程协作里大多是花架子**。
**PLAYBOOK** —— 来自 Google SRE Book 和 DevOps 文化，本来就是 SOP 的形式化。AI 时代它被原样继承，因为流程性知识恰好是 AI 最容易消化的形态。
**Ground** —— 这个术语没有统一来源，在不同团队里叫法不同（也有叫 Context、Domain、Knowledge Base 的）。它是 AI 协作元范式还没收敛的那一块，**谁先把它做成可复用资产，谁就掌握了组织级的话语权**。


二、演化中浮现的五个启发

把这条时间线拉远看，对组织有用的启发有五个。

启发一 · 标准是被使用催生的，不是被规定的

markdown 没有 RFC，没有委员会，它赢了。CLAUDE.md 没有标准化文档，但全行业都看得懂。

这意味着：组织在 AI 范式上的决策，不应该追求"先定标准再推行"，而应该追求"先跑起来再收敛"。**提前定标准只会做出无人遵守的文档**。

启发二 · 从单 prompt 到文件系统，是从命令式到声明式的跃迁

早期写 prompt 是"告诉 AI 怎么做"，今天写 SKILL/PLAYBOOK 是"声明组织有什么能力、AI 自己决定怎么用"。

这跟 SQL 之于过程式数据库、React 之于 jQuery 是同一种跃迁。

组织的认知配置应该是**声明性的**——描述事实和约束，而不是规定步骤。

启发三 · AI 在反向倒逼组织做知识管理

一个让团队多年下不了决心做的事情（把口口相传的知识写下来），被 AI 协作的需求倒逼着完成了。

隐藏红利
---
很多团队回头发现，**写 CLAUDE.md 的过程比 AI 用 CLAUDE.md 更有价值**。这是 AI 给组织的隐藏红利。

启发四 · 上下文边界在持续外扩

会话级 → 项目级 → 用户级 → 组织级 → 行业级（AGENTS.md 标准化）。

这条曲线还没到顶。组织今天纠结的事——CLAUDE.md 怎么写、SKILL 怎么沉淀——五年后会显得很基础。

真正的赌注在更上一层：**业务知识的组织级资产化**。

启发五 · 接口规范的"维度正交"是组织能力的体现

如果一个组织的 CLAUDE.md 里塞着 ADR、SOP、术语表、个人偏好，说明这个组织还没想清楚边界。

能把"身份 / 能力 / 上下文 / 记忆 / 编排"五层分清楚的组织，在 AI 协作上的效率会拉开数量级。


三、业务组织构建 AI 新范式的路径

如果一个业务组织真的想认真做这件事，下面是四个阶段的推进顺序。

不要跳级
---
四个阶段是台阶，不是并行选项。跳了基本会塌——上层的知识沉淀依赖下层的使用习惯，使用习惯没养成，沉淀就是空中楼阁。

Phase 0 · 工具基础设施（2-4 周）

在做任何文件之前，先把"地"修好：

**选定 harness 默认栈** —— Claude Code / Cursor / Copilot / 自研，定一个，允许例外但不要每人各异
**解决内网与合规** —— 大企业绕不开的坎。内网代理、数据出境、license 审计、提示词中的敏感信息过滤
**统一最小约定** —— 比如"所有项目根目录必须有 CLAUDE.md"，仅此一条，不要一开始就规定怎么写

这一阶段的关键产出是**让 AI 真的能跑起来**，不是文档。

Phase 1 · 个人层（持续累积）

让每个工程师先把自己用顺：

用户级 `CLAUDE.md` —— 个人偏好、角色、常用上下文
用户级 `MEMORY.md` —— 跨会话的个人沉淀
个人 SKILL —— 把自己重复做的事固化成命令

**这一阶段不要强推、不要 review，让人自发产生需求**。一个团队里如果连 10% 的人主动维护 CLAUDE.md 都做不到，后面再大的投入都是空中楼阁。

Phase 2 · 项目层（项目维度自然发生）

每个项目逐步沉淀：

项目根 `CLAUDE.md / AGENTS.md` —— 行为约束、技术栈、注意事项
项目 `commands/` —— 项目级 slash command
项目 `PLAYBOOK.md` —— 上线、应急、调试流程
项目 `DECISIONS/` —— 不可逆决策的 ADR

这一阶段开始有协作价值。**新人接手项目，第一件事就是读 CLAUDE.md + PLAYBOOK，效率提升立竿见影**。

Phase 3 · 组织层（关键跃迁，大多数组织卡在这里）

关键跃迁
---
这一步是质变，不是项目级文件的简单堆叠。它要求组织承认"业务知识是资产，且这份资产是给 AI 读的"——这个共识不容易达成。能跨过这一步的组织，会进入下一个层级的协作效率。

需要明确的产出：

**Ground 仓库** —— 业务真源，组织 / 域 / 项目三级结构，独立仓库或独立目录，**不能塞进任何单项目**
**组织级 SKILL 库** —— 跨项目复用的能力，比如"调内部 SSO"、"查内部知识库"、"发内部消息"
**组织级 ADR** —— 技术选型、架构原则、合规决议
**组织级 PLAYBOOK** —— 上线 SOP、安全应急、新人入职

很多组织停留在"AI 是辅助工具"的认知里，不愿意为 AI 投资业务知识的结构化——这就是这一步会卡住的根本原因。

Phase 4 · 生态层（长期演化）

到这一步，AI 协作不再是"工具"，而是"环境"：

**AI 友好的 API 文档** —— 不只是 OpenAPI，还包括语义层（这个接口的业务含义、典型用法）
**AI 友好的内部知识库** —— 学城 / Confluence / Notion 这类工具，需要被 AI 可读、可索引、可引用
**度量与反馈循环** —— AI 使用率、SKILL 命中率、CLAUDE.md 准确率，纳入工程效能指标
**范式治理** —— 谁负责 Ground 的演进、SKILL 的审核、ADR 的归档

这一阶段不再是技术问题，**是组织治理问题**。


四、Ground 应该写什么

Ground 是 Phase 3 跃迁里最核心的产出。它最容易被写成"业务介绍 PPT 的 markdown 化"——堆一堆愿景、产品、组织架构。**这是错的**。

Ground 的判别原则
---
Ground 是给 AI 读的，目的是让 AI 在缺少人解释的情况下，能做出符合业务直觉的判断。

**一段 Ground 内容，如果删掉、AI 就会做出违背业务直觉的判断——它就该在；如果删掉没影响——它就不该在**。

按三级结构，Ground 该写的内容是：

组织级 Ground（写一次，五年不大动）

**业务定义** —— 我们到底是做什么的，一句话能讲清楚。**不是 slogan，是约束**。比如"我们是一个 to B 的本地生活履约平台"会决定后面所有判断
**关键利益相关方** —— 商家、骑手、用户、监管、合作伙伴，各自的核心诉求和冲突点
**业务术语表（glossary）** —— 行话和缩写。**这是 AI 最容易被坑的地方**："履约"在你们这里是什么意思？"承诺时间"和"承诺时长"区别是什么？
**核心原则与红线** —— 用户隐私、资金安全、合规底线。这些不是教条，是 AI 在边界判断时的依据
**商业模式骨架** —— 钱从哪来、成本结构、KPI 树。AI 在建议任何业务方案前，必须知道这个

域级 Ground（按业务域，各自维护）

**本域的边界** —— 我们负责什么、不负责什么、和邻接域的接口
**核心概念和实体** —— 这个域内的"名词"，尤其是和组织级 glossary 不同的本域专属术语
**关键流程** —— 不是组织级 SOP，是这个域独有的业务流程
**历史背景** —— 这个域为什么以现在的形态存在，哪些是历史包袱、哪些是有意为之
**典型场景与反例** —— 哪些是"标准情况"、哪些是"千万别这么做"。**后者尤其重要**，AI 没有反例就只会建议看起来合理的方案

项目级 Ground（短周期、随项目演化）

**项目目标与非目标** —— 我们这次想达成什么、明确不做什么
**当前状态** —— 不是 changelog，是"此刻的项目处在什么阶段、谁在 block 谁"
**关键决策指针** —— 指向项目相关的 ADR，不要复述
**协作上下文** —— 主要参与者、决策链路、当前 blocker

Ground 不该写什么（同样重要）

不该写
因为它属于……
技术架构
README 和 ARCHITECTURE.md
代码规范
CLAUDE.md
操作 SOP
PLAYBOOK
PPT 那一套 vision / mission / values
如果不能转化为 AI 在边界判断时的依据，不要放
季度 OKR、当前活动
时效性强，会快速腐烂


结语

md 是引子，看到的是 AI 协作的范式转移。

这场转移的核心不是"AI 更聪明了"，而是**组织本身正在被 AI 倒逼着重构成"可读的"**——可被新人读、可被 AI 读、可被未来的自己读。

这件事的赌注比工具选型大得多。今天纠结 CLAUDE.md 怎么写的组织，五年后会发现自己讨论的是"业务怎么被结构化、知识怎么被资产化、决策怎么被显性化"。**md 文件只是这个过程留下的痕迹**。

一句话
---
真正的问题从来不是"我们要不要用 AI"，而是"我们的组织能不能被 AI 读懂"。
