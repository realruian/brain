## 一、项目背景

AI 发展正在重塑产研范式，从"人规划 + 执行"向"人规划 + AI 执行"演进。AI Coding 大幅提升 Demo 和代码生产效率，但 AI 生成代码缺乏对规范的理解，**验收将成为整条链路上的效率瓶颈**——生产越快，验收压力越大，自动化验收是产研闭环的必备一环。

当前痛点：

- 验收人力投入是产研链路中成本高、价值低的典型环节，设计部全年设计验收投入约 12,000 pd（每人每月约 2.5 pd），研发调优代码成本远高于设计；
    
- 一致性治理依赖人工推动，每次需求迭代都可能引入劣化，体验问题大多被动发现、事后修复；
    
- 旧生产链路缺乏自动化监控手段，无法形成持续运转的质量守护机制。
    

---

## 二、项目目标

建设覆盖"设计规范验收、一致性防裂化、体验问题识别"的全链路 AI 自动验收工具，逐步解放设计、研发、测试人力，支撑新产研范式下 AI Coding 的质量闭环，保障产品体验质量。

|期望目标|预计达成时间|衡量标准|
|---|---|---|
|提效|2026-12-31|AI 交付代码符合规范占比 >80%|

## 四、聚焦：新生产范式验收闭环

### 4.1 核心思路

新生产范式下，AI Coding 工具（Roo / Cursor 等）负责代码生成，**自动验收 Skill 作为质量守门员嵌入生产链路**，形成"生成 → 验收 → 修复 → 再验收"的自动闭环。

```
RRD 文档 / 需求输入
        ↓
   AI Coding 生成代码
        ↓
  [自动验收 Skill 触发]
  Git Diff 扫描变更
        ↓
  构建结构树 + 提取样式
        ↓
  索引规范知识库（四层比对）
        ↓
  生成合规报告（优秀/良好/一般/较差）
        ↓
  一键修复建议 → 重新验收
```

### 4.2 技术实现：四层规范比对体系

当前已落地的自动验收工具（`food-onion-ux-judger`）采用**四层规范比对**架构，覆盖从基础 Token 到场景语义的完整规范体系：

|层级|比对内容|规范来源|说明|
|---|---|---|---|
|**第一层：分子层（Molecule）**|组件识别 + Props 配置合规性|样式规范 → 组件规范（Leez / MTD）|使用框架组件时必查|
|**第二层：原子层（Atom）**|基础 Token 值合规性（颜色、字号、间距、圆角等）|样式规范 — 原子规范|始终必查，无论是否使用组件|
|**第三层：行为层（Behavior）**|交互行为 / 动效 / 热区 / 排版|行为规范|所有节点必查|
|**第四层：场景层（Scene）**|元素排布合理性|本地 `docs/` 场景知识文件 + 存量代码实践|所有节点必查|

**两类规范的关系：**

- **样式规范（Style Spec）** 是最根本的合规判断标准，所有 UI 内容都必须符合，不因组件规范是否可查到而改变；
    
- **组件规范（Component Spec）** 是辅助 AI 理解框架组件渲染表现的工具，查不到时不影响样式规范层面的比对结论。
    

**支持框架：**

|端类型|框架|识别特征|
|---|---|---|
|C 端（手机 / 消费者）|**Leez**|`@leez/...` / `leez-*` / import 路径含 `leez`|
|B 端（商家 / 管理后台）|**MTD**|`@ss/mtd-react3` / `@mtd/...` / `mt-design`|

### 4.3 执行流程（Skill 工作流）

自动验收 Skill 共 **5 个核心步骤 + 1 个可选修复步骤**，全程自动连续执行：

**Step 1 — 采集（Scan）**  
读取项目配置缓存 → 确认端类型与检测范围 → 采集目标代码（全量分批 / Git Diff BFC 提取）→ 框架识别 → 读取组件索引 → 输出 `output/.cache/scan_result.md`。

**Step 2 — 分析（Analyze）**  
变更结构树构建 → 节点规范层级标注 → 节点属性提取 → 输出 `output/diff_analyse.md`。

**Step 3 — 比对（Compare）**  
逐节点四层规范比对（分子层 → 原子层 → 行为层 → 场景层）→ 联合决策去重 → 输出 `output/structure.json` + `output/result.md`。

**Step 4 — 截图并打标（Screenshot & Annotate）**（可选）  
获取运行时截图 → 生成网格参考图 → AI 定位坐标 → 绘制水滴 Pin 标注，直观呈现问题位置。

**Step 5 — 上报（Report）**  
自动获取 MIS ID → 校验 `structure.json` 字段完整性 → POST 上报至服务端 → 询问是否进入修复步骤。

**Step 6 — 修改（Fix）**（用户确认后触发）  
最多循环 4 轮：引导勾选节点 + 粘贴 Prompt → 按 P0→P1→P2→P3 优先级执行代码修改 → 重新比对 + 截图打标 + 上报 → 询问继续 / 结束。

### 4.4 全量分析约束

工具对所有变更点进行**完整的结构树分析和四层规范比对，严禁抽样**。变更点数量庞大时分批连续处理，每批上限 200k 字符，按布局结构和代码依赖关系归组，确保模块语义完整性。

### 4.5 设计知识库构建

自动验收的准确率上限由知识库质量决定，知识库建设是核心基础工程：
![[../../../_attachments/Pasted image 20260520154835.png]]

**知识库建设路径：**

- 4 月：完成 C/B 端通用规范的知识库搭建；
    
- 5 月：完成业务核心流程/页面/模块规范的知识库搭建，根据使用反馈补充缺失规范，评测知识库准确率并调优；
    
- 6 月：知识库覆盖核心业务 90% 以上规范。
    

**评测机制：** 根据已生成的知识产出构建 good case / bad case，明确评测流程与标准；通过"检索被测和知识之间索引准确率"专项评测，由设计提供验收诉求、研发构造对应 Case 进行验证。
