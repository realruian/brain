## 产品设计验收 · 可上线标准 v2

> 两种生产方式（AI Coding 无设计稿 / 设计稿前端还原）的统一验收体系，包含四层检测机制与完整打分算法。

---

### 一、验收体系总览

验收分为**四层**，层层递进，最终输出一个综合分数与上线结论。

|层级|名称|检测对象|实现方式|是否可阻断上线|
|---|---|---|---|---|
|L1|视觉规范层|样式属性是否符合规范 / 100% 还原设计稿|读取代码样式 / Git Diff|✅ 是|
|L2|行为规范层|动效、热区、文案符号等交互规范|读取代码 + 人工补充口子|✅ 是|
|L3|适配压测层|多机型适配效果、极端场景压力测试|云真机 + 场景 Badcase 库匹配|✅ 是|
|L4|人工一票否决层|机器无法识别的严重体验问题|人工验收|✅ 是（最高优先级）|

**两种生产方式的差异点仅在 L1 层**，L2 / L3 / L4 完全通用。

---

### 二、两种生产方式的 L1 差异

#### 方式一：AI Coding 验收（无设计稿）

L1 以「规范知识库」为基准，检测代码是否符合设计规范三层体系：

- **原子层**：色值 Token、字号字阶、圆角值、阴影层级、间距栅格
    
- **分子层**：Leez/MTD 组件使用正确性、变体匹配、废弃组件检测
    
- **场景层**：页面模块结构、信息层级、通用模块一致性
    

**L1 准入标准**：三层规范合规率均 ≥ 90%，检测准确率需达到 98%（即系统召回的问题中，真实问题占比 ≥ 98%，避免误报干扰研发）。

#### 方式二：设计稿前端还原度验收（有设计稿）

L1 以「设计稿」为基准，通过图片比对 + 结构树节点匹配量化还原偏差：

- 新容器（洋葱）：数据层完全一致，对比准确度无风险，可直接放行
    
- 旧容器：依赖当前探索方案进展，置信度需在报告中标注，低置信度项需人工复核
    

**L1 准入标准**：标准机型（375pt）还原度 = 100%，检测准确率需达到 98%。

---

### 三、L2 行为规范层

检测对象：动效、热区、标题文案符号、操作反馈等交互规范问题。

**实现方式**：读取代码（静态分析）为主，人工补充口子为辅。

**自动检测项**：

|检测项|标准|检测方式|
|---|---|---|
|热区尺寸|≥ 44×44pt|代码静态分析|
|相邻热区间距|≥ 8pt|代码静态分析|
|标题符号规范|符合「标题呈现6原则」|代码静态分析|
|提示文案规范|符合弹窗文案撰写规范|代码静态分析|
|操作反馈存在性|点击/异步/成功/失败均有反馈|代码静态分析|
|动效规范（知识库完善后）|时长、缓动曲线符合规范|代码静态分析|

**人工补充口子**：自动检测完成后，验收人员可在报告中手动追加问题，追加的问题同样参与打分，且支持标注严重程度（P0 / P1 / P2）。动效规范在知识库完善前，强制要求人工确认并在报告中标注结论。

**L2 准入标准**：自动检测 + 人工补充后，P0 级问题 = 0。

---

### 四、L3 适配压测层

**实现方式**：调取云真机，按不同机型运行页面，将截图与「场景 Badcase 库」中的已知问题模式进行匹配，自动生成适配报告。

**覆盖机型矩阵**：

|机型类别|代表机型|检测重点|
|---|---|---|
|标准机型|iPhone 14（375pt）|基准还原|
|大屏机型|iPhone 15 Pro Max（430pt）|布局拉伸、留白异常|
|小屏机型|iPhone SE 3（375pt，低分辨率）|内容截断、字号过小|
|异形屏|iPhone 14 Pro（动态岛）|顶部遮挡、安全区|
|安卓旗舰|小米 14 / 华为 Mate 60|字体渲染、圆角差异|
|安卓折叠屏|华为 Mate X5|大屏布局适配|

**场景 Badcase 库**：持续沉淀历史验收中发现的适配问题模式，作为云真机检测的匹配基准。新发现的适配问题经人工确认后自动入库。

**极端场景压力测试**：

- 超长文本填充（商品名 50 字、地址 100 字）
    
- 超多列表项（1000 条数据滚动）
    
- 弱网环境（2G/3G）下的加载状态
    
- 无数据/空态场景
    

**L3 准入标准**：云真机检测无 P0 级适配 Bug，适配报告自动生成并归档。

---

### 五、L4 人工一票否决层

机器无法识别所有体验问题，人工验收具有最高优先级的一票否决权。

**触发条件**：验收人员（设计师 / 产品经理）在使用真机体验时，发现任何**严重阻塞用户完成核心任务**的体验问题，无论机器是否已识别，均可直接标记为「人工否决」，阻断上线。

**人工否决的记录要求**：

- 问题描述（必填）
    
- 复现路径（必填）
    
- 严重程度（必填：用户完全无法完成任务 / 用户完成任务但体验极差）
    
- 截图或录屏（必填）
    

**人工否决不可被覆盖**：即使 L1 / L2 / L3 全部通过，只要存在人工否决项未解决，结论为「阻断上线」。

### 六、打分算法方案

#### 6.1 设计思路

打分的目标不是给出一个「好看的分数」，而是**精准反映用户感知到的体验质量**。因此：

- 不同层级的问题权重不同（视觉问题 < 交互问题 < 阻塞任务问题）
    
- 存在「硬阻断」机制：某些问题无论其他分数多高，直接判定不可上线
    
- 分数仅作为参考和趋势追踪，最终结论由「是否存在阻断项」决定
    

#### 6.2 分层权重设计

总分 = 100 分，四层各有基础分，问题按严重程度扣分。

|层级|基础分|权重占比|说明|
|---|---|---|---|
|L1 视觉规范层|40 分|40%|视觉是用户最直接感知的质量|
|L2 行为规范层|30 分|30%|交互体验直接影响任务完成|
|L3 适配压测层|20 分|20%|覆盖面广但单个问题影响相对局部|
|L4 人工否决层|10 分|10%|兜底层，有否决项直接清零|

#### 6.3 各层扣分规则

**L1 视觉规范层（满分 40 分）**

- 方式一（无设计稿）：合规率每低于 90% 一个百分点，扣 2 分；合规率 < 80% 直接扣完 L1 全部分数并触发硬阻断
    
- 方式二（有设计稿）：每发现 1 个 P0 级还原偏差扣 8 分，P1 级扣 3 分，P2 级扣 0.5 分；存在任意 P0 级偏差触发硬阻断
    

**L2 行为规范层（满分 30 分）**

- 每发现 1 个 P0 级行为问题（含自动检测 + 人工补充）扣 10 分，存在任意 P0 级问题触发硬阻断
    
- 每发现 1 个 P1 级问题扣 3 分
    
- 每发现 1 个 P2 级问题扣 0.5 分
    
- 动效未完成人工确认：扣 5 分（不触发硬阻断，但计入分数）
    

**L3 适配压测层（满分 20 分）**

- 每发现 1 个 P0 级适配 Bug（任意机型）扣 8 分，存在任意 P0 级 Bug 触发硬阻断
    
- 每发现 1 个 P1 级适配问题扣 2 分
    
- 每发现 1 个 P2 级适配问题扣 0.5 分
    
- 极端场景压力测试未通过：每项扣 2 分
    

**L4 人工否决层（满分 10 分）**

- 存在任意人工否决项：L4 得分清零，且触发全局硬阻断（总分直接判定为「不可上线」）
    
- 无人工否决项：L4 得 10 分
    

#### 6.4 综合评级

|综合分数|评级|上线结论|
|---|---|---|
|90 ~ 100 分|🟢 优秀|可上线，建议作为质量基准|
|75 ~ 89 分|🟡 良好|可上线，P1 问题需本迭代修复|
|60 ~ 74 分|🟠 一般|建议修复后上线，P1 问题较多|
|< 60 分|🔴 较差|不建议上线，需系统性修复|
|存在硬阻断项|🚫 阻断|**无论分数多少，不可上线**|

> ⚠️ **重要说明**：评级仅供参考，**硬阻断优先于一切评级**。即使综合分数为 95 分，只要存在任意硬阻断项（P0 问题 / 人工否决），结论为「不可上线」。

#### 6.5 检测准确率要求

系统自动检测的准确率需达到 **98%**，定义如下：

- **精确率（Precision）≥ 98%**：系统召回的问题中，真实问题占比 ≥ 98%，避免误报干扰研发修复效率
    
- **召回率（Recall）目标 ≥ 85%**：真实存在的问题中，系统能识别的比例（剩余 15% 由人工兜底）
    
- 准确率低于阈值时，系统需在报告中标注「当前检测置信度不足，建议加强人工复核」
    

#### 6.6 分数趋势追踪

每次验收的分数自动归档，支持以下分析：

- **需求维度**：同一需求多次提测的分数趋势（是否在持续改善）
    
- **页面维度**：同一页面历史验收分数对比（防止劣化）
    
- **团队维度**：不同研发团队的平均验收分数（识别质量洼地）
    

---

### 七、验收结论输出格式

每次验收自动生成结构化报告，包含：

1. **综合评级**：分数 + 评级 + 上线结论（一句话）
    
2. **各层得分明细**：L1 / L2 / L3 / L4 分别得分及扣分原因
    
3. **硬阻断项清单**：若存在，逐条列出，标注修复优先级
    
4. **人工补充问题**：验收人员追加的问题列表
    
5. **适配报告**：各机型截图 + 问题标注
    
6. **置信度说明**：标注当前检测方式（新容器/旧容器）及准确率
    

---

### 八、上线结论判定（精简版）

**可上线的充要条件**：

1. L1 无硬阻断（有稿：无 P0 还原偏差；无稿：合规率 ≥ 90%）
    
2. L2 无 P0 行为问题（自动检测 + 人工补充均清零）
    
3. L3 无 P0 适配 Bug（全机型矩阵通过）
    
4. L4 无人工否决项
    
5. 综合分数 ≥ 75 分（低于 75 分即使无硬阻断也建议修复后上线）
    

> 💡 **关于 P2 建议项**：不阻断上线，但需在报告中记录。连续 3 个迭代未修复的 P2 项自动升级为 P1 项。
