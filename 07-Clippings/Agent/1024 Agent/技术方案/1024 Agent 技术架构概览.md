## 一、整体架构

1024 作为Agent托管平台， 整体架构分为三层：接入层、Harness 层和基础设施层。

![[Pasted image 20260513110022.png]]

### 1.1 接入层

接入层负责承接所有进入系统的用户请求，目前有两个入口。

**大象机器人**：主要的使用场景，覆盖私聊和群聊两种模式，负责 IM 场景下的实时对话。

**OpenAPI**：面向外部系统提供标准化接入能力，支持三种调用方式：同步调用、异步调用和流式（SSE）实时推送，内置了限流保护，控制请求频率，防止单个调用方打爆服务。

### 1.2 Harness 层

Harness 层是系统的执行核心，所有 Agent 的推理、工具调用和上下文管理都在这一层完成。

### 1.3 基础设施层

基础设施层提供数据持久化和检索能力：

- MySQL：存储会话、消息记录、Agent 配置和工具调用日志，支撑平台管理和数据回溯
    
- EFS 文件存储：用户的工作空间，Skill 文件、对话 Session、记忆文件、用户工作区文件都存在这里，跨会话持久保留
    
- Redis：缓存 Agent 配置、API Key 映射等高频读取的数据
    
- AI 数据库：存储 Agent 执行过程中产生的业务数据
    
- Milvus（向量数据库）：支撑记忆系统的语义检索，用向量相似度找到与当前问题最相关的历史记忆
    

## 二、核心流程

一次完整的 Agent 对话，从用户发消息到拿到回复，经过以下节点：

![[Pasted image 20260513110227.png]]

1、请求进来后，Session 模块先创建或恢复会话，加载历史消息。

2、记忆系统同时检索与当前对话相关的长期记忆，把检索结果注入上下文。

3、然后 Agent Loop 带着完整上下文和工具列表发给 LLM 推理。

LLM 每轮推理有两种结果：直接给出最终回答，或者决定调用某个工具。直接回答时，结果流式推送给用户，同时写入会话日志，本次对话结束。决定调用工具时，工具系统并行执行所有被调用的工具，普通工具（文件读写、语义搜索等）同步返回结果，run_subagent 则异步派发给子 Agent 执行。工具结果追加到上下文后，进入下一轮推理，如此循环直到 LLM 不再发起工具调用。

## 三、部署架构

![[Pasted image 20260513110316.png]]

Agent Loop 是无状态的。每次推理所需的上下文，消息历史、工具结果、会话状态，全部持久化到外部存储，不依赖本地内存。这意味着任意节点崩溃，请求可以立即路由到其他节点继续执行，不丢数据。

无状态设计是水平扩展的基础：业务量增加时直接加节点，每个节点等价，负载均衡器随机分配即可，不需要做任何有状态的迁移。状态全部集中在基础设施层管理，Agent Loop 集群只负责计算。

## 四、租户隔离体系

**最关心的问题通常是：数据安不安全？别人能不能看到我的文件？不同业务的 Agent 会不会互相干扰？**

1024 的隔离体系分三个维度：Agent 隔离控制每个 Agent 的能力边界，群空间隔离控制群维度的数据归属，用户隔离控制个人维度的数据归属。

### 4.1 Agent隔离

1024 平台支持部署多个 Agent，每个 Agent 是一个独立的租户。不同 Agent 之间的数据、配置、能力完全隔离：

- 文件存储隔离：每个 Agent 的用户文件存在独立的租户目录下，跨 Agent 的目录访问在文件系统层直接拦截
    
- 工具能力隔离：每个 Agent 的工具集独立配置，A Agent 的工具对 B Agent 不可见，也无法调用
    
- Skill 隔离：每个 Agent 在数据库里绑定自己的 Skill，各 Agent 的 Skill 互不干扰
    
- 目录枚举防护：租户根目录、用户列表目录、群组列表目录均禁止非管理员遍历，防止通过列目录的方式探测其他用户或群组的存在
    

### 4.2 群空间隔离

群聊场景下，Agent 运行在群维度而非个人维度：

- 文件访问隔离：群目录只有当前群的成员才能访问，其他群的目录访问直接被拦截，系统校验路径中的群 ID 是否与当前会话一致
    
- Session 隔离：每个群有独立的 Session 文件，群内所有成员共享同一上下文，但与其他群完全隔离
    
- Skill 隔离：群内安装的 Skill 存在群目录下，只对本群生效，不影响其他群或个人私聊
    
- 记忆隔离：群记忆文件存在群目录下，仅本群的会话可读写
    

### 4.3 用户隔离

每个用户在自己所属的 Agent 下有专属的个人目录：

- 文件访问隔离：Agent 执行文件操作时，系统校验路径中的用户身份是否与当前登录用户一致，不一致直接拒绝，代码层硬拦截，不依赖模型判断。
    
- 沙箱隔离：每个用户有独占的沙箱实例，代码执行环境和文件对其他用户完全不可见
    
- 记忆隔离：个人记忆文件存在个人目录下，仅当前用户的会话能读取和写入
    

## 五、Harness 设计

Harness 是 1024 的执行骨架，下面按模块分别描述各自的设计。

### 5.1 Agent Loop

用户发来一条消息，Agent 不是直接回答，而是进入一个"推理 → 工具调用 → 推理"的循环，反复迭代直到任务完成。

Agent Loop 实现 ReAct 循环：Think（思考）→ Act（行动）→ Observe（观察），循环往复直到任务完成。每轮 Think 把当前上下文和工具列表发给 LLM，LLM 要么给出最终回答，要么决定调用工具。Act 执行工具调用，Observe 把结果追加到消息列表，开启下一轮 Think。

循环有三个终止条件：LLM 不再发起工具调用（正常完成）；达到最大轮次上限（流式默认 35 轮，非流式默认 20 轮，通过配置中心动态调整，不需要发版）；检测到模型反复工具调用失败时，主动清空工具列表，引导模型用现有信息直接回答，避免卡死在循环里。

每轮推理过程实时流式推送给前端，用户可以看到 Agent 正在思考什么、调用了哪个工具、拿到了什么结果，而不是等到最后才拿到一个黑盒结果。

### 5.2 Session管理

Session 管理每次对话的生命周期：会话创建、消息历史持久化、对话日志记录。每条消息、每次工具调用的入参出参、每轮推理的完整过程，都结构化地写入数据库，支持事后回溯和审计。用户跟 Agent 说话，Agent 不能"只记得这一句"。Session 管理就是让 Agent 记住这一次对话里发生的所有事，你问了什么、它怎么思考的、调用了哪些工具、最终回复了什么。

|动作|描述|
|---|---|
|**记录对话历史**|每轮对话结束后，系统会把用户消息和 Assistant 回复逐条写入 EFS 上的 JSONL 文件。写入不是粗粒度的"问答对"，而是完整保留了中间的工具调用过程：Agent 每次调用工具，调用了什么、参数是什么、结果是什么，都会作为一条记录存下来。用户消息在推理开始前就先写入，Assistant 回复在推理结束后写入，这个顺序设计是为了解决并发问题：下一条消息来的时候，上一轮的用户问题已经落盘，不会丢。|
|**多轮上下文恢复**|下次对话开始时，系统读取当前活跃会话的历史记录，把它注入到发给大模型的 prompt 里，Agent 就能"记得"之前说过什么。读取时有截断逻辑，只取最近 N 轮（默认 10 轮，可通过 Lion 配置调整），避免历史过长撑爆 context window。|
|**活跃会话追踪**|每个用户（或群）都有一个 .sessionId 文件，记录当前"在聊哪个会话"。用户发起新对话时，系统覆盖写入新的 conversationId，后续的上下文读写都基于这个 ID 进行。|
|**同步记录到数据库**|除了 EFS 文件，每轮对话的消息和工具调用记录也会同步写入 MySQL（ai_message_log、ai_tool_call_log 表）。EFS 是给 Agent 下次对话用的，数据库是给平台侧用的，比如用量统计、调试回溯、历史查询。|

**对用户的价值，一句话：让 Agent 有连续记忆，不用每次重头解释。**

用户在私聊里说"帮我写一个 Python 脚本处理 CSV"，Agent 写完了；下一条说"再加个参数支持按列过滤"，Agent 知道你在说什么。没有 Session，每次都是白板，用户必须反复把背景交代一遍。

群聊里，群内所有人的消息都写进同一个 Session 文件，群成员共享上下文，相当于大家在同一个对话里协作，不会出现"甲说了一半、乙问问题、Agent 忘了甲在说什么"的割裂感。

### 5.3 工具调用

用户的需求往往不是"问一个问题拿个答案"，而是"帮我做一件事"：查数据、写文件、跑代码、搜文档。工具调用就是 Agent 的"手"，让它能真正触达外部系统，把事情做完。

#### **5.3.1 Skill**

Skill 是平台最重要的扩展机制。它的本质是把一段专业的执行指令（Markdown 格式的提示词）打包成一个可调用的工具，让 Agent 在面对复杂任务时，不用自己从零摸索步骤，而是激活一套已经设计好的工作流。调用方式很简单：Agent 只需传入 Skill 名称，系统就会把对应的完整指令内容作为 tool 消息返回，Agent 读完之后按步骤执行。Skill 的内容不做任何 Token 截断，因为截断意味着步骤残缺，任务无法完成。

Skill 来自三个地方，优先级从高到低：

- Workspace Skill：用户通过 /install 指令从 Friday 平台安装，私聊装到个人目录，群聊装到群目录。群聊里群 Skill 优先于个人 Skill。
    
- Agent 关联 Skill：平台管理员在后台为某个 PaaS 配置，存在数据库里，所有该 PaaS 的用户共用。
    
- 平台默认 Skill：平台级的通用能力，所有 PaaS 默认可用，各 PaaS 可单独排除不需要的。
    

每次对话开始前，系统会把三层 Skill 合并去重，注入 Skill 元工具的描述里，大模型在工具列表里看到的是一张完整的 Skill 目录。

#### **5.3.2 内置工具**

除了 Skill，平台还提供一组内置工具，覆盖核心的执行能力：

- 代码执行：bash，统一入口，支持 shell 命令、Python、Node.js
    
- 文件操作：read、write、edit（精确片段替换）、apply_patch（批量跨文件变更）
    
- 文件传输：上传到 S3、从 URL 或大象 IM 下载文件
    
- 知识检索：semantics_search（语料库语义搜索）、codebase_search（代码库语义搜索）
    
- 子任务调度：run_subagent，启动子 Agent 并行处理独立子任务，仅主 Agent 可用，防止嵌套递归
    

#### 5.3.3 MCP 工具

通过 MCP 协议挂载的外部扩展能力，各 PaaS 可以按需接入自己的业务工具。

### 5.4 上下文管理

上下文管理解决的是一个根本性问题：大模型每次调用都是无状态的，但用户的对话是连续的。上下文管理负责把"过去发生了什么"以合适的形式喂给模型，让它能接着上次的思路继续工作，同时不超出模型的 Token 上限。

#### 5.4.1 上下文组装

每次推理开始前，系统把本轮的消息列表拼好交给大模型。历史消息直接从 Session 文件里读取，内容是完整的消息块：user、assistant、工具调用过程、工具结果，一条不少。按对话轮次从后往前截取，保留最近 N 轮（Lion 配置 context.maxRounds，默认 10），截断点严格落在轮次边界，不会把一个工具调用拆成两半。

System Prompt 单独注入，支持动态参数替换，每次请求时自动填入当前用户的上下文信息。有一个细节：当前时间不写进 System Prompt，而是注入到 user 消息里，原因是 Claude 的 Prompt Cache 以 system 消息为基准，时间每分钟都在变，写进去会导致缓存全部失效，白白多花 Token。

#### 5.4.2 上下文压缩

随着对话轮次增加，上下文会越来越长。系统有两级防线：**工具剪枝和历史消息压缩。**

**工具剪枝**是日常防线，在每轮推理前执行。从当前用户消息往前回溯，统计历史工具调用结果的 Token 占用，保留最近 40k Token 的工具结果（Lion 配置 compaction.pruneProtect），超出部分替换为占位文本"旧工具结果已清除，如需可重新获取"。只有待剪枝内容超过 20k Token 才真正执行，避免频繁触发。

**历史消息压缩**是应急防线，在 LLM 实际返回溢出信号之后触发。系统检测到 Token 使用量接近模型上限时，把当前用户消息之前的所有历史消息送给 LongCat-Flash-Chat 模型生成一段摘要，用这段摘要替换历史消息，当前轮次的消息原样保留继续推理。摘要以 [历史摘要] 标记存储，后续轮次重建上下文时识别到这个标记，从摘要位置开始，不再往前找已被压缩掉的旧消息。

两条机制都有保护：Token 紧张如果只是工具 Schema 占用大、消息内容本身不多，不会触发对消息内容的无效压缩。

### 5.5 沙箱调用

沙箱解决的是代码执行的安全问题：执行 bash 工具时，命令不直接跑在服务器上的，而是运行在一个独立的沙箱环境。

沙箱基于 E2B 平台构建，每个用户有一个专属的沙箱实例，相互完全隔离。沙箱是按需创建的，默认超时 20 分钟，只要用户持续使用，系统会自动续期，超过 20 分钟无操作才会回收。用户下次再用时，如果旧沙箱还活着就直接复用，挂掉了则自动重建。

Agent 在沙箱里能做的事：

- 执行任意 shell 命令和脚本
    
- 读写文件，分析处理数据
    
- 把沙箱内生成的文件上传到 S3，返回可分享的链接
    
- 从外部 URL 下载文件到沙箱供 Agent 处理
    
- 操控浏览器，完成网页交互类任务
    

首次使用需要授权：沙箱与用户身份绑定，首次使用时需要完成一次 MOA 授权，授权后 Agent 才能以用户自己的身份访问公司内网服务，后续不需要重复操作。如果沙箱因为某些原因被重建，需要重新授权一次。

### 5.6 记忆系统

每次新建会话都是全新启动的，用户不得不反复交代背景："我是做什么的"、"我喜欢什么风格的回答"、"上次我们讨论到哪了"。Session 只保留最近几轮对话，开了新会话就清空；用户积累下来的偏好、习惯、业务知识，应该跨对话长期保留，记忆系统可以让 Agent 真正认识用户、记住过去。

具体方案：[1024 Agent 记忆功能方案](https://km.sankuai.com/collabpage/2757637705)

#### 5.6.1 记忆存在哪

记忆以文件形式存在用户的个人工作区里，主要是这几个文件：

- MEMORY.md：长期记忆。存放提炼后的重要信息——用户的偏好、工作背景、重要决策、反复出现的约定。这是 Agent 认识用户最核心的文件，每次主会话启动时都会读取。
    
- USER.md：用户档案。记录用户的基本信息、正在做什么项目、关心什么、什么让他们烦恼。Agent 了解得越多，帮助就越有针对性。
    
- SOUL.md：Agent 的人格设定。定义这个 Agent 的说话风格、行为准则、个性特征。用户可以自己编辑，打造专属的 Agent 风格。
    
- memory/YYYY-MM-DD.md：每日日志。发生的事情的原始记录，Agent 会在每次启动时读取今天和昨天的日志，了解近期发生了什么。
    

#### 5.6.2 记忆怎么写

记忆写入完全由 Agent 主动触发，没有自动机制。触发时机有几种：

- 用户说"记住这个"、"帮我记一下" → Agent 判断写 [MEMORY.md](http://MEMORY.md) 还是每日日志，偏好和长期约定写前者，临时事件写后者
    
- 对话中产生了值得沉淀的结论、教训或决策 → Agent 自行判断写入
    
- Agent 犯了错误 → 应该记下来，避免下次重蹈覆辙
    

Agent 每隔几天还会做一次"记忆维护"：翻看最近的每日日志，把值得长期保留的内容提炼进 [MEMORY.md](http://MEMORY.md)，同时清理其中已经过时的内容。

#### 5.6.3 记忆怎么用

每次对话开始时，系统把用户工作空间的这些文件内容注入到 System Prompt 里，Agent 读完就知道"这个用户是谁、有什么偏好、最近在做什么"，不需要用户重新介绍自己。注入有总量上限，文件过大时按"保头保尾"策略截断，Agent 可以用 read 工具读取完整内容。

### 5.7 多 Agent 协作

复杂任务可以拆解成多个相互独立的子任务，比如同时查多份文档、并行处理多个代码模块。多 Agent 协作就是为此设计的：主 Agent 把子任务分派给多个子 Agent 并行执行，自己不阻塞等待，全部完成后汇总结果继续推进。

具体方案：[1024 Custom Subagent 设计方案](https://km.sankuai.com/collabpage/2758059054)

#### 5.7.1 派发机制

主 Agent 通过 run_subagent 工具派发子任务。调用后系统立即返回一个占位符，主 Agent 不等待，可以继续推理或同时派发更多子任务。子 Agent 在后台独立跑完整的 ReAct 循环，它有自己的工具调用能力，能查文件、跑代码、调接口，和主 Agent 没有区别。

子 Agent 完成后把结果写入共享队列。主 Agent 每轮推理开始时检查队列，有结果就把之前的占位符替换成真实内容，再触发新一轮推理。如果同时派发了多个子任务，必须等所有子 Agent 都完成，主 Agent 才会继续，避免拿到不完整的信息做出错误判断。

#### 5.7.2 限制设计

子 Agent 有完整的 ReAct 执行能力，但做了几处关键限制：

- 不能递归派发：子 Agent 的工具列表里没有 run_subagent，从工具层和代码层双重拦截，防止无限嵌套
    
- 上下文独立：子 Agent 只看到自己的任务描述，父 Agent 的历史消息不传入，保持上下文干净
    
- 有超时保护：每个子 Agent 在派发时可以指定超时时间（run_timeout_seconds），超时后返回终止消息，不会无限挂起
