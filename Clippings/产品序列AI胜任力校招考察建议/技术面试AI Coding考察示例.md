### 考察目的

AI编程考察并非是传统面试的替代，而是面试工具之一。我们可以从候选人与面试官、与AI的交互过程中，有效考察到候选人在<strong>学习并好奇、苦练基本功</strong>维度的基本素养。

1.<strong>学习并好奇</strong>：候选人能否在有限时间内掌握新工具，能否理解AI工具的边界与适用场景，是否关注AI编程的底层原理

2.<strong>苦练基本功</strong>：候选人能否理解需求并给出解决方案，能否将需求拆解为AI可执行的Prompt，能否识别AI生成代码的错误或漏洞

### 考察前准备（面试官）

1.熟悉catpaw、cursor等AI编程工具的使用环境

2.准备并熟悉题库

### 考察示例

#### 场景一：考察AI编程经验少的候选人使用AI优化代码

| <strong>考察背景</strong> |
|---|
| 人选较少使用AI编程工具，面试中，面试官逐步引导候选人使用AI编程工具优化代码、验证优化效果 |
| <strong>出题</strong> |
| 尝试编写一段代码，实现路径规范化的功能。具体任务是编写一个函数 `canonicalize(string path)`，用来处理路径字符串。<br>在 Linux 环境中，经常需要处理路径变化的问题，比如：<br>`cd /foo/bar`：用于切换目录。<br>`cd ..`：用于进入上级目录。<br>`cd .`：表示当前目录。<br>函数的目标是输入一个路径字符串 `path`，返回其规范化的形式。例如：<br>输入 `/foo/bar/..`，返回 `/foo`。<br>输入 `/foo/bar/.`，返回 `/foo/bar`。<br>你可以借助任何编程语言实现这个函数。 |
| <strong>追问思路</strong> |
| <strong>1.引导候选人使用cursor优化手写的代码</strong><br>"选中这段目录规范化函数（约50行C++代码），用Ctrl+K触发AI建议，逐条评估提出的优化方案"<br>"重点讨论`string_view`避免字符串拷贝的可行性，分析其与数据库视图的相似性"<br>"对AI建议的`move`语义进行验证，分析其与`const`的冲突问题"<br><strong>2.判断AI优化思路的可取性</strong><br>"性能优化边界：字符串拼接操作能否合并到处理流程中？为什么最终未采纳？"<br>"工具局限性：大代码库场景下AI的上下文长度限制具体表现是什么？"<br>"若用`deque`替代`stack`，如何避免最终的反转操作？请解释数据结构选择的影响"<br><strong>3.验证优化效果</strong><br>"预分配空间对性能提升的实际影响有多大？请对比扩容开销"<br>"`string_view`优化后，为什么55行仍需创建新字符串？能否通过swap进一步优化？"<br>"若去除`const`限制使用`move`，可能引发哪些风险？"<br><strong>4.考察候选人的设计思维</strong><br>"并发处理路径规范化时，如何解决退栈操作的顺序依赖问题？"<br>"实时拼接路径的挑战在哪里？为什么说字符串移位开销更大？" |

#### 场景二：考察候选人利用AI解决业务场景问题

| <strong>考察背景</strong> |
|---|
| 面试官与候选人讨论了mysql主从延时问题，由此提问候选人，“如果要写一个工具来检查公司里面所有的集群组成延时的情况，你会怎么做？” |
| <strong>出题</strong> |
| 要求候选人编写一个程序来检查分布式缓存系统中热点数据在多个节点上的一致性，以规模化手段验证数据对齐情况 |
| <strong>追问思路</strong> |
| <strong>1. 候选人理解偏差时，简化任务，聚焦问题核心</strong><br>候选人将问题误解为检查延迟的具体步骤时，通过问题转换（从MySQL主从延迟转换为分布式缓存热点数据一致性）来简化任务，聚焦一致性检查<br>“就比如说你现在有一个分布式缓存也可以吧？我们把这个题目稍微换一下”<br><strong>2. 针对性提问候选人的逻辑漏洞，引导候选人给出新方案</strong><br>候选人提出分区方案，但未说明如何减少数据量时，追问“为什么这样会变少？”<br>反问“所以我们就只验一个分区的，不用验所有分区的，是这个意思吗？”，引导候选人意识到需解释分区逻辑（如只验部分数据），促使其补充优先级策略（先验热点数据）<br><strong>3.让候选人结合AI工具功能优化代码</strong><br>当候选人讨论代码实现时，鼓励候选人按新方案思路，使用cursor生成或优化代码 |

#### 场景三：考察有AI编程经验候选人的使用深度

| <strong>考察背景</strong> |
|---|
| 候选人主动投屏展示DeepSeek，有丰富的使用记录。面试中聚焦候选人之前使用AI处理的一个较难问题，延伸提问 |
| <strong>出题</strong> |
| 询问候选人此前用AI工具求解时，花费时间最长的问题是哪一个？<br>（候选人提到使用DeepSeek来理解索引下推在数据库优化中的概念）<br>聚焦索引下推的技术难点延伸提问，候选人可借助AI作答 |
| <strong>追问思路</strong> |
| <strong>1.要求候选人评价AI的回答是否准确</strong><br>“你怎么看deepseek给的解释”<br><strong>2.引导候选人深度提问AI，并对AI的答案做出解释，考察基本功</strong><br>让候选人举例说明索引下推如何减少回表次数，可借助AI作答<br>AI举例索引失效或复杂表达式等不适用场景后，让候选人回答不适用场景的原因<br>深度质疑候选人的回答，进一步考察底层理解<br><strong>3.扩展到实际代码能力，让候选人提出优化方案</strong><br>可借助AI实现一个函数（如`canonicalize`），输入一个文件路径字符串（可能包含点`.`表示当前目录和点点`..`表示上一层目录），输出规范化后的路径字符串（去除`.`和`..`符号）<br>要求候选人提出优化方法（如使用线程池），并讨论线程池实现细节（如为何避免使用`Executors.newFixedThreadPool`，推荐自定义线程池）<br>询问`ConcurrentHashMap`的好处（如避免并发问题）及潜在风险，并解释`Future`对象的作用（如代表异步计算结果，用于任务状态检查）<br>探讨设置线程数为CPU核心数的原因（针对计算密集型任务），并深入询问“N+1”线程数策略的含义及适用场景<br>结合线程池大小，讨论超线程概念及逻辑核心与物理核心的关系 |
