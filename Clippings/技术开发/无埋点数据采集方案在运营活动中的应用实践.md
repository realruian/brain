# 无埋点数据采集方案在运营活动中的应用实践

> 来源：[https://km.sankuai.com/collabpage/170983430](https://km.sankuai.com/collabpage/170983430)

> ---
作者：@孙光

### 引言

流量分析和用户行为分析是一款产品或一次运营活动进行效果回收时必不可少的环节。业务开发中，工程师依据产品打点需求，手动在业务代码中增加埋点逻辑来实现数据回收。目前公司内，我们有较为成熟的一站式打点解决方案：灵犀打点[OCEAN](https://ocean.sankuai.com/)。灵犀打点利用通道(lx:category)、应用(lx:appnm)、页面(cid)和模块(bid)标识来规范每次的打点上报行为。根据这些标识信息在[OCEAN](https://ocean.sankuai.com/)平台上就可以进行打点数据的查看。

为了降低打点的开发成本，部分团队还基于灵犀的打点进行了相应的封装：自动化cid和bid的申请过程，暴露更便于开发理解的参数和API等等。利用封装后的打点工具进行打点时并不复杂。但对于打点需求强烈的场景：如运营活动页面。直接应用前述的开发手动进行灵犀打点的方案，实际中会存在一些问题。

### 问题分析

运营活动这类对于打点需求强烈的场景，在实际打点中，可能会遇到下述情形：       

- 产品打点需求文档中，存在漏点的情形。

- 事后发现漏点，需要开发重新埋点。重新增加打点后，之前的漏点数据无法捞回。打点数据永久丢失。

- 运营活动等对埋点数据敏感的场景，对丢点忍耐度低。

- 打点功能需要业务开发。

- 部分通用型的埋点需求，在每次需求文档中，都需要进行明确。否则业务开发有可能遗漏。每次打点需求都有一定的沟通成本。

- 业务开发周期短，业务数量大。埋点代码侵入业务逻辑。整体打点需求有一定的开发成本。

根据上面的分析，我们梳理了以下需要解决的问题：

- 如何解决漏点问题？

- 漏点问题可以通过实施全量打点的方案来解决。如何在页面中实施全量打点？

- 全量打点方案下：上报的埋点数据之间如何区分？

- 打点数据如何进行数据捞回？按照什么样的规则？规则如何配置？

- 页面布局发生变化，数据捞回时如何进行聚合？

- 如何简化业务打点开发？

### 解决方案

为了解决漏点问题，最大化的简化打点需求的开发成本，简化打点数据的查看和回收。我们设计了一套基于无埋点技术的数据采集方案。所谓的无埋点指的是开发人员只需要引入埋点SDK即可，无需在业务代码中进行手动埋点。开发只需针对目标页面，在平台上配置好圈定规则后就可以获取对应的埋点数据。这一整套全量无埋点打点技术方案，包括：全量打点SDK模块、捞点规则配置、数据清洗三个模块。通过各个模块协调配合，共同解决上文提出的问题。

整体方案的设计如下图所示：

> *[流程图: https://km.sankuai.com/api/file/cdn/170983430/170947096?contentType=0&isNewContent=false]*

- 全量打点SDK：完成与灵犀打点相关的环境配置和基础配置。页面内注入全量打点逻辑，对打点行为进行标识。并在捞点规则配置时处理与配置平台的消息通信。

- 捞点规则配置：自助打点数据的取数规则的配置过程。对规则进行规则标识。

- 数据清洗：根据点击点实际上报的行为标识与配置的捞点规则标识，采用一定的匹配策略清洗数据。

### 方案实施

上文中提出的问题，通过各模块的协调配合得以有效地解决，但各个模块的实现仍然需要进一步深入思考，设计合理的实现方案。本章将对方案的三个核心模块进行详细介绍。

#### 全量打点SDK

这一模块主要包括：灵犀基础配置、全量打点（含打点标识部分）和消息通信三个部分组成。灵犀基础配置部分主要依赖于灵犀的Web SDK接入文档来进行配置。全量打点背后依托的还是灵犀打点。因此必须首先接入灵犀打点。这一部分的配置参照[SDK文档](http://corn.sankuai.com/docs/lib/sdk-introduce)即可，本文不做详述。

- 全量打点

全量打点的实施，较为简单。通过在document上面，绑定click事件，我们就可以借助灵犀打点API对用户在页面上所有的点击行为进行打点数据上报。

```JavaScript nodeId="50db580b58a94bafa0df8a4082dad8cd"
document.addEventListener('click', 
   LXAnalytics('moduleClick', bid, {
      custom: {
        locationId: '7835939e-1114-4d17-aa65-b7666e0f3279/DIV/DIV/DIV/P#南京东路830号第一百货五楼',
      }
	 });                     
);
```

上述代码中，可以看出：在打点上报行为中，我们定义了locationId这个自定义参数来区分标识打点数据。

- ##### 打点标识

由于是全局打点，所以我们不能通过灵犀给的bid字段来标识点击事件发生的模块。因此，在全量打点中，所有的点击事件均共用一个模块bid。我们采用点击事件发生的位置来标识打点。具体地：打点行为采用类X-Path路径(location_path)、模块ID(location_id)、点击目标内容(location_innerText)三个参数组合在一起进行标识。下图展示了该自定义参数的基本组成：

> *[流程图: https://km.sankuai.com/api/file/cdn/170983430/170936157?contentType=0&isNewContent=false]*

**类XPath路径**：利用点击事件的冒泡特性，在点击事件发生后，通过[e.path](https://developer.mozilla.org/zh-CN/docs/Web/API/Event/composedPath)（e.composedPath）获取到按照事件冒泡顺序构成的NodeList节点集合。

![](_attachments/170983430_170927314.png)

基于这个NodeList集合，遍历之后就可以获取目标点的XPath：/tagName/tagName1/tagName2...。

> *[流程图: https://km.sankuai.com/api/file/cdn/170983430/170927315?contentType=0&isNewContent=false]*

**模块ID**：事件冒泡会从目标元素逐层冒泡到window上。模块ID的定义是为了优化X-path路径的生成过程。避免每次搜寻过程都搜索到window。考虑到组件化开发已经较为普及，我们在每个组件模块容器上定义一个id或者data-component-id标识。该标识可以优化Xpath的生成过程。模块ID的另外一个用途是：区分出两个完全相同的组件实例。如在一个页面中，有一个组件被使用了两次。在点击事件发生时，就可能存在两次Xpath路径相同的情况。每次初始化一个组件，对应产生唯一的组件标识ID，可以唯一的进行区分。

**目标内容**：目标内容指示的是点击的目标内容。通常情况下，采用点击事件发生时的InnerText(e.target.innerText)即可。但并不是所有的点击事件都对应有innerText，针对A标签和IMG标签这类场景需要进行特殊处理。这里进行了特殊处理：将A标签的href属性与IMG标签的SRC属性作为当前点击行为的目标内容。

#### 捞点规则配置

页面接入全量打点SDK后，页面内每次的用户点击行为都将被记录下来。每条打点记录都在灵犀custom字段中对应有自定义的locationId字段。现在的关键问题是：如何根据实际的打点需求将数据捞取出来呢？根据无埋点数据采集方案的设计思路：需要配置一定的数据规则来匹配数据。为了获取对应的数据规则，就需要将产品提出的打点需求文档映射出对应的打点规则。传统上通过开发业务埋点来完成这部分工作。实际就是开发同学通过埋点代码来完成了需求到规则的映射。在实施全量打点之后，肯定不能继续依赖开发工程师来映射规则了。这里采用了一种自助式的规则配置过程。

- ##### 自助规则配置

打点需求对应了一个规则，规则的本质就是前文所述的locationId，这里方案用自定义的conditionId进行标识。与之对应的，每个conditionId应由：规则Id(condition_id)，规则路径(condition_path)，规则目标内容(condition_innertext)组成。conditionId的生成逻辑，可以采用全量打点SDK中的生成locationId方法进行生成。规则标识和打点标识的数据是一一对应的。

实际配置中，只需要主动地重新模拟一次用户点击行为，就可以完成规则配置。这里的重点是如何简化这个配置过程。我们采用了一种通过鼠标hover点击选择勾选目标区域的方式来简化规则配置。下图展示了实际规则配置平台的配置过程。

- ##### 消息通信

自助规则的配置过程中包含了全量SDK与配置平台之间大量的交互通信。当待设定规则的目标页面在配置平台打开时，会和配置平台建立消息通信机制。消息的通信主要通过iframe的postMessage来完成。

#### 数据清洗

数据清洗是最后依据规则从灵犀表中捞取数据的过程。这里基于灵犀的宽表(mart_flow.detail_flow_mv_wide_report)，利用协同任务初步筛选出打点数据转存到中间表中。然后基于中间表，再按照规则进行获取。这里重点阐述点击事件对应的行为数据清洗过程。根据conditionId规则字段与locationId字段的是否匹配来筛选数据。匹配规则取决于：规则标识id与打点标识id、规则标识路径与打点标识路径以及规则目标内容与打点标识目标内容之间的对应关系。

- ##### 模糊匹配

模糊匹配是指规则标识condition_id和打点上报的数据location_id相同，且condition_innertext和location_innertext相同时，condition_path和location_path表现为包含关系。这时我们认为规则命中匹配了这个打点数据。如下图所示。

> *[流程图: https://km.sankuai.com/api/file/cdn/170983430/170964107?contentType=0&isNewContent=false]*

举例来说：

规则配置数据: 7835939e-1114-4d17-aa65-b7666e0f3279/DIV/DIV#南京东路830号第一百货五楼 

condition_id =  7835939e-1114-4d17-aa65-b7666e0f3279

condition_path = /DIV/DIV

condition_innerText = 南京东路830号第一百货五楼 

数据点击上报数据：7835939e-1114-4d17-aa65-b7666e0f3279/DIV/DIV/DIV/DIV#南京东路830号第一百货五楼

location_id = 7835939e-1114-4d17-aa65-b7666e0f3279

location_path = /DIV/DIV/DIV/DIV

location_innerText = 南京东路830号第一百货五楼

上例数据点的 location_path 包含在规则的 condition_path 之内，且id和innerText 相等，所以最终会被捞点规则匹配。

#### 兼容主动埋点

方案描述了一种全量无埋点的数据采集方案。在实际应用中，开发业务自身也会对一些模块进行一些自定义埋点。方案采取了约定的规则来兼容业务主动打点和方案全量打点。具体地，采用在DOM元素上增加data-bi属性。页面开发者主动将自己的业务打点信息通过data-bi的自定义属性添加到DOM元素上，如下图所示。

### 方案应用

除了上述整体方案外，为了让开发者使用起来更加方便，我们还进行了一些配套工作：

- [官网建设](http://corn.sankuai.com/)：维护了全面的使用指南，包括[平台介绍](http://corn.sankuai.com/docs/lib/introduce)、[玉米SDK介绍](http://corn.sankuai.com/docs/lib/sdk-introduce)，[玉米平台使用指南](http://corn.sankuai.com/docs/lib/guidelines)等详细说明。

- 工程化集成：容量仅有3KB的全量打点SDK，直接集成在活动页的脚手架中，自动成为团队的默认方案，在工程初始化时即可完成接入。

- 持续维护升级：响应运营和开发的持续需求，持续进行[迭代升级](http://corn.sankuai.com/docs/versions/changelog)。

- 常见使用问题: 拥有专门的客服支持群，支持运营开发产品在使用或接入时遇到问题快速响应处理。

目前，整套方案已经接入了广告平台运营活动的项目开发中。基于平台分析，我们收集得到以下数据：

- 方案支持了品牌、到餐、KA等多个业务方的运营活动。

- 方案已接入个800+左右的活动页开发。

- 已配置平台捞点规则：144+个。

总体来看，方案的实施可以一定程度上地简化业务打点开发需求，避免开发漏点对运营活动造成不可挽回的损失。

### 规划和思考

上述方案到目前已经在稳定应用，除了已经实现的功能，我们也在思考是否可以更进一步，提供更丰富的能力，由此规划了一些仍未落地的方向：

- 扩展对innerText的兼容性。虽然方案已经支持对普通文本、A标签和IMG标签的处理。但是对于一些情况仍没有覆盖：通过标签style属性，指定背景图片地址等。这类情况会造成方案无法正确获取目标内容。需要case by case的梳理出来，并进行针对性的处理，避免数据清洗时，规则不匹配造成数据无法正确捞回。

- 梳理开发或运营配置的捞点规则。分析平台配置捞点规则的特点，争取将打点需求下沉到组件层面。这样可以增加组件的复用效率，避免重复配置，重复打点。

- 目前方案支持的是全量点击行为的数据采集，对于模块层面的曝光mv数据采集还不支持。如果要采集这部分数据，目前只能靠开发手动埋点，后续从方案维度支持对模块组件曝光数据的自动采集。

- 进一步完善数据展示环节。目前平台默认支持了对流量数据(PV  UV 有效PV等)和行为数据(点击事件)的支持。但在使用方案完成打点需求时，业务方多有一些增加自定义参数的需求。目前平台并未支持，只能通过主动去数据库里拉取数据来实现。考虑数据展示的完整性，后续考虑支持这部分需求，这样使得平台可以基本囊括住运营所需的打点数据需求。

上述便是广告平台前端团队在全量打点应用过程中的一些实践，欢迎大家提出建议、一起沟通交流。
