# 点评 AI Native 研发工作流实践：从思考到落地

| 本文作者：@高超


#### 引言

AI Coding 工具已从代码补全演进到自主 Agent（详见[附录 A](https://km.sankuai.com/collabpage/2748610032#b-dc102bc651854139856947221914e0fd)），个体编码效率获得了质的提升。但在企业层面，研发流程仍停留在前 AI 时代——工具链碎片化、AI 使用效果方差大、缺乏业务上下文、大量人工交接环节。当 AI Agent 能在 10 分钟内完成一个小需求的编码时，生产力的瓶颈已从「产能不足」变成了「流程摩擦」。

在公司内部，这些问题同样突出。本文分享我们构建 DF（Delivery Flow，云端一站式智能交付系统）的实践——一个 AI Native 的研发工作流平台，旨在从流程层面释放 AI 的生产力。

#### 一、背景

2026 年 Agentic Coding 趋势报告[^2026 年 Agentic Coding 趋势报告：https://resources.anthropic.com/hubfs/2026%20Agentic%20Coding%20Trends%20Report.pdf?hsLang=en]中指出了几个关键数据：在基准测试（SWE-bench）上，顶级 AI Agent 的自主编程成功率已达到 72%，相比一年前的 13.8% 有了跨越式提升。与此同时，开发者的角色正在向「**系统架构师 + AI 指挥官**」的方向演进——你不需要亲自谱写每个音符，但你需要定义旋律、指挥乐团。

但是，这个阶段对研发流程的冲击是根本性的。如果一个 AI Agent 能够在 10 分钟内完成一个小需求的编码，那么传统的「提需求 → 排期 → 开发 → 联调 → 测试 → 部署」流程中，每个环节之间的等待和交接就成了最大的浪费。**生产力的瓶颈，从「产能不足」变成了「流程摩擦」。**因此，仅仅给工程师配上好的 AI 工具是不够的。驱动我们启动 DF 项目的，是三个核心洞察：

![图片](_attachments/2748610032_224105077867.png)

图 1：传统研发流程 vs AI Native 研发流程

**1. 工具链碎片化制约端到端效率。** 一个典型的小需求要经过 Ones（需求管理）→ 本地 IDE（编码）→ Code（代码评审）→ 流水线（CI/CD）→ Plus（发布）。每次切换系统，都意味着一次上下文丢失。AI 再快，也无法解决在多个系统间来回复制粘贴信息的低效。工具升级了，但流程没有跟上。

**2. 个体提效不等于团队提效。** 个人使用 AI 的能力方差极大——有人通过精准的 Prompt 一次性生成高质量代码，有人还在以手工写代码为主。团队的整体效率取决于最弱的一环而非最强的。真正的提效不是让每个人都会用 AI，而是把最佳实践固化成流程和平台，让所有人站在同一个起跑线上。

**3. AI 需要组织级的上下文。** 通用 AI Coding 工具只理解「代码」，不理解「业务」——它不知道 PRD 的意图、仓库的架构惯例、内部中间件的用法、服务间的依赖关系、公司的发布审批流程。要让 AI 从「能写代码」到「能交付需求」，必须在 AI 和企业知识体系之间建立一座桥梁，这需要平台级的解决方案。

这三个洞察，构成了我们启动 DF 项目的核心动机：**给 AI 铺路（流程重构），也给 AI 造车（Coding 标准化）。**

#### 二、DF：AI Native 研发工作流平台

##### 2.1 定位：不是又一个 AI Coding 工具

DF 的全称是 Delivery Flow——交付流。从名字就能看出，我们关注的不只是「Coding」这一个环节，而是从需求到上线的完整交付流。

与通用 AI Coding 工具相比，DF 解决的是两个层面的问题：

**第一，「路」的问题——流程重构。** AI 带来了生产力的质变，但过去分散的工具链已经成了制约效率的瓶颈。我们需要构建一站式、端到端、智能化的工作流，为高效率研发提供配套的「高速公路」。

**第二，「车」的问题——AI Coding 标准化。** 消除个人使用 AI Coding 工具的能力方差，将最佳实践固化为统一的 Code Agent。预先完成 80%-90% 的工作量，让全团队开箱即用，从「单兵作战」的提效转变为「集团军」整体水位的跃升。

具体来说，DF 支撑两类不同的开发模式：

- **AI 协同开发（Human-in-the-loop）**：面向较复杂的业务需求，比如涉及多模块联动的新功能开发。AI 负责生成技术方案草稿和代码初版，工程师在关键节点进行方案确认、逻辑校验和最终决策。这是一种「AI 并行生成 + 人工关键校验」的深度协作模式。

- **AI 自主开发（Human-on-the-loop）**：面向标准化的小型需求，比如 API 迁移、安全漏洞修复、依赖版本更新等。AI 根据预定义的 Skill（自定义技能[^自定义技能：https://support.claude.com/en/articles/12512176-what-are-skills]）自主完成从编码到验证的全流程，工程师仅在最后进行代码 Review。这是一种「AI 按技能执行 + 人工兜底审查」的轻量监督模式。

两类模式面向不同类型的需求，共享同一套 Pipeline 基础设施，由研发人员根据需求特征选择对应的流程。

##### 2.2 核心理念：双模驱动

DF 的核心理念围绕两类开发模式展开，分别对应不同的方法论：

**1）AI 协同开发：SDD + Vibe Coding 的融合**

**SDD（Spec-Driven Development，规格驱动开发）** 解决的是 AI 生成代码的「准确性」问题。在 AI 开始写代码之前，系统先自动解析需求，生成结构化的技术规格文档（Design Document），包括详细的接口定义、数据模型变更、业务逻辑描述。AI 根据这份规格来生成代码，而不是直接从模糊的自然语言需求出发。这就像建房子先画图纸，而不是凭感觉砌砖。

**Vibe Coding（交互式编程）** 则解决「最后一公里」的问题。AI 生成了 90% 的代码后，工程师通过对话、微调来完成最后的 10%。这种方式比从零开始高效得多，因为你是在一个已有的、大方向正确的基础上做精修，而不是面对空白文件。

两者结合，形成了一个高效的人机协同模式：

图 2：AI 协同开发工作流

每个环节都有明确的 AI 和人类职责分工，人的注意力聚焦在高价值的判断和决策上。

**2）AI 自主开发：Skill 驱动的自主交付**

对于标准化程度高的需求，DF 采用了一种不同的理念——Skill 驱动的自主交付。核心思路是：把人类专家处理此类任务的经验，封装为可复用的 Skill，Agent 通过调用对应的 Skill 自主完成全流程。

> Skill 是 Claude Code 推出的自定义技能机制。团队可以将标准化的工作流程、领域知识和最佳实践，以 Markdown 文件的形式定义在 `.claude/skills/` 目录下，注册为 Agent 可调用的 Slash Command。与传统的 SOP 文档不同，Skill 不仅仅是「给 AI 阅读的文档」，而是「AI 可以直接执行的技能」——它可以包含参数化的输入、结构化的执行步骤、工具调用指令和验证逻辑，让 Agent 像调用函数一样调用一整套标准化流程。

> *[流程图](https://km.sankuai.com/api/file/cdn/2748610032/224105398486?contentType=0&isNewContent=false)*

图 3：AI 自主开发工作流

这里的关键区别在于「人机分工」的位置不同。在人机协同模式中，人参与了过程中的多个环节；而在 AI 自主模式中，人的工作前置到了 Skill 定义阶段——一次定义、反复调用。AI 执行过程中无需人工介入，工程师只在最终产物（代码 PR）层面进行 Review。

这两种模式不是非此即彼的关系，而是互补的。复杂需求走人机协同，确保质量和正确性；标准化需求走 AI 自主，追求效率和规模化。随着 Skill 覆盖的场景不断增多，越来越多的需求将从前者转移到后者。

##### 2.3 端到端闭环：从 Requirement 到 Code 到 Deploy

DF 实现了完整的研发闭环，并根据需求类型提供两条差异化的流程路径。

**1）路径一：AI 协同开发**

面向复杂业务需求，DF 提供了覆盖六个核心阶段的完整流程：

**阶段一：需求接入。** 工程师从 Ones 平台选择需求（或通过大象机器人直接创建新需求），关联代码仓库和 AppKey，一键启动 AI 研发流程。系统自动拉取需求描述、PRD 文档、关联附件等上下文信息。

**阶段五：测试验证。** 系统自动生成单元测试和接口测试用例，支持快速的 API 验证和移动端扫码验证。

**2）路径二：AI 自主开发**

面向标准化需求，DF 提供了一条更加精简的自主执行路径：

![图片](_attachments/2748610032_224116280727.png)

图 9：AI 自主开发流程界面

步骤一：Skill 加载与任务初始化。 系统根据需求类型自动匹配对应的 Skill，加载预定义的执行步骤、验证标准和异常处理策略。Agent 通过调用 Skill 开始工作，不需要工程师参与需求澄清和方案设计。

步骤二：自主执行。 Agent 按照 Skill 定义的流程自主完成代码修改。例如，一个 SDK 升级的 Skill 会定义：依赖版本更新 → API 兼容性检查 → 调用方式适配 → 编译验证，Agent 逐步执行每个环节。

步骤三：自动验证。 Agent 自动运行 Skill 中定义的验证项——编译检查、单元测试、回归测试等。只有全部通过后，才会进入下一步。如果验证失败，Agent 会根据 Skill 中的异常处理策略自动重试或调整方案。

步骤四：提交 Review。 验证通过后，Agent 自动创建 PR 并生成详细的变更说明。工程师只需对最终产物进行代码 Review，确认无误后合入。

#### 三、DF 的核心技术架构

DF 在技术设计上采用了Multi-Agent协作架构，以实现关注点分离、灵活编排和可控性。平台通过Pipeline引擎编排Agent协作，并采用双引擎Agent架构，支持自研ReAct引擎与外部Claude Code引擎的灵活切换与降级。为让AI理解特定代码库，平台实施了包括背景注入、规则集成和领域知识在内的上下文工程。此外，基于MCP标准构建了可扩展的工具生态，并设计了分层的会话管理与记忆系统以维持任务连贯性。最后，平台强调实时可观测性，允许工程师全程监控、追踪AI行为，并在必要时进行中断与回滚。

图 10：DF 技术架构

##### 3.1 Multi-Agent 协作架构

DF 的核心引擎不是一个单一的大 Agent，而是由多个专职 Agent 协作组成的系统。主要包括：

- **需求分析 Agent**：负责解析 PRD，理解需求意图，生成澄清问题和结构化的技术方案
- **代码生成 Agent**：负责阅读代码仓库、理解项目结构、生成符合规范的代码
- **测试 Agent**：负责根据代码变更自动生成单元测试和接口测试
- **Review Agent**：负责对生成的代码进行 AI 代码审查

![图片](_attachments/2748610032_224104439694.png)

图 11：Multi-Agent协作架构

为什么选择 Multi-Agent 而不是 Single Agent？核心原因有三：

**第一，关注点分离。** 需求理解和代码生成是两种本质不同的能力。需求分析需要理解业务语义和产品意图，代码生成需要理解技术栈和编码规范。将它们拆分到不同的 Agent，每个 Agent 可以有独立优化的提示词、工具集和评估标准。

**第二，灵活编排。** 不同的场景需要不同的 Agent 组合。一个简单的 Bug 修复可能只需要代码生成 Agent，一个新功能开发需要全链路 Agent 协作。Multi-Agent 架构允许我们像搭积木一样灵活组合。

**第三，可观测性和可控性。** 每个 Agent 的输入输出都是可追溯的，出了问题容易定位到具体环节。相比一个大黑盒 Agent，这种架构更适合企业级场景对可控性的要求。

##### 3.2 Pipeline 编排引擎

在 Agent 编排领域，业界存在两种经典模式：一种是**工作流（Workflow）**——预定义的固定步骤图，每一步都是确定的；另一种是**纯 Agent 循环**——给 LLM 一组工具，让它自行决定下一步做什么。前者可控但僵化，后者灵活但不可预测。

DF 的 Pipeline 引擎选择了一条中间路径——「混合状态机」模式，核心思想是：**将确定性代码节点与自由度较高的 Agent 节点混合编排在同一条流水线中**。核心理念是“把 LLM 放进有约束的盒子里”，通过确定性地处理可预测的子任务（如编译构建、git 操作），节省 token 开销的同时减少 Agent 出错的机会。

具体来说，DF 的 Pipeline 中存在两类节点：

- **确定性节点**：不涉及 LLM 调用，执行固定逻辑——创建特性分支、触发编译构建、执行测试用例、提交 PR。这些操作的行为完全可预测，用确定性代码执行既快又可靠。

- **Agent 节点**：调用 LLM 进行自主决策——需求理解与澄清、技术方案生成、代码编写、CI 失败修复。这些任务本质上需要 AI 的创造力，给予 Agent 较大的自由度。

两类节点在同一条 Pipeline 中交替出现，一个典型的执行序列是：需求解析（Agent）→ 创建分支（确定性）→ 代码生成（Agent）→ Lint 检查（确定性）→ 编译验证（确定性）→ 测试执行（确定性）→ 失败修复（Agent）→ 提交 PR（确定性）。确定性节点保证关键步骤永远不会被跳过，Agent 节点聚焦于真正需要智能的环节。

另外，Pipeline 的设计有几个关键点：

- **可组合性**：系统定义了多种 Pipeline 模板，适配不同场景。比如完整的需求交付 Pipeline、纯编码的交互式 Pipeline、前端（Picasso）专属的 Pipeline 等。新增场景时，只需要用已有的 Stage 重新组合即可。

- **左移反馈**：一个在业界被反复验证的理念——如果一个错误会在 CI 阶段被发现，那就应该更早地在本地捕获它。DF 在代码生成完成后、推送到 CI 之前，会先执行一轮本地快速检查（ 单元测试、基础编译），确保代码“有希望首次通过 CI”。这种前置验证显著降低了无效的 CI 运行次数。

- **有限重试**：这里有一个重要的工程洞察：**LLM 的迭代修复存在边际收益递减**。业界的最佳实践通常将 CI 修复限制为最多两轮——第一次 CI 失败后自动修复并重试一次，之后无论结果如何都交回人类处理。我们在实践中观察到相同规律：Agent 对同一个 CI 失败反复重试，从第三次开始成功率骤降。因此 DF 同样对 Agent 的自动修复次数设置了上限，在成本、时间和完成度之间取得平衡。

##### 3.3 双引擎 Agent 架构

系统同时支持两种截然不同的 Agent 执行引擎，可以通过配置灵活切换。

**引擎一：自研 ReAct Agent。** 基于经典的 ReAct（Reasoning + Acting）模式自研的 Agent 循环。系统维护对话历史，每轮调用 LLM 获取下一步行动，解析并执行工具调用，将结果反馈给 LLM，如此循环直到任务完成。这个引擎与内部 LLM 平台深度集成，可以利用内部的模型服务和降级策略。

**引擎二：Claude Code Agent**[^Claude Code Agent：https://platform.claude.com/docs/en/agent-sdk/overview]**。** 直接通过 CLI 子进程的方式调用 Claude Code，利用其原生的文件编辑、终端命令等能力。通过流式协议与 Claude Code 进程通信，实时获取 Agent 的思考过程和操作动作。

两个引擎共享相同的 Pipeline 和上下文管理层，上层业务逻辑不感知底层用的是哪个引擎。这种设计的好处是：

- **渐进式演进**：可以逐步从自研引擎迁移到更强大的外部 Agent
- **灵活降级**：当外部 Agent 服务不可用时，可以自动降级到自研引擎
- **对比实验**：可以在同一个场景下对比两个引擎的效果，用数据驱动决策

##### 3.4 上下文工程：让 AI 理解「我们的代码」

通用 AI 模型不了解美团的技术栈和编码规范。DF 通过一整套「上下文工程」来解决这个问题：

**代码仓库深度索引。** Agent 启动时，系统自动对目标仓库进行结构化分析：目录结构、关键模块、技术栈识别、依赖分析。这些信息被注入到 Agent 的系统提示词中，让 AI 从一开始就对项目有全局理解。

**项目规则注入。** 适配仓库内的.cursor/rules、CLAUDE.md等规则文件，对齐项目特定的编码规范、技术约束、命名惯例等。Agent 在执行任务时会严格遵守这些规则。这意味着团队可以把自己积累的最佳实践，变成 AI 可以理解和执行的标准。

**Skill 驱动的任务执行。 **对于标准化的任务类型（比如 SDK 升级、安全漏洞修复等），我们将标准操作流程封装为自定义 Skill。Agent 在执行这类任务时，会自动调用对应的 Skill，按照技能中定义的步骤、工具链和验证逻辑完成任务。

**组织知识图谱。** 通过与 Ones、Avatar 等内部系统的集成，AI 能够获取需求的业务上下文、服务的架构拓扑、团队的协作关系等组织级信息。同时，结合公司内部通用中间件知识库（美团Context7[^美团Context7：https://km.sankuai.com/collabpage/2724449628]）及点评私有开发知识库（如海马、halo、mapi、picasso等），AI不再仅仅是「写代码」，而是在一个真实的业务和组织上下文中「交付需求」。

##### 3.5 会话管理与记忆系统

AI Agent 的一个核心挑战是「记忆」——既要记住当前对话的上下文（短期记忆），也要在多次交互间保持连续性（长期记忆）。DF 设计了一套分层的记忆管理机制：

**对话级记忆**：每次 Agent 执行的完整对话历史，包括思考过程、工具调用、中间结果。这些信息保证了 Agent 在单次任务执行中的连贯性。

**会话持久化**：对话结束后，会话内容被持久化存储。当工程师需要继续之前的工作时（比如第二天回来继续调试），系统可以恢复完整的会话上下文，Agent 无需重新理解项目。

**智能记忆压缩**：当对话历史过长超出模型的上下文窗口时，系统会自动触发记忆压缩——保留关键的系统提示和最新的交互，将中间过程浓缩为摘要。这个机制让 Agent 能够处理复杂的长时间任务，而不会因为上下文溢出而「失忆」。

**异常恢复**：如果会话因为各种原因中断（网络问题、超时等），系统能够从持久化存储中恢复，并智能地裁剪过长的历史，确保 Agent 可以平稳地续接工作。

##### 3.6 实时可观测性

对于一个依赖 AI 的系统，可观测性尤为重要——你需要知道 AI 在做什么、为什么这样做、做得对不对。DF 构建了一套实时的可观测体系：

**行为流实时推送**：Agent 的每一步操作（思考、工具调用、文件编辑、代码生成）都被解析成结构化的事件，实时推送到前端。工程师可以像看直播一样，实时观察 AI Agent 的工作过程。

**全链路追踪**：每个 Stage、每次 LLM 调用、每次工具执行都被纳入 APM 监控，可以追溯完整的执行路径和耗时分布。

**中断和回滚**：如果发现 Agent 走偏了方向，工程师可以随时中断执行、回滚代码变更、调整策略后重新开始。

#### 四、实践复盘：我们学到了什么

DF 自2025年12月正式上线以来，已经<u>在点评技术部累计助力超过100个需求完成交付，端到端交付时长平均 0.5 天，上线代码量突破20万行</u>。在这一过程中，我们积累了丰富的经验与深刻教训，现将其中最为关键的几点分享如下：

##### 4.1 两种模式，各得其所

在项目初期，我们也曾有过「全自动化」的愿景——从需求到上线完全不需要人工介入。但很快我们意识到，问题不是「要不要人参与」，而是「在什么场景下、以什么方式参与」。

**概率连乘问题：人机协同模式的核心依据。** 对于复杂业务需求，完整的研发流程由多个连续环节组成（需求理解、方案生成、编码实现、测试验证、部署上线……）。即便每个环节的 AI 成功率做到 95%（这已经很理想了），五个环节串接后的整体成功率也只有 0.95^5 ≈ 77%。而研发是对可靠性要求极高的活动，接近四分之一的失败率是不可接受的。所以对于这类需求，我们采用「AI 并行生成 + 人工关键校验」的协同模式：AI 负责大量的草稿生成和重复劳动，人类聚焦在方案确认、逻辑校验、最终决策这些高价值环节。这种分工恰好匹配了当前 AI 的能力曲线——在 80-90% 完成度的区间极其高效，但最后 10% 的精确度需要人类的判断力。

**AI 自主模式的突破口：标准化需求。** 但我们也发现，并非所有需求都需要人深度介入。对于标准化程度高的任务——比如 API 迁移、安全漏洞修复、依赖版本更新——任务的边界清晰、步骤确定、验证标准明确。这类需求的「概率连乘」问题天然较轻，因为每个环节的不确定性本身就很低。通过高质量的 Skill 进一步约束 Agent 的行为，AI 自主完成全流程的成功率可以达到相当高的水平。实践中，我们在 SDK 升级等场景的自主交付成功率超过 85%，工程师只需在最终 Review 环节把关。

**两种模式的选择标准：** 我们总结出一条简单的判断准则——如果一个需求能够被 Skill 完整封装（包括执行步骤、验证标准、异常处理），那它就适合走 AI 自主模式；如果需求涉及业务判断、方案权衡、跨团队协调等不易标准化的环节，就走人机协同模式。两种模式不是对立的，而是互补的——它们共同构成了 DF 应对不同需求类型的完整能力图景。

##### 4.2 好的上下文胜过好的模型

一个反直觉的发现是：在企业场景中，影响 AI 编码效果的第一要素不是模型有多强大，而是上下文有多完整。

同一个模型，当你只给它一句「实现一个用户查询接口」时，生成的代码可能质量很一般。但当你提供了完整的技术方案、数据模型定义、项目编码规范、已有的同类代码示例后，生成质量会有质的飞跃。**你喂给 AI 的上下文，决定了 AI 产出的上限。**

这也是我们走过的一段弯路。一开始把精力放在调优 Prompt、选模型、调参数上，效果总是不稳定。后来把重心转到「怎么给 AI 提供更完整的上下文」上——把仓库结构分析做好、把项目规则文件写好、把 Skill 定义好——效果立竿见影。

这个经验可能对每个在用 AI Coding 工具的同学都适用：与其纠结用 Copilot 还是 Cursor 还是 Claude Code，不如先把 rule.md 或者 skill.md 写好。这些文件花不了多少时间，但对 AI 的执行效果是决定性的。

##### 4.3 从小需求开始，渐进式扩展

DF 当前专注于 3PD 以内的小型需求场景。这不是因为我们没有野心，而是经过审慎评估后的策略选择：

- 小需求的边界清晰，AI 容易理解和执行
- 小需求的验证周期短，可以快速迭代和改进
- 小需求在日常研发中占比大（很多团队 30% 以上的需求是小需求），优化它们的性价比最高
- 小需求的成功经验可以逐步推广到更复杂的场景

试点结果显示，在小需求场景下，端到端交付效率提升超 50%，代码一次性采纳率超过 90%。这证明了路径的可行性。

##### 4.4 标准化是规模化的前提

很多人觉得 AI 时代一切都在快速变化，搞标准化是不是太重了？恰恰相反。

个人使用 AI Coding 工具，最大的问题在于不可复制。A 同学摸索出了一套高效的 Prompt 技巧，B 同学完全不知道。这种知识是「隐性」的，锁在个人的脑子里。DF 的一个核心价值就是把这些隐性知识「显性化」和「标准化」：

- 最佳实践变成了项目规则文件，所有 Agent 执行时自动加载
- 高效的 Prompt 策略固化在了系统提示词模板中
- 标准化的任务流程封装为 Skill，Agent 按技能定义执行

**标准化不是固化某种技术方案，而是固化「如何高效做事的方法」。** 当模型在换、工具在换的时候，你的项目规则文件、你的 Skill、你的质量标准，这些东西是可以跨模型、跨工具复用的。

换个角度看：标准化是把个人能力变成组织能力的唯一途径。你自己 Prompt 写得再好，那也只是你自己的能力。只有把它变成大家都能用的规则和流程，AI 的效果才能从「取决于使用者的水平」变成「取决于平台的能力」——后者才是可持续改进和规模化推广的。

#### 五、展望：未来会怎样演进

##### 5.1 短期：深化人机协同

近期最可能看到的变化是人机协同模式的深化和成熟：

**AI 的参与度逐步提高**：从当前的代码生成为主，逐步扩展到需求拆解、架构设计、测试策略制定等更高层次的活动。

**人的角色逐步升级**：从「写代码的人」变成「定义意图和验证结果的人」。正如趋势报告所预言的，开发者正在演变为高级协调者，兼具技术、战略和验收责任。

**更智能的多 Agent 协作**：Agent 之间的协作将从简单的串行 Pipeline 演变为更复杂的并行和动态编排。比如，前端 Agent 和后端 Agent 同时工作，由一个协调 Agent 确保接口对齐。

##### 5.2 中期：流程级的范式变革

更激动人心的变化可能发生在流程层面：

**需求即代码**：随着 AI 理解能力的增强，从自然语言需求到可部署代码之间的「翻译层」将越来越薄。产品经理写的 PRD 可能直接被解析为可执行的技术方案。

**持续验证取代阶段验收**：传统的「开发完了交给测试」将被「边生成边验证」取代。AI 在生成代码的同时就生成测试、运行测试、修复问题，交付给人的是已经通过验证的成品。

**研发效能的度量方式变革**：当 AI 承担了大量编码工作后，传统的代码行数、PR 数量等度量指标将失去意义。新的度量会聚焦于交付的业务价值、需求的端到端时长、AI 辅助的采纳率等。

##### 5.3 长期：组织形态的重塑

如果把视线拉得更远，AI Native 研发流程可能会重塑软件工程团队的组织形态：

**小团队大产出**：一个配备 AI Agent 的小型精英团队，可能达到过去大型团队的产出水平。这将推动组织向更扁平、更敏捷的方向演进。

**T 型人才的崛起**：AI 让工程师更容易跨越技术栈的边界——后端工程师借助 AI 可以写出合理的前端代码，前端工程师可以生成服务端逻辑。但这不意味着专业深度不重要，恰恰相反，**当每个人都有 AI 时，区分优秀工程师的关键在于深度的系统思维和架构判断力**。正如一位资深工程师所说：「最好的软件工程师不是最快的编码者，而是那些知道何时不信任 AI 的人。」

**人才培养管道的变化**：今天的初级开发者是明天的技术领导者。如果 AI 取代了大量入门级任务，我们需要重新思考如何培养工程师——可能需要设计全新的「AI 时代的工程师成长路径」，让初级工程师在 AI 的辅助下接触更有挑战性的工作，而不是被简单地替代。

#### 结语

软件工程的本质是控制复杂度。在 AI 时代，这一本质并未改变，只是我们的「武器」变了。如何用好 AI Coding 这把武器是一个系统工程问题，它涉及工具链的整合、研发流程的重构、组织能力的升级、人才培养模式的变革。这些事情不会因为某个 AI 模型足够强大就自动解决。需要有人去思考、去设计、去构建——去铺那条高速公路，去造那辆好车。

DF 是我们在这个方向上的一次实践尝试。它还远不完美，但它验证了一个重要的命题：在企业内部构建 AI Native 的研发流程是可行的，也是必要的。软件行业正处在一个历史性的转折点。预测未来的最好方式，就是积极地去塑造它。与各位共勉。

#### 附录 A：AI Coding 的三次浪潮

AI Coding 经历了三个阶段的演进，每个阶段对研发流程的影响程度截然不同。

![图片](_attachments/2748610032_224119004629.png)

**A.1 第一波：智能补全（2021-2023）**

以 GitHub Copilot 为代表的代码补全工具，本质上是一个更聪明的「Tab 键」。它在你编码时预测接下来的内容，帮你减少打字量。这个阶段的特征是：

- 触发方式：被动——你写代码，它补全
- 作用范围：行级到函数级
- 对流程的影响：几乎为零——只是加速了编码这一个环节

**A.2 第二波：对话式编程（2023-2025）**

ChatGPT、Claude 以及 Cursor 这类工具将 AI 编程推入了「对话」模式。你可以用自然语言描述需求，AI 生成整段代码甚至整个文件。这个阶段的变化更加深刻：

- 触发方式：主动——你提需求，它生成方案
- 作用范围：文件级到模块级
- 对流程的影响：开始显现——工程师的角色从「代码编写者」向「代码审查者」偏移

这一阶段带来了一个重要的新概念：Vibe Coding——你用自然语言描述意图，AI 生成初版代码，你在此基础上微调。编程从「逐行构建」变成了「意图驱动 + 精修」。

但同时，问题也开始暴露。当单纯地让 AI 帮你写代码时，它不知道你们公司用的是什么中间件，不知道你们的代码规范是什么，不知道这个需求的上下文和关联模块。个体效率提升了，但系统效率的瓶颈更加凸显。

**A.3 第三波：自主 Agent（2025-）**

这是当前正在发生的浪潮。以 Claude Code、Devin、OpenAI Codex 为代表的 Agentic Coding 工具，具备了自主规划、工具调用、多步骤执行的能力。它们不再只是「给你写代码」，而是能够：

- 自主阅读代码仓库，理解项目结构
- 制定修改计划，执行多文件编辑
- 运行测试，根据结果自动修复
- 生成 PR 描述，发起代码审查
