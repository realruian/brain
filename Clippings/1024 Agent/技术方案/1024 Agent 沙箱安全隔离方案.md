## 1. 背景

1024 Agent 面向外网C端用户提供服务，Agent 服务本身部署在公司内网。用户通过 Agent 可以触发两类高危操作：

- **Bash 命令执行**：在沙箱内执行任意 Shell 命令
    
- **浏览器操作**：通过 CDP 协议控制沙箱内的 Chrome 浏览器
    

外网恶意用户可能通过以下手段利用 Agent 访问内网系统：

|攻击方式|具体手段|危害|
|---|---|---|
|SSRF（服务端请求伪造）|指示 Agent 用 curl/wget/浏览器访问内网 IP|获取内网 API 数据、配置信息|
|内网探测|bash 执行 nmap、ping、telnet 扫描内网|绘制内网拓扑，为后续攻击做准备|
|DNS Rebinding|注册外网域名，TTL 极短，先解析外网通过检查，再切换解析到内网 IP|绕过域名白名单，访问内网服务|
|沙箱逃逸后横向移动|结合容器逃逸漏洞，直接访问宿主机内网|完全绕过沙箱隔离，用户数据泄露|

假设“恶意用户能够从沙箱逃逸”，要确保恶意用户不能攻入美团内网，并且确保恶意用户不能窃取其他用户隐私数据。本次邀请安全和设施团队评估推荐方案的安全性与落地可行性，评估问题详见第 4 章节。

## 2. 沙箱隔离方案对比

### 方案一：同容器 nsjail（内网）风险高（不考虑）

nsjail 在 Agent Loop 服务所在的 Hulk 容器内启动，无需跨网络访问，所有到内网 IP 段的流量在网络层直接 DROP

缺点：1、沙箱逃逸后横向移动，存在内网访问问题。2、没有MOA。

### 方案二：独立 nsjail 服务（公网）存在风险（不考虑）

nsjail 在独立服务中启动，部署在公网环境，Agent Loop 通过公网域名访问。

网络风险：沙箱逃逸后，对公司内网入侵风险低。但由于 nsjail 是容器技术，会导致逃逸后，存在用户信息泄漏等风险。

缺点：nsjail 沙箱维护成本（路由、管理），没有MOA（需支持）。

### 方案三：e2b Sandbox（多层网络软隔离）（推荐）

沙箱底层通过 Firecracker 实现 VM 级隔离，部署在内网环境 VPC 或者专区，Agent Loop 通过内网域名访问。

网络风险：两层网络软隔离，安全可控。第一层：沙箱自身通过VM/物理机层面的网络隔离能力，拒绝内网访问。第二层：沙箱数据面集群通过VPC或者专区在最外层做兜底网络卡控，确保不会溢出到内网。

落地成本：成本低，不涉及改造。但需要提供VPC或专区。

### 方案四：e2b Sandbox（公有云部署或直接使用公有云沙箱）成本高

沙箱通过 Firecracker 实现 VM 级隔离，部署在公公有云环境，Agent Loop 通过公网域名访问。

网络风险：网络彻底隔离，沙箱层面没有风险。

落地成本：改造成本高。若在公有云部署沙箱，需要去依赖改造，改造成本高。若直接使用公有云沙箱，Agent侧需要改造，来适配公有云沙箱，且技术可控性较差。

## 3. 推荐方案：e2b + 独立 VPC 纵深防御

### 3.1 整体架构

建议补充一个架构图，说明：外部用户 → WAF → API Gateway → AI Agent → 单向调用 → 沙箱 VPC（e2b Orchestrator + Firecracker VM）。 AI Agent 不持有任何内部服务凭证。

### 3.2 网络层管控

Agent 服务到沙箱 VPC 的通信方式（单向 API 调用，禁止反向连接）

|要求|具体措施|
|---|---|
|**独立 VPC**|C 端集群部署在独立 VPC，与美团办公内网、业务内网完全物理隔离，不打通网络。|
|**出口管控**|沙箱默认禁止所有出站流量；仅白名单外网域名可达（微信接口、公开 CDN、业务公网 API）。<br><br>通过沙箱出口白名单、流量审计与批量操作限速。<br><br>禁止暴露任何公司内网服务入口。|
|**内部服务访问**|一律通过公网 API Gateway 暴露服务，网关侧做鉴权+限流+审计，禁止沙箱直连内网服务 IP|
|**入口管控**|微信访问 agent 回调仅经过专用 Ingress 进入，禁止直接访问 Pod；Ingress 侧做请求合法性校验（签名验证）|
|**端口嗅探防护**|沙箱间网络隔离（Calico/Cilium NetworkPolicy），禁止 Pod 间直通，防止横向移动|
|**外网访问审计**|沙箱所有出站流量过透明代理，实时记录，高危 IP/域名自动告警|

### 3.3 命令执行管控

|类别|策略|示例|
|---|---|---|
|**默认禁止 Bash**|Bash 命令执行走受控 shell wrapper，不允许直接 exec|`bash -c`、`sh -c` 直接调用 → 拦截|
|**高危命令黑名单**|chmod/chown、iptables、nmap、curl -o（写文件）、dd 等 → 直接阻断；参考 Claude Code 默认 blocklist（curl、wget）|结合 seccomp profile 做 syscall 级拦截|
|**受控工具集**|将 bash 能力抽象为内置工具调用（read_file、write_file、web_search），由 Agent Runtime 托管执行|业务方按需申请开通，审批后白名单放行|

## 4. 评估的具体问题

推荐方案是 e2b（Firecracker microVM）部署在独立 VPC，Agent 单向调用沙箱 API，沙箱出口只允许白名单外网域名。

### 4.1 安全团队评估

1. 推荐方案（e2b + 独立 VPC）的网络隔离措施，是否满足 C 端 Agent 场景安全合规要求？有哪些必须补充的控制措施？
    
2. e2b（推荐方案）是开源项目，控制面部署在内网，数据面部署在独立VPC/专区，控制面和数据面之间的通信是否存在风险？
    
3. e2b Firecracker microVM 的沙箱逃逸风险评估，主要是逃逸后借助此通道横向移动的风险。
    
4. 出口白名单的审批与变更流程，白名单由谁审批。
    

### 4.2 设施团队评估

1. 独立 VPC或者网络专区方案落地可行性。
    
    - 能否单独划一个不与办公内网、业务内网互通的网络区域？
        
    - 出口流量是否可以统一经过受控出口（便于审计和防火墙规则管理）？
        
    - 大致改造周期？
        
2. e2b 自建部署的宿主机资源需求，以及现有环境是否满足。
    
3. 出口防火墙白名单管理、DNS 过滤服务的运维支持意愿和能力。
