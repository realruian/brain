## 一、概述

### 1.1 1024 是什么

1024 是美团内部的 Agent 平台，它不是问答机器人，而是**能执行任务的 AI Agent**：理解需求之后，会自主拆解步骤、调用内部系统工具、最后交付结果。

核心价值是让任何人都能低成本创建专属 Agent，无需深度技术背景，通过配置提示词、安装 Skill，即可快速搭建一个面向特定场景的 AI Agent。

### 1.2 与OpenCLaw的区别

1024 和龙虾解决的是不同层面的问题，核心差距在**企业级能力**上：

## 二、技术架构

### 2.1 集群化部署

1024 采用多节点集群部署，所有节点对等运行，支持按需水平扩容。

路由层通过**一致性哈希**将同一用户的请求始终分配到同一节点，保证对话上下文连续；某个节点故障时自动摘除，流量实时切换到其他节点，**服务不中断、用户无感知**。

这是龙虾单沙箱部署做不到的：一旦沙箱异常，当前会话直接中断。1024 的集群架构天然规避了这个问题，适合承载团队级、业务级的持续运行场景。

### 2.2 隔离机制

1024 在架构上做了三层隔离，从上到下依次收紧：

**第一层：集群隔离** 多台节点独立运行，单节点异常不影响整体。每个节点内部运行完整的 Agent 运行时，互不依赖。

**第二层：Agent 隔离** 同一台节点上可以同时运行多个 Agent，每个 Agent 有独立的：

- 提示词（Prompt）
    
- 工具权限（MCP / Skill）
    
- 模型配置
    
- 知识库
    

消息根据大象机器人的绑定关系路由到对应 Agent，互不干扰。一个 Agent 的异常不会影响同节点其他 Agent。

**第三层：用户隔离** 同一个 Agent 服务所有用户，但每个用户拥有完全独立的：

- 工作空间（独立目录，跨用户访问代码层强制拦截）
    
- 会话历史（独立存储，不会串话）
    
- 记忆文件（独立更新，A 的记忆不会出现在 B 的上下文）
    
- 沙箱实例（独立的浏览器实例和文件系统，20 分钟不活跃自动回收）
    

三层隔离共同保证：**多租户场景下数据安全、故障不扩散、资源不争抢。**

### 2.3 可观测性

每一次对话，1024 都会记录完整的执行链路（Trace），可以清楚看到：

- Agent 总共经历了几轮「思考 → 工具调用 → 再思考」循环
    
- 每一步调用了什么工具，入参是什么，返回了什么
    
- 哪一步耗时最长，哪一步出现了异常
    
- 本次对话合计消耗了多少 Token
    

对平台研发来说，这意味着 **Agent 的行为不再是黑盒**。线上问题可以直接通过 Trace 定位是哪个工具调用失败、是模型判断出了问题还是工具返回异常，排查效率和传统日志排查相比有质的提升。

## 三、接入方式

### 3.1 大象 IM 直接对话

最轻量的方式，无需任何开发。在大象 IM 搜索 1024 官方机器人，发起私聊或拉进群 @ 触发，直接用自然语言描述需求即可。

适合：个人临时使用、快速验证场景可行性、团队内部答疑。

### 3.2 Gateway API

1024 提供标准 HTTP 接口，支持将 Agent 能力集成到任意系统或自动化流程中。

**基本信息：**

|项目|说明|
|---|---|
|接口地址|`POST /openapi-v2/react/chat`|
|域名（线上）|`https://1024.sankuai.com`|
|鉴权方式|请求头 `X-API-Key`，在 Agent 详情页「渠道接入 / OpenApi」Tab 生成|
|多轮对话|服务端自动维护上下文，客户端只需携带上一轮返回的 `conversationId`，无需手动传历史消息|

## 四、业务场景实践

### 4.1 告警根因自动分析

**痛点：** 线上告警触发后，研发需要手动翻 Raptor 日志、查 COE 历史、对照监控指标，定位根因往往要花 10-30 分钟，深夜告警尤其痛苦。

**1024 解法：** 将告警信息直接丢给 1024，它自动关联 Raptor 日志、检索历史相似 COE 案例，输出根因分析和处置建议，整个过程 1-2 分钟内完成。

**示例 prompt：**

```
服务 ai-agent 在 01:32 出现告警，错误信息如下：
[upstream timed out, Nginx 502, 应用服务器 CPU 95%]
Raptor logviewID：xxxxxxxx
帮我分析根因，并查询历史是否有相似 COE 案例。
```

**效果：** 输出根因判断（如连接池耗尽、GC 停顿等）+ 历史相似案例 + 推荐处置步骤，研发拿到结果直接决策，不再从零开始排查。

### 4.2 故障工单自动流转

**痛点：** 故障发生时，需要人工判断影响的 PaaS 组件、找到对应值班人、手动创建并流转 TT 工单，流程繁琐且容易遗漏。

**1024 解法：** 结合告警信息自动识别受影响组件，查询当前值班人，一步完成工单创建和流转，故障响应链路全程自动化。

**示例 prompt：**

```
刚才的告警确认是 Rhino 限流配置问题，
帮我创建一个 TT 工单，标题「Rhino 限流异常-ai-agent服务」，
查询 Rhino 当前值班人，流转过去，优先级：高。
```

**效果：** 工单自动创建、自动路由到 Rhino 值班人，研发无需登录 TT 后台手动操作，故障响应时间显著压缩。

### 4.3 代码变更周报自动生成

**痛点：** 每周整理代码周报需要手动翻 Git 提交记录，筛选有效变更、归类、写摘要，重复劳动，容易遗漏。

**1024 解法：** 指定仓库、分支、作者和时间范围，1024 自动拉取提交历史，按功能模块归类，生成结构化周报。

**示例 prompt：**

```
生成 ai-agent 仓库最近一周的代码周报：
- 分支：origin/develop
- 作者：yeming
- 输出格式：按模块分类，每条变更一句话摘要，
  最后附本周提交总数和变更文件数统计。
```

**效果：** 1 分钟内输出结构化周报，可直接粘贴到周会或发到大象群，不再手动翻 commit log。

### 4.4 定时数据播报到大象群

**痛点：** 业务数据需要每天人工查询、整理、发群，长期执行容易遗漏，且数据格式不统一。

**1024 解法：** 配置一次定时任务，1024 按时自动查询数据库或魔数看板，生成图表，推送到指定大象群，完全无人值守。

**示例 prompt：**

```
创建定时任务，每天早上 9 点执行：
1. 查询 ai_message_log 表，统计昨日各 Agent 消息量 Top10
   （排除 Agent_Router、Question_Identify、Response_Checker）
2. 查询 friday_user_conversation 表，统计昨日新增用户数
3. 生成趋势图和汇总表，推送到大象群 69457453956
```

**效果：** 配置一次，永久自动运行。数据播报准时、格式统一，从此不用每天盯盘手动发群。
