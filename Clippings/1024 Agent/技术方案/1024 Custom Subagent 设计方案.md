
# 1024 Custom subagents 设计方案

仅开启了工作区模式的Agent可用。

## 一、背景与目标

1024 平台（ReactMind）的主 Agent 是一个通用型智能助手，面向用户的所有请求都由它统一处理。随着接入场景增多，一些问题逐渐暴露：

- **主 Agent 职责过重**：处理"SQL 慢查询分析"和"客服退款咨询"需要完全不同的 system prompt、工具集和推理风格，通用配置难以在所有场景都表现出色
    
- **无法精细化**：不同业务场景对工具权限、模型选择、最大轮次的需求各不相同，主 Agent 做不到按场景差异化配置
    
- **必须经过主 Agent 中转**：即使意图非常明确，也需要先启动主 Agent 推理一轮才能转发给专属逻辑，存在额外延迟
    

**Custom SubAgent** 的设计目标是：**让平台和用户能定义专门处理特定领域的子 Agent，并支持三种方式灵活触发**——主 Agent 委托、系统自动路由、用户主动指定——实现"对的任务交给对的 Agent 处理"。

---

## 二、核心能力

### 2.1 子 Agent 是什么

Custom SubAgent 是一个**具有独立身份的 Agent 实体**，拥有：

- **独立的角色定义**：专属 system prompt，可针对特定领域深度定制
    
- **独立的工具权限**：可以只开放必要工具，也可以限制危险操作
    
- **独立的模型配置**：可为不同场景选择最合适的模型
    
- **独立的执行上下文**（工具调用模式下）：不共享主 Agent 的对话历史，专注执行分配的任务
    
- **持久化记忆**：跨会话保留用户偏好和历史经验
    

### 2.2 三种触发方式

子 Agent 支持三种触发方式，适用不同场景：

#### 方式一：主 Agent 委托（`run_subagent` 工具调用）

主 Agent 在推理过程中，判断某个子任务适合交给专门的子 Agent 处理时，通过 `run_subagent` 工具委托执行。

- 子 Agent 在独立上下文中运行，不共享主 Agent 的对话历史
    
- 主 Agent 可以同时委托多个子 Agent **并发执行**，等全部完成后汇总结果回复用户
    
- 用户感知：正常的流式响应，主 Agent 收到子 Agent 结果后汇总输出
    

**适用场景**：复杂任务需要拆解，如"多个专家并行分析"、"主 Agent 编排多个子任务"。

#### 方式二：路由自动匹配

消息进入时，系统先调用一个**轻量路由 LLM**，基于各子 Agent 的描述判断意图，若命中则**跳过主 Agent 决策轮次**，直接以该子 Agent 的身份响应用户。

- 子 Agent 继承完整的对话历史，上下文连续
    
- 未命中任何子 Agent 时，自动降级走主 Agent 正常流程
    
- 用户感知：发消息 → 稍等片刻 → 直接收到完整结果（无流式打字过程）
    

**适用场景**：高频、意图明确的场景，希望减少主 Agent 中转延迟。

#### 方式三：用户指令直接指定（`/sa:xxx`）

大象会话支持 /sa-l 查看已有subagent。 /sa-st 查看各agent状态。

用户通过指令精确指定子 Agent，完全绕过路由 LLM：

```
/subagent:sql-analyzer 帮我分析这条慢查询：SELECT * FROM order WHERE uid=xxx
/subagent:code-reviewer 看下这段代码有没有问题
```

- 子 Agent 继承完整的对话历史，上下文连续
    
- 指定的子 Agent 不存在时，回复提示卡片列出当前可用的子 Agent
    

**适用场景**：用户明确知道要用哪个子 Agent，追求精确控制。

### 2.3 三种方式对比

|维度|主 Agent 委托|路由自动匹配|指令直接指定|
|---|---|---|---|
|触发方|主 Agent LLM 主动决策|系统路由 LLM 自动判断|用户主动输入|
|对话历史|任务描述自包含|完整继承|完整继承|
|响应方式|结果汇总后由主 Agent 回复|直接回复用户|直接回复用户|
|典型场景|复杂任务拆解与编排|意图明确的高频请求|用户精确控制|

## 三、子 Agent 定义

### 3.1 定义作用域

子 Agent 支持两种作用域，可按需灵活配置：

|作用域|配置者|生效范围|
|---|---|---|
|**PaaS 级**|平台管理员|该 PaaS 下所有用户|
|**用户级**|用户个人|仅对自己生效|

同名子 Agent 时，用户级定义优先于 PaaS 级定义。

### 3.2 子 Agent 类型

|类型|说明|适用场景|
|---|---|---|
|`custom`|使用本地自定义的 AGENT.md 定义角色|全新定制专属 Agent|
|`platform`|引用平台上另一个已有 PaaS Agent 的角色定义|复用已有 Agent 能力，无需重复维护 prompt|
|`rolehub`|从 rolehub 平台拉取 Agent 定义包|复用 rolehub 上已维护的 Agent，通过 `rolehub` 类型子 Agent，可以将 RoleHub 上已维护的 Agent 能力直接下发到 1024，实现**"角色在 RoleHub 开发、能力在 1024 执行"**的分工模式。|

### 3.3 核心配置说明

子 Agent 的核心元数据配置项：

|配置项|说明|
|---|---|
|`name`|子 Agent 名称，用户通过 `/subagent:xxx` 指定时使用|
|`description`|子 Agent 的能力描述，是路由 LLM 判断意图匹配的依据，应尽量准确|
|`type`|Agent 类型（`custom` / `platform` / `rolehub`）|
|`belongPaas`|归属的主 agent|
|`belongMis`|用户级子agent需要填写，归属用户|
|`model`|专属模型，不配置则继承 PaaS 全局配置|
|`maxTurns`|最大推理轮次|
|`allowedTools` / `disallowedTools`|工具白名单 / 黑名单，精细控制工具权限|
|`skills`|Skill 白名单，不配置则继承主 Agent 的全量 Skill|
|`roles`|预留字段，用于商家不同角色需求的扩展|

### 3.4 持久化记忆

子 Agent 支持跨会话的持久化记忆（MEMORY.md），每次启动时自动注入 system prompt。子 Agent 在任务结束后可将值得记住的信息（用户偏好、经验总结、环境约定等）主动写回，供下次使用。

## 四、执行机制

### 4.1 路由决策流程

消息到达时按以下优先级处理：

```
收到用户消息
  │
  ├─ 1. 检测 /subagent:xxx 指令前缀
  │       命中 → 直接按名称查找子 Agent 执行，跳过路由
  │
  ├─ 2. 调用路由 LLM（基于各子 Agent 的 description 判断意图）
  │       命中 → 以该子 Agent 身份直接执行
  │       未命中 → 走主 Agent 正常流程
  │
  └─ 3. 主 Agent（兜底）
```

路由 LLM 调用失败或解析异常时，自动降级走主 Agent，不影响正常使用。

### 4.2 主 Agent 委托执行流程

```
主 Agent 调用 run_subagent
  │
  ├─ 立即返回占位结果（主 Agent 推理不阻塞）
  │
  ├─ 子 Agent 在独立线程池中并发执行
  │     以子 Agent 专属的 prompt / 工具 / 模型运行 ReAct 循环
  │
  └─ 子 Agent 完成 → 结果注入主 Agent
        主 Agent 汇总所有子 Agent 结果 → 流式回复用户
```

多个子 Agent 可以并发执行，主 Agent 等待所有子 Agent 完成后再汇总。

### 4.3 执行约束

- **禁止递归**：子 Agent 不能再启动子 Agent，`run_subagent` 工具在子 Agent 中自动禁用
    
- **停止联动**：用户触发"停止思考"时，主 Agent 和所有关联的子 Agent 同步中断
    
- **超时保护**：可在配置中指定最大执行时长，超时自动退出并返回提示

## **五、上下文派生机制**

### **5.1 派生子agent的上下文**

当主 Agent 委托子 Agent 执行任务时，子 Agent 基于父 Agent 的上下文派生出独立的执行环境。哪些信息传递、哪些隔离，规则如下：

|**上下文类别**|**是否传递**|**说明**|
|---|---|---|
|用户身份（mis）|✅ 传递|子 Agent 需知道当前用户是谁，用于权限控制和个性化|
|PaaS 标识|✅ 传递|子 Agent 继承父 Agent 的 paas，使用相同的工具和 Skill 体系|
|大象机器人 ID / 卡片请求 ID|✅ 传递|用于停止信号联动，主 Agent 停止时同步中断子 Agent|
|主 Agent 对话历史|❌ 隔离|子 Agent 运行在独立上下文中|
|子 Agent 角色定义（AGENT.md / SOUL.md / IDENTITY.md）|✅ 注入|子 Agent 使用自己专属的系统提示词，替换主 Agent 的提示词|
|子 Agent 专属 Skill / 工具配置|✅ 注入|按 META.json 中的 skills / allowedTools / disallowedTools 配置过滤|
|工具调用进度回调|✅ 传递|子 Agent 执行工具时，进度实时推送到主 Agent 的流式卡片|

### **5.2 路由 / 指令模式下的上下文切换**

路由自动匹配和指令直接指定两种模式下，子 Agent 直接成为本轮的响应方：

- **会话 ID**：复用主 Agent 的当前会话 ID，对话历史天然连续
    
- **系统提示词**：切换为子 Agent 的专属 AGENT.md 内容
    
- **工具集**：切换为子 Agent 配置的工具集（allowedTools / skills）
    
- **用户工作区**：子 Agent 有自己专属的 AGENT.md/SOUL.md/IDENTITY.md，用户工作区中的同类文件自动跳过，避免冲突
    

## 六、会话与可观测性

### 6.1 会话 ID 策略

三种触发方式下，会话 ID 的处理策略不同：

|触发方式|会话 ID|原因|
|---|---|---|
|主 Agent 委托（`run_subagent`）|新建独立子会话|子 Agent 运行在隔离上下文中，与主会话逻辑独立|
|路由自动匹配|复用主 Agent 当前会话|子 Agent 是本轮的直接响应方，需延续对话历史|
|指令直接指定|复用主 Agent 当前会话|用户显式指定，历史上下文需延续|

### 6.2 会话日志与可观测性

三种触发方式下，消息日志均完整记录，管理员在**主 Agent 会话详情页**能看到完整的多 Agent 协作过程：

- **路由/指令模式**：子 Agent 消息归入主会话，日志中标注本条消息由哪个子 Agent 产生
    
- **委托模式**：子 Agent 有独立子会话日志，同时记录与主会话的归属关系，管理端按时间顺序合并展示
    

为支持按子 Agent 维度统计分析，消息日志中会记录：

- 本条消息由哪个子 Agent 产生（`subagent_name`）
    
- 工具调用模式下，子会话归属于哪个主会话（`parent_conversation_id`）
    

### 6.3 Trace 链路

- **路由/指令模式**：Trace 记录在主会话下，可直接查看，并标注本次以哪个子 Agent 身份执行
    
- **委托模式**：子 Agent 有独立的 Trace 链路，可通过主会话关联聚合，还原完整的多 Agent 执行过程
    

## **七、高可用设计**

子 Agent 执行的是长任务，面临机器重启、进程崩溃等场景时，需要有能力从中断处恢复继续执行，而不是让用户重新发起整个任务。

### **7.1 故障恢复：基于完整会话记录**

平台对每一轮 Agent 执行都做了完整的会话记录（消息日志 + 工具调用日志）。当进程异常中断时，恢复路径如下：

```
进程崩溃 / 机器重启
  │
  └─ 服务恢复后，用户重新发送消息或触发续跑
        │
        ├─ 从会话记录中还原已执行的消息历史（LLM 输出 + 工具调用结果）
        │
        └─ 将已有历史作为上下文继续推理
              子 Agent 不需要重新执行已完成的步骤
              从上次中断的位置继续 ReAct 循环
```

会话记录中包含：

- 每一轮 LLM 的输入和输出
    
- 每次工具调用的参数和结果
    
- 子 Agent 的 `subagent_name` 和 `parent_conversation_id`，可精确定位到哪个子 Agent 执行到哪一步
    

## **八、Webhook 扩展能力**

### **8.1 现有 Webhook 事件**

平台已支持以下 Webhook 事件，PaaS 开发者通过注册回调 URL 即可接入：

|**事件**|**类型**|**触发时机**|**可做什么**|
|---|---|---|---|
|`SESSION_START`|通知类|新会话创建后|记录会话、初始化业务状态|
|`USER_MESSAGE_RECEIVED`|**拦截类**|消息入队前|拦截非法消息、注入业务上下文（占位符参数或直接替换系统提示词）|
|`PRE_TOOL_USE`|**拦截类**|工具实际执行前|权限校验、修改工具入参|
|`PRE_SKILL_USE`|**拦截类**|Skill 调用前|权限校验、阻止危险 Skill|
|`POST_TOOL_USE`|通知类|工具执行完成后|工具结果脱敏、慢工具告警|
|`AGENT_COMPLETE`|通知类|推理完成后|费用统计、质量审计、外部通知|
|`AGENT_ERROR`|通知类|推理失败时|异常告警|

**规划中的子 Agent 专属事件**

|**事件**|**类型**|**触发时机**|
|---|---|---|
|`SUBAGENT_START`（规划）|**拦截类**|子 Agent 即将启动前|
|`SUBAGENT_COMPLETE`（规划）|通知类|子 Agent 执行完成后|
|`SUBAGENT_ROUTED`（规划）|通知类|路由命中子 Agent 时|

所有**拦截类**事件均实现了 fail-open 机制：Webhook 回调超时（默认 5 秒）或发生异常时，平台自动视为"放行"，业务方的服务故障不会导致用户消息处理阻塞。
