## 一、背景：为什么需要记忆？

大语言模型本身没有持久记忆——每次对话结束，它就"忘了"一切。

1024 通过三种机制弥补这个缺陷，让 Agent 能够：

- 记住你的偏好和工作习惯
    
- 回忆起几天前、几周前发生的事情
    
- 避免反复问你同样的问题
    

---

## 二、如何使用

进入到1024平台的 Agent详情页面，打开组件信息Tab页面，开启跨会话记忆，后续的对话记录会自动同步到Milvus记忆库，然后即可检索出记忆。

### 3.1. 静态注入（Static Injection）— 常驻记忆

**状态：✅ 已支持**

**一句话理解**：每次对话开始前，Agent 自动读取一份"小抄"，把重要信息带进这次对话。

这份"小抄"就是 `MEMORY.md` 文件，存放在每个用户的专属工作区中。它的内容会在**每次会话开始时自动注入**到 Agent 的上下文里，让 Agent 从一开始就了解你。

**Agent 主动维护，无需用户提醒**

Agent 会在对话过程中主动判断哪些信息值得长期记住，并自动写入 `MEMORY.md`——不需要你说"记住这个"。

典型的写入场景（按优先级排列）：

1. **你纠正了 Agent**，或说"以后不要用 X，改用 Y"——最优先，防止下次重犯
    
2. **你的偏好和习惯**：操作系统、编辑器、时区、编码风格、沟通方式
    
3. **项目约定和环境信息**：禁止在循环中查数据库、禁止写单元测试、Maven 项目结构
    
4. **稳定的工具技巧或 API 约定**：某个 SDK 的特定用法、已知的 Bug 规避方式
    

> 最有价值的记忆，是让你不必再重复说一遍的那条。

**什么不写进去？**

任务进度、本次会话的操作结果、临时调试内容——这些不适合占用有限的 `MEMORY.md` 空间，而是通过 `session_search` 按需检索。

**容量上限与精简策略**

`MEMORY.md` 上限约 **2200 字符**（大约 10–15 条）。当使用量超过 80% 时，Agent 会主动将多条相关条目合并为一条高密度条目来腾出空间，确保最有价值的信息始终留存。

**好的记忆条目示例（紧凑、信息密度高）：**

```
用户使用 macOS，Shell: zsh，Java 项目用 Maven，4 空格缩进，UTF-8 编码。
禁止在循环中访问 DB；禁止写单元测试；接口返回统一用 Result<T> 封装。
```

### 3.2. 自动预取（Automatic Prefetch）— 智能补充

**状态：❌ 暂不支持**

**一句话理解**：每次你发消息，系统在 Agent 回复之前，自动从历史记录里搜出相关内容，悄悄补充进上下文。

这相当于 Agent 在回答你之前，会先快速"翻一下以前的笔记"，把相关片段准备好。

这种能力需要接入外部记忆插件（如 Mem0、OpenViking 等）。1024 目前**尚未支持**，是后续的规划方向。

---

### 3.3. 主动检索（Active Retrieval）— 按需查找

**状态：✅ 已支持**

**一句话理解**：当 Agent 觉得"这件事我们可能聊过"，它会主动去历史会话里搜索，把相关对话总结后告诉你。

这是 1024 实现的 `session_search` 工具。Agent 用关键词在**所有历史会话**中进行全文检索（类似搜索引擎），找到相关对话后，再用 AI 生成一段摘要，告诉你"那次你们聊了什么"。

**举个例子**：

> 你问 Agent："上次我们讨论那个数据库慢查询是怎么解决的？"

> Agent 用 `session_search("数据库慢查询")` 检索历史 → 找到 3 周前的对话 → 生成摘要："3 周前你遇到了一个 SQL 缺少索引导致的慢查询，最终通过添加联合索引解决，相关表名为 ai_message_log。"

#### 数据是怎么存进去的？

每一轮对话结束后，系统会将本轮的用户消息和 Agent 回复**异步写入 Milvus 向量数据库**，整个过程在后台自动完成，不影响对话响应速度。

写入时，每条消息会附带以下信息，用于后续精准过滤：

- 所属用户（私聊隔离）或群组（群聊隔离）
    
- 所属 Agent
    
- 消息角色（用户消息 / Agent 回复）
    
- 消息时间
    

Milvus 在存储时会对消息内容进行 **BM25 中文分词索引**，这是一种在搜索引擎中广泛使用的关键词相关度算法，能够准确理解中文词语的切分和权重，比简单的字符串匹配效果好得多。

#### 检索过程是怎么工作的？

Agent 调用 `session_search` 后，后台依次执行以下步骤：

**第一步：全文检索**  
用关键词在 Milvus 中进行 BM25 检索，按相关度排序，召回最相关的若干条历史消息。同时根据用户身份自动缩小范围（只搜自己的或本群的历史）。

**第二步：按会话聚合**  
将命中的消息按所属会话分组，取每个会话中相关度最高的分数代表该会话，挑选出最相关的 Top N 个历史会话（默认 3 个）。

**第三步：加载完整对话**  
从数据库中取出这几个会话的完整对话记录，还原成"用户说了什么 → Agent 做了什么 → 结果如何"的可读格式。若会话过长（超过 10 万字符），以关键词命中位置为中心智能截取最相关的片段。

**第四步：AI 生成摘要**  
对每个会话并行调用轻量 AI 模型（LongCat-Flash），生成一段聚焦于检索主题的摘要，包含：用户当时想做什么、采取了哪些行动、最终结论、关键技术细节、遗留问题。

**第五步：返回结果**  
将所有会话的摘要汇总返回给主模型，主模型再结合这些历史信息回答用户的问题。

**检索范围**：

- 私聊场景：只检索你个人的历史对话
    
- 群聊场景：检索整个群组的共享对话历史
    

**历史不丢失**：即使某次对话时间很长，系统会自动将老消息归档（类似日志轮转），`session_search` 依然能检索到归档前的内容。

## 四、三种机制对比

||静态注入|自动预取|主动检索|
|---|---|---|---|
|**当前状态**|✅ 已支持|❌ 规划中|✅ 已支持|
|**触发方式**|每次会话自动注入，无感知|每次回复前自动触发，无感知|Agent 主动调用，可见工具调用过程|
|**适合的场景**|常驻偏好、约定、教训|每轮实时补充相关背景|"我们聊过 X 吗？" 类历史回溯|
|**容量**|约 2200 字符|取决于插件|无限制|
|**后续计划**|—|接入 Mem0 / OpenViking 等|—|
