作者是做什么的、为什么写这个

作者 codejunkie99 是社区里的 AI 内容创作者，不是某家公司的官方代言人。他写这份路线图的动机原话是：

"大多数工程师并不清楚自己该学什么。有人挑了 CrewAI 因为 Twitter 上的 demo 看着炫，有人追每一个新框架什么都没真正做完，有人一上来就扎进多 agent 系统对基础概念一概不懂——结果都是浮光掠影一堆框架，真正能上生产的技能没多少。"

**没有自动评测系统，"我感觉变好了" = 没改进**——这是大部分团队卡住的地方
**上线之后才真正开始**——成本、延迟、安全、监控、韧性，5 个事情永远要做

模块 2：6 大核心观点（人话版）

1. 同一个 Claude，装在不同"工装"里，分数差 36 分

Anthropic 自己的实测：Claude 4.5 这同一个模型，跑同一个测试：

装在 Claude Code 里：78 分
装在另一个开源框架 Smolagents 里：42 分

**模型完全一样**。差距全部由"工装"（也就是 harness）贡献。

类比：同一台保时捷的发动机，装在保时捷上是跑车，装在拖拉机上是拖拉机。

启示：**别迷信"哪家模型最厉害"**。在 2026 年，"工装能力"比"模型能力"更决定上限。

2. 写 prompt 已经不是核心技能了，叫 context engineering

老路：花一周琢磨怎么写 ChatGPT 提示词，让它输出更稳定。

新路：花一周琢磨**每一步该把哪些字塞进模型面前**——历史对话留多少、文档读多少、工具说明放在哪、记忆怎么压缩。

类比：以前比的是"会不会问问题"，现在比的是"信息整理能力"——同样的助理，给他递的资料整理好坏，决定他汇报的水平。

四个具体动作（路线图反复强调，记下这一组就够了）：

**Write（写下来）**：把易变的业务知识沉到文件里，别每次都塞进对话
**Select（按需检索）**：用到时再捞，不要预加载
**Compress（压缩）**：对话长到一定程度，把老的总结掉，留 token 给新的
**Isolate（隔离）**：复杂任务派子 agent 去做，主 agent 只看摘要

3. 2026 只有两条值得深耕的技术栈

栈
大白话
适用场景
**LangGraph + Deep Agents**
开源派的"agent 编排框架"。灵活强、可定制深、能上生产
想把 agent 接到自家服务、要做复杂多步流程时
**Claude Agent SDK + Claude Code**
Anthropic 自家出的"参考实现"，直接拿来用就是工业级
想最快搭出一个能用的 agent，或者想学"标准答案"长什么样时

**为什么是这两条而不是别的**：

它们都解决了 agent 真上线必须的几件事：多步控制、持久化、可观测、可调试
其他框架（CrewAI / AutoGen / Smolagents / OpenAI Swarm）要么停更，要么只能做 demo

**学这两条对一线工作有什么用**：

选型不用迷茫——内部要做新的 agent 项目，盯死这两个里挑
招聘有标准——简历上写"做过 LangGraph + Deep Agents 的项目"和"做过 CrewAI demo"，含金量差很多
评估外部供应商——他们用的栈是这两个之一吗？

4. AI agent 的"骨架"由 10 个零件组成

任何一个能用的 agent 系统，背后必须有这 10 件事在转。少一件就漏：

零件
大白话
1. Loop control（循环控制）
决定 agent 什么时候继续、什么时候停
2. Tool dispatch（工具调度）
解析 AI 想调什么工具，去执行，把结果送回
3. Context management（上下文管理）
历史对话、文档、工具说明的拼装和压缩
4. Persistence（持久化）
每一步存档，崩了能恢复
5. Sub-agent 编排
派子 agent、收摘要
6. Skills 加载
按需把"技能包"塞进上下文
7. Hooks（钩子）
工具调用前/后做拦截，比如禁敏感操作
8. Observability（可观测性）
每一步的耗时、token、错误都能看到
9. Sandbox（沙箱）
模型生成的代码必须隔离执行
10. Auth & Secrets brokering
凭据托管在模型外，模型永远看不到密码

类比：一辆汽车不是只有发动机，还有变速箱、刹车、车灯、仪表盘、安全气囊。

启示：当你评估一个 agent 系统/工具/平台好不好，**对照这 10 件清单走一遍**。

5. 没有 eval（自动评测），改进就是自欺欺人

大部分团队的真实情况：

上周改了一版 prompt，"感觉好像更好了"
这周加了个新的 skill，"感觉好像没影响"
老板问"到底好了多少"，回答不出来

**Eval 就是给 agent 写的自动测试**：建一个 30-50 道题的金标题库，每次改完跑一遍，分数变没变一目了然。

四种评测：

**单步对不对**（输入 X 输出该是 Y 吗）
**过程对不对**（agent 是否按预期调了工具）
**成品好不好**（开放式输出，用另一个 LLM 当评委打分）
**真实操作对不对**（agent 改完文件、写完 DB 后，结果状态对不对）

启示：**没接 eval，就别说"我们在改进 agent"**。

6. 上线之后才真正开始

agent 跑通 demo 只是 0 到 1。1 到 N 是 5 件事，永远要做：

**成本**：每跑一次多少钱？多 agent 烧 token 是单 agent 的 ~15 倍，值不值
**延迟**：用户等多久？能并行的别串行
**安全**：代码进沙箱跑、密钥别让模型看到、不可逆操作必须人审
**监控**：今天分数比昨天低了告警；模型一升级先重做评测
**韧性**：进程崩了能从断点恢复

PDE 视角：**"上线" ≠ "完成"**。运维要写到设计阶段，不是上线后再想。


模块 3：6 阶段路线图（这是要把谁练成什么样？）

**整体目标**：把一个普通工程师，练成 **"能独立把一个生产级 AI 功能从 0 上线"** 的人。注意：

不是 AI 研究员（造发动机的）
是 AI agent 工程师（用发动机造车并交付的）

全职 17 周，兼职 10-15 小时/周大约 40 周。

Phase 0：建立心智模型（1-2 周）

**目标**：先理解概念，**这一阶段不要写代码**
**关键产出**：用大白话讲清楚一组词（agent vs workflow / 上下文 4 原语 / harness 是什么）
**非技术同学怎么参与**：这阶段全是看资料、写文档，**没门槛**。做完之后你能在产品讨论会上有话语权
**里程碑**：在饭桌上 5 分钟讲清"AI agent 是什么"

Phase 1：搭出第一个会用工具的 agent（2-3 周）

**目标**：写两遍同一个 agent。一遍用最底层 SDK"100 行手撸"，一遍用现成的 Claude Agent SDK
**关于"100 行手撸"——非技术同学的疑惑**：

不是从零造发动机。**是用 Anthropic 提供的最底层 API 拼一个最小可用的 agent**
类比：自己装一台台式机 vs 买戴尔。"手撸"是装机——CPU、内存、主板都是现成的，你只把它们插对位置
100 行 ≈ 50 行 Python + 50 行配置，跟着教程一天就能跑通
所以不是吹牛，**是有准备的拼装练习**——目的是让你看见 agent 内部怎么转
**非技术同学**：可以让工程师同事跑给你看，重点不是写代码，是理解"哦原来 agent 内部是这么转的"
**关键产出**：每天定时跑的"个人简报 agent"——读你的笔记和 RSS feed，自动生成一份带引用的日报
**PDE 闯关在这里**：第一关大致就在这个阶段。后面专讲

Phase 2：用真正的架构搭一个真实 agent（3-4 周）

**目标**：在工业级编排框架上，搭一个**有持久化、有状态、能恢复**的 agent
**具体例子**：作者给的标准案例是 **"研究分析师 agent"**：

输入一个研究问题（比如"美团 2024 外卖业务护城河分析"）
主 agent 拆任务，派 3 个子 agent 并行去搜资料
子 agent 各自查不同来源、写中间笔记到文件
一个"引用核对 agent"检查每个论点有没有出处
一个"写作 agent"产出最终带引用的 markdown 报告
中途断电能从断点恢复，超过 1 美元 token 必须人工确认
**同类的可类比场景**（放到我们语境）：

客户经理日报助手（自动汇总今天客户进展 + 下一步建议）
研发周报自动生成器（从 git log + JIRA 拉数据 + 团队群消息抓上下文）
需求评审会议纪要 agent（边听会边记纪要 + 提问 + 整理 action item）
**agent 和 skill 在这里的关系**：见模块 4

Phase 3：自己拼一个最小可用的 agent 骨架（3-4 周）⭐ 全路线最关键

**目标**：不用现成框架，自己写一套 ~1500 行 Python 的最小 harness（agent 骨架）
**"harness" 这个词在这阶段最重要**：

不是新概念。**只要做了 agent 编排，背后必然有一个 harness——你只是平时没意识到**
你打开 Claude Code 时，背后的 harness 就在工作（决定什么时候调工具、什么时候压缩对话历史、什么时候问你权限）
这一阶段做的事：把这个**隐式存在的"管理层"，自己亲手拆开造一遍**
**目的不是产品化，是让你"看见齿轮"**——以后再用别人的框架，知道每个开关背后是什么
**类比**：不是要你造一辆能卖的车，是让你**自己拼一辆能开 50 米的卡丁车**，搞清楚发动机、刹车、方向盘怎么连
**真正的交付物**：1500 行代码 + **一份 1000 字事后复盘**——复盘比代码值钱
**为什么这是杠杆最大的阶段**：之前都在用别人造的车，这一阶段第一次让你看见齿轮

Phase 4：搭评测和回归体系（3-4 周）

**目标**：让每次改动可量化
**关键产出**：30-50 道金标题库 + 自动评分 + CI 门禁（不通过不让代码合并）
**类比**：给 agent 的开发流程接上 QA 和测试套件
**为什么这阶段最容易卡住**：搭出能跑的 agent 容易，搭出"改了一下后能说清楚比改之前好/差多少"的体系，难

Phase 5：上线后的运维（永远）

5 个方向（见模块 2 的第 6 条）
这阶段没有结束，永远在做


模块 4：Agent 和 Skill 到底是什么关系（重要）

最近大家都在说 "Skill"。Anthropic 官方推出了 Skills 系统，社区都在分享 skill 库，Claude Code 里有一堆 skill。

但**单独看 skill 没意义**。它必须装在 agent 里跑。

用餐厅类比

概念
餐厅类比
**Skill（技能）**
一张 SOP 卡片：「做番茄炒蛋时按下面顺序操作」
**Tool（工具）**
厨具：锅、铲子、燃气灶
**Agent（智能体）**
厨师本人：拿到"今晚要出 8 道菜"的命令，自己决定先做哪道、哪些可以并行、用哪张 SOP 卡
**Harness（工装）**
整个厨房：操作台、订单系统、订单流转、食材库存管理

用互联网产品类比

概念
产品类比
Skill
一个微服务 / 一个组件库
Tool
一个第三方 API
Agent
一个完整的产品
Harness
应用框架 + 后端基建 + 监控

为什么作者一上来不讲 skill 而讲 agent

**Skill 是组件，Agent 是产品**
你做产品时，先想"这个产品是什么"，再想"它有哪些功能模块"——不会先讲组件再讲产品
类比：不会跳过"汽车是什么"直接讲"火花塞是什么"

现在大家说的 skill 风潮，到底是什么意思

是 agent 工程的"零件市场"被分出来了
好处：组件化复用，一份 skill 多个 agent 共用
风险：把 skill 当成银弹，以为"我做了 100 个 skill 就完成了"——其实 skill 只是 agent 内部的一种实现方式
**正确姿势**：先想清楚要做什么 agent，再倒推它需要哪些 skill

为什么我们鼓励大家去闯关

PDE 闯关的设计目的不是"考核"，是 **"提供一份手把手的、有真实业务场景的、有交付检验的入门路径"**——比起自己看 Anthropic 文档摸索，**闯关帮你绕开 80% 的坑**。

完成闯关之后你拿到的是：

路线图 Phase 1 末尾要求的能力："能讲清 agent 内部怎么运转 + 能动手搭一个最小可用版"
实际的代码产物（不是 PPT）
跟工程师协作时不再"鸡同鸭讲"的术语储备
**进入 Phase 2-3 的资格**——而 Phase 2-3 才是真正决定能不能上线生产 agent 的阶段

一个重要的提醒

**闯关 ≠ 学完**。闯完关只是拿到入场券。

按外部路线图的 17 周计算，PDE 闯关大约对应前 5 周。后面 12 周（搭真实 agent / 自己写 harness / 评测体系 / 上线后运维）才是 agent 工程的硬核部分。

但前 5 周是 **"必经之路"**——没有这一段，后面 12 周是空中楼阁。所以闯关的价值不在闯关本身，在于它把你 **送到了能开始走后面 12 周的位置**。

模块 8：风险提示（路线图里的反鸡汤）

不全信路线图。作者也写了几条警示：

**Benchmark 分数是移动靶且部分被博弈**：单看"X 模型得 Y%"没意义，要看 harness 配置；学术论文记录 17 个流行 benchmark 普遍高估 5-33%
**多 agent 在大多数场景被过度吹**：90% 提升仅在"广度优先研究"成立；编码和耦合紧的任务多 agent 经常更差且烧 15× token
**Eval 套件会过期**：今天的金标题库几个月内就会被新模型饱和；每季度从生产失败 10-20% 的迭代不可少
**押抽象，不押库**：LangChain 自己 18 个月里把表述换了两次。框架会变，"loop / tools / context / 子 agent / 持久化 / 评测" 这些抽象不会变
