## 一、核心原则

**阻断优先**：P0 问题存在时，总分无意义，直接输出阻断结论，不进入评分流程。

**分层独立**：四层各自评分，不做跨层平均，防止高分层掩盖低分层的真实问题。

**置信度透明**：每层报告标注检测方式（自动 / 人工）和当前准确率置信度，让验收结论可信可追溯。

---

## 二、两条验收路径

### 路径 A · 新生产范式（AI Coding，无设计稿）

AI Coding 生成代码，无设计稿作为视觉基准。以「规范知识库」为语义底座，通过自动化工作流（Git Diff 扫描 → 结构树提取 → 规范索引 → 评分）完成主要检测，人工负责交互体验和状态覆盖的兜底。

**可上线准入条件（全部满足）：**

- 规范合规率 ≥ 90%（原子层 + 分子层 + 场景层三层均需达标，任一层不达标即阻断）
    
- 设计标准问题 = 0（含操作反馈缺失、热区不达标、标题符号违规等）
    
- 大屏适配无布局 Bug，小屏无内容截断
    
- 真实数据填充后无文案截断 / 溢出
    
- 关键异常态（空态 / 加载态 / 错误态）无设计标准问题
    

### 路径 B · 旧生产链路（有设计稿）

人工设计稿驱动研发，有明确视觉基准。以「设计稿还原度」为核心，通过图片比对 + 结构树节点匹配量化偏差。

**可上线准入条件（全部满足）：**

- 标准机型（375pt）设计稿还原度 = 100%（新容器数据层完全一致；旧容器依探索方案进展，置信度需标注）
    
- 设计标准问题 = 0（含操作反馈缺失、热区不达标、标题符号违规等）
    
- 大屏适配无布局 Bug，小屏无内容截断，异形屏安全区适配正确
    
- 真实数据填充后无文案截断 / 溢出
    
- 设计稿中的空态 / 错误态 / 加载态需一并验收，不可仅验收正常态

## 三、四层验收能力

### Layer 1 · 视觉还原层

|维度|内容|
|---|---|
|**验收目标**|路径 A：规范合规率 ≥ 90%；路径 B：标准机型还原度 = 100%|
|**检测维度**|色值（Token 体系）、字号 / 字重 / 行高、组件变体正确性、容器尺寸 / 圆角 / 背景色 / 阴影、间距 / 对齐 / 层级关系|
|**实现方式**|读取代码样式 + Git Diff 扫描 → 结构树提取 → 规范知识库索引比对（路径 A）；结构树节点匹配 + 属性逐项比对（路径 B）|
|**目标准确率**|结构化属性（色值、字号等）≥ 98%；相对位置属性（间距、对齐）当前 ~70%，迭代目标 ≥ 90%，报告中标注置信度|

### Layer 2 · 行为交互层

|维度|内容|
|---|---|
|**验收目标**|操作反馈缺失 = 0、热区不达标 = 0、标题符号违规 = 0，动效符合规范|
|**检测维度**|热区尺寸（≥ 44×44pt）、相邻元素间距（≥ 8pt）、点击 / 加载 / 成功 / 失败反馈、页面跳转 / 返回逻辑、动效时长与缓动曲线、标题文案符号规范|
|**实现方式**|自动检测：读取代码中热区属性、标题符号规则匹配；人工兜底：操作反馈、动效、跳转逻辑需人工确认，验收报告留开放补充入口|
|**目标准确率**|可量化项（热区、符号）≥ 98%；操作反馈 / 动效依赖人工，以「问题是否被记录」为完成标准|

> ⚠️ **动效规范当前盲区**：动效规范知识库完成度约 10%，自动化检测暂不覆盖动效维度。在知识库完善前，动效验收依赖人工对照规范文档进行，验收报告中显式标注「动效验收：人工确认」。

### Layer 3 · 适配兼容层

|维度|内容|
|---|---|
|**验收目标**|大屏（428pt+）无布局错位、小屏（320pt）无内容截断、异形屏安全区适配正确|
|**检测维度**|标准机型（375pt）、大屏机型（428pt+）、小屏机型（320pt）、异形屏（刘海 / 动态岛 / 底部安全区）、深色模式（如产品支持）|
|**实现方式**|调取云真机按机型截图 → 与库中已录入的场景 badcase 做图片比对 → 自动生成适配报告|
|**目标准确率**|布局级问题（错位 / 截断 / 遮挡）≥ 95%；深色模式色值问题 ≥ 90%|

### Layer 4 · 人工一票否决层

|维度|内容|
|---|---|
|**验收目标**|机器未识别但人工发现的、导致用户任务被阻塞的严重体验问题，直接阻断上线|
|**检测维度**|核心用户路径是否可完成、特定条件下的召回体验问题、无法被规则化的感知类问题|
|**实现方式**|设计师 / 产品经理人工走查，触发条件：「机器未识别」且「用户任务被阻塞（无法完成核心操作）」同时成立|
|**目标准确率**|不适用，人工判断即最终结论|

## 四、打分 / 算法方案

### 4.1 问题等级定义

|等级|含义|上线影响|等效问题数（用于评分）|
|---|---|---|---|
|**P0**|严重问题，用户任务被阻塞或视觉严重偏差|直接阻断，不参与评分|—|
|**P1**|明显问题，影响体验但不阻塞任务|不阻断，拉低健康度，需纳入下迭代修复计划|3|
|**P2**|轻微问题，用户基本无感知|不阻断，记录即可|1|
|**P3**|像素级偏差 / 平台渲染差异等合理差异|豁免，仅记录说明|0.3|

> P0 不参与评分的原因：一旦有 P0 存在，门控已经阻断，此时算分会产生误导性结论（如「有 P0 但得了 85 分」），因此 P0 只触发阻断，不折算等效问题数。

### 4.2 统一评分公式

```
某层得分 = max(0,  (总检测项数 - 等效问题数之和) / 总检测项数 × 100)

等效问题数之和 = P1数量×3 + P2数量×1 + P3数量×0.3
```

当等效问题数之和超过总检测项数时，得分取 0（表示该层问题密度极高，通常门控已阻断）。

### 4.3 两阶段验收流程

**第一阶段：门控检查（Pass / Fail）**

在计算任何分数之前，先做阻断项扫描。只要存在以下任一情况，直接输出「阻断上线」，不进入评分流程：

- 路径 A：任一层规范合规率 < 90%
    
- 路径 B：标准机型存在可感知视觉偏差（P0 级）
    
- 大屏 / 小屏存在布局 Bug 或内容截断（P0 级）
    
- 关键交互路径存在 Badcase（P0 级）
    
- 人工一票否决触发（Layer 4）
    

**第二阶段：分层健康度评分（仅在通过门控后计算）**

四层独立评分，汇总为综合健康度，用于反映页面质量水位和 P1/P2 问题的欠债程度。

### 4.4 分层评分示例

**视觉层示例**

某页面共检测 20 个属性节点，发现：圆角不符 ×1（P1）、间距偏移 >4pt ×2（P1）、1~2px 像素偏差 ×2（P3）。

```
等效问题数 = 1×3 + 2×3 + 2×0.3 = 9.6
视觉层得分 = (20 - 9.6) / 20 × 100 = 52 分
```

结论：无 P0，门控通过；52 分健康度偏低，3 个 P1 需纳入下迭代修复计划。

---

**行为层示例**

行为层共 10 个可验收项，发现：某按钮热区 38pt 不达标 ×1（P1）、某弹窗标题含全角符号 ×1（P1）、下拉刷新无加载动效 ×1（P2）。

```
等效问题数 = 1×3 + 1×3 + 1×1 = 7
行为层得分 = (10 - 7) / 10 × 100 = 30 分
```

结论：无 P0，门控通过；30 分健康度较低，2 个 P1 需纳入下迭代修复计划。

---

**适配层示例**

适配层共测试 20 个场景，发现：大屏导航栏文字截断 ×1（P1）、异形屏底部按钮被 Home Indicator 遮挡 ×1（P1）、小屏某图片轻微溢出 1px ×1（P3）。

```
等效问题数 = 1×3 + 1×3 + 1×0.3 = 6.3
适配层得分 = (20 - 6.3) / 20 × 100 = 68.5 分
```

结论：无 P0，门控通过；2 个 P1 需纳入下迭代修复计划。

---

**内容层示例**

内容层共 12 个验收项，发现：空态缺失 ×1（P1）、加载态文案不规范 ×1（P2）。

```
等效问题数 = 1×3 + 1×1 = 4
内容层得分 = (12 - 4) / 12 × 100 = 66.7 分
```

结论：无 P0，门控通过；1 个 P1 需纳入下迭代修复计划。

### 4.5 综合健康度

```
综合健康度 = 视觉层×35% + 行为层×30% + 适配层×25% + 内容层×10%
```

以上示例综合健康度 = 52×0.35 + 30×0.30 + 68.5×0.25 + 66.7×0.10 = 18.2 + 9 + 17.1 + 6.7 = **51 分**

> ⚠️ **权重待校准**：当前权重为基于「用户感知优先级」的经验估算。建议后续收集过往设计师验收记录，统计各层问题数量占比、严重程度分布、用户感知概率后，用真实数据反推权重。需要输入的数据维度：各层问题历史发生频率、各层问题导致阻断的比例、各层问题被用户感知到的概率。

### 4.6 验收结论输出格式

```
【门控结果】通过 / 阻断（列出具体阻断项）

【分层健康度】（仅在通过门控时展示）
  ├── 视觉层：XX 分（自动检测，置信度 XX%）
  ├── 行为层：XX 分（自动 + 人工，人工状态：已完成 / 待确认）
  ├── 适配层：XX 分（云真机，覆盖 X 个机型）
  └── 内容层：XX 分（自动检测）

【综合健康度】XX 分

【上线结论】
  ✅ 准入上线：所有 P0 = 0，P1 问题已纳入下迭代修复计划
  🚫 阻断上线：存在 P0 问题（列出具体问题）
  ⏳ 待人工确认：行为层 / Layer 4 人工走查未完成
```

## 五、阻断项汇总（P0 · 必须清零方可上线）

**视觉 & 规范类：**

- 路径 B：标准机型下存在可感知视觉偏差（字色错误、组件缺失、布局错乱等）
    
- 路径 A：任一层规范合规率 < 90%
    
- 使用了非规范色值、非规范字号（未走 Token 体系）
    
- 组件使用错误（错误变体或已废弃组件）
    

**交互 & 体验类：**

- 热区 < 44pt 导致用户无法正常点击
    
- 操作无反馈（点击无响应、加载无状态）
    
- 页面跳转 / 返回逻辑错误，用户被困在错误状态
    
- 大屏 / 小屏下出现内容遮挡、截断、布局错乱
    
- 真实数据填充后文案溢出容器或被截断

## 六、验收流程

1. **触发验收**：研发提测 / AI Coding 完成
    
2. **自动化检测**：知识库扫描 + 图片比对（Layer 1 / 2 可量化项 / 3）
    
3. **门控检查**：扫描 P0 阻断项，存在则直接阻断
    
4. **生成报告**：问题分级 + 定位代码 + 分层健康度评分
    
5. **人工复核**：行为层交互 / 动效 / 状态覆盖（Layer 2 人工部分 + Layer 4）
    
6. **修复迭代**：P0 清零后重跑自动化检测
    
7. **准入上线**：所有 P0 = 0，P1 纳入下迭代计划
