# 洋葱简介

[洋葱](https://yangcong.sankuai.com/)是一套UI验收提效解决方案，通过为开发者和为设计师提供丰富的诸如真机UI测量、微调与差异比对、标注等能力，大幅简化UI验收工作的复杂度、提高开发视觉交付质量，提高产研协作效率。

我们放上一段基于洋葱的UI验收工作过程演示，读者观看后会对此有一个直观的感受和理解。

# 背景

终端业务是和用户交互比较强的业务，终端开发的重要职责之一便是将用户需要的数据以更准确、合理的方式展示给用户，并响应用户的各种交互行为。在此期间，终端业务的设计、研发无时无刻不在和UI的展示打交道，UI的开发和修改占据了整个业务交付流程中的很多工时。如下图所示，终端在需求开发过程中大部分的节点都与UI有关，而研发和设计同学分别作为UI的实现方和设计方，相互之间的沟通和协作存在较多的效率问题。

研发同学在需求开发完成后需要交给设计同学进行验收，如果验收出现bug，会返到研发同学进行UI bug的修改，修改完以后研发同学会再次提交验收。如此的轮次验收，在一些需求上甚至可以回滚三次，部分研发甚至需要多出一天工时来专门修改UI bug问题。整个验收流程走完，整个过程会产生较大的研发与设计资源工时投入。

![[../../../_attachments/Pasted image 20260520155257.png]]

在没有有效的UI检测和验收工具时代，针对终端业务UI的开发和验收，我们遇到了上述很头疼的问题，这些问题不仅会影响到研发同学的开发效率，还会影响到最终的UI提测质量。如果出现疏漏可能将UI问题带到线上，最终可能会影响用户的使用体验。

针对于这些遇到的问题，我们做了一个总结，主要有如下几方面原因：

- **多次重复打包：**在研发和验收阶段，研发同学需要针对UI修改频繁地进行打包，有时候1px的修改都需要进行一波重新打包的操作。
    
- **缺少工具自测：**在研发和验收阶段，研发同学在完成开发后，只能大概看看“UI长得像不像”，“对不对”很难判断，设计同学也是缺少有效的工具去测试UI。
    
- **重复构造数据：**在开发阶段与验收阶段，后端接口往往没有上线，研发、产品经理、设计 三方都需mock数据，而产品经理和设计对数据与UI之间的对应关系是无感的，最终只能求助研发mock数据，造成了大量人力浪费。
    
- **反馈途径不一：**在进行验收，设计同学只能靠肉眼看，或者截图到Sketch中去对比，再通过大象、口述等等方式反馈给研发。
    

我们的目的就是为了消除掉灰色框里的那些耗时点。

# 目标

解决UI开发、提测、验收各阶段耗时长、UI自检精确度不高的问题，提升UI开发效率、提高UI验收质量是最直接的目标。

- 在研发阶段，减少研发同学UI调试成本。
    

- 在视觉验收阶段，减少设计同学验收成本。
    

# 方案设计

## 设计思路

对于不同开发节点产生的问题，我们需要针对性地开发不同的工具去解决，最后将这些工具集成在一起，就形成了我们的终极UI检测方案。

- 如何解决用户打包过于频繁的问题
    
    我们的想法是既然用户打包时间过长，能否让用户不进行打包操作就能直接看到修改后的效果呢？
    
- 针对缺少工具自测的问题
    
    我们考虑到，研发同学是无法用肉眼看出30dp和31dp的差别的，只能将具体数字展示出来，研发同学才能意识到错误。或者将实现图和视觉原型图重合在一起，这样就能很清晰地看出来二者之间的不同了。
    
- 针对mock数据的问题
    
    我们考虑使用提前录制case数据，能下拉到客户端，后续研发和设计同学在使用时可以一键mock数据，省去了找人、找数据等沟通时间。
    
- 针对验收时，设计同学验收不方便，问题标注方式多样等问题
    
    我们设计了一套方案，和印迹平台打通，让设计同学在移动设备上发现问题时可以直接标注问题，并将问题提交后便可立即反馈给相应的研发同学。
    

## 架构全景图

![[../../../_attachments/Pasted image 20260520155321.png]]

下面将洋葱平台的整体架构按照从下到上，从左到右的视角进行解读：

- 最底层代表着需要验收的洋葱的APP视图层，由于洋葱直接嗅探了原生视图层组件，所以支持包括Android原生、iOS原生、MRN、Picasso等在内的底层基于原生视图体系构建的应用视图，
    
- Inspector层：该层能够发现和定位视图元素、提取视图元素的属性、布局层级、截图，并能够运行时实时修改视图属性。
    
- Pxe Layer层：该层提供在运行时手势的拦截与处理并予以信息反馈的能力，该层是衔接洋葱底层能力与应用场景的桥梁，对下能借助Inspector降触控意图对应的组件进行属性嗅探与修改，对上对接洋葱的功能交互能力。
    
- 功能化插件层：该层是对应UI验收中具体应用场景的一系列功能插件，包括运行时UI元素测量、属性浏览与标注、间距嗅探、界面元素取色、动画调速，标注截图、打标、注释等功能，该层可基于插件协议进行持续的功能扩展。
    
- UI自检层：包括当前验收工作的印记设计稿关联、截图、自动对比。
    
- 环境模拟设施：包括mock数据治理（汉堡平台）和设备模拟功能，mock数据治理提供mock数据管理、appmock数据同步等功能方便验收期提供与提供设计稿一致的展示层所需数据；设备模拟提供修改像素密度等设备仿真功能。
    
- 再往上，就是垂直的业务，洋葱提供了接近于零侵入式的接入方式，目前包括到餐的各子业务和包括外卖、酒旅在内的多个BU已经接入使用。
    

## 方案详情

### 基础测量功能

目标：基础测量的目的是提供给研发和设计同学使用的通用测量功能，为了解决研发和设计在进行UI验收的时候没有有效的量化工具的问题。

方案：基础功能设计分为两块内容，第一块就是下面所描述的通用测量功能。通过基础测量功能，能够直接获取到Android设备和iOS设备上控件的各种属性。这些属性不仅研发同学能够看得懂，设计同学也能够看得懂。以此为目的，我们以View树的构建和解析为基本原理在功能上进行了开发。

![[../../../_attachments/Pasted image 20260520155337.png]]

整个页面都是以View为基本单位进行组织的，在native中存在着多种页面或者是类似于页面的弹框，我们对这些“页面”进行了统一化处理：

1. 首先获取到这些页面的根View，以这个根View作为View树的根节点，遍历出这个页面的所有View，形成一个View树。
    
2. View树形成后，我们就可以根据点击Point获取处于点击Point的View。
    
3. 如果在某一固定的区域有多个View或者ViewGroup，那么就优先取叶子节点的View，重复点击后进行向上遍历。
    

View树形成后，就可以根据点击Point来匹配对应的View，如果在某一固定的区域有多个View或者ViewGroup，那么就优先取叶子节点的View，重复点击会向上遍历。

以【间距功能】为例来介绍一下通用功能的具体实现，下图展示了间距功能提供的能力。
![[../../../_attachments/Pasted image 20260520155349.png]]
![[../../../_attachments/Pasted image 20260520155355.png]]

洋葱是如何实现这个功能的呢？主要依托于以下两点：

- 如何获取点选的View？
    
- 如何计算点选的两个View之间的间距？
    

首先，我们来看一下【如何获取点选的view】这一问题。

![[../../../_attachments/Pasted image 20260520155404.png]]

从上图中，可以清晰地了解到，关键点在于 给一个浮层加一个手势，这个浮层会接住用户点选手势。当接收到点选手势后，就可以获取到点选的点Point，

下一步遍历所有Window上的View，找出所有包含此Point的View，取最上层的View即可。

接下来，我们再来看一下如何计算两点选View之间的间距。

![[../../../_attachments/Pasted image 20260520155413.png]]

这里的关键点在于将获取到的两个View的坐标系统一到同一个坐标系中，在统一的坐标系中，获取View的frame，得出这两View的位置关系（相交、相离、包含）。

根据位置关系去决定计算各个间距，并作以显示。

### 自动化mock工具

目标：自动化mock工具是提供给研发和设计同学能够方便的进行构造数据、存储数据和使用数据的平台。研发录入数据case后，case永久保存，设计同学在验收阶段便能自主验收，不需要再找RD进行验收。

方案：洋葱构建了一个快捷验收系统，可以在手机上0成本构建验收和回归数据，一键直达测试场景。下图是对mock功能的拆分和分析。

![[../../../_attachments/Pasted image 20260520155422.png]]

如图描述一键mock有操作简单、实现成本低、数据方便扩充使用、数据扩展性强等目标，于是对目标进行了拆解和分析得到解决方案。

- 首先是操作步骤简单，点击即可获取配置数据，点击即可就直达场景。
    
- 其次是实现成本低，开发周期短，那么就要在在已有工具上开发，这里洋葱选择了APPMOCK二次开发，
    
- 然后是希望可以一人造数据，多业务方使用，那么需要需要数据存储中心，于是洋葱选择了平台的Avatar数据库存储。
    
- 最后需要做到数据覆盖多种数据场景，那么洋葱的做法是，提供录入数据平台，根据一份数据自动生成多种数据例如超长、空数据、异常数据等。
    

基于上述分析，我们做出了如下图所示的架构设计。

![[../../../_attachments/Pasted image 20260520155435.png]]

mock功能主要涉及客户端（Android/iOS）、Hamburger服务端、Hamburger前端、数据库、APPMOCK平台，其可以视作一个半自动化mock数据管理与数据重现平台。

此功能是基于APPMOCK平台公共API实现的。在开发阶段，研发同学按照需求纬度将不同场景（譬如超长展示场景）对应的数据录入到数据库中。当研发同学或设计同学需要时，在洋葱工具中就可以轻松获取到某一需求所有场景数据，最后就可以针对某一case进行一键mock、一键跳转到目的页。

另外，为了能够快捷地获取空或超长场景数据，Hamburger后台还提供了空数据和超长数据的mock功能供使用者使用。

### 研发自检工具

目标：研发自检工具是提供给研发同学使用的视觉还原工具。使用该工具，研发同学可以方便的进行UI调试、在不打包的情况下便可随意修改UI属性，以达到减少打包频率的目的。且方便的图片对比能力也能够方便研发同学进行自主UI验收。

方案：为提高研发同学在UI开发阶段的视觉还原度，我们开发了UI调试、半自动化验收工具，以帮助研发同学进行UI黑盒、白盒验收。

![[../../../_attachments/Pasted image 20260520155449.png]]

在洋葱中，通过抓取点击Point处的native的View树并展示，以方便研发同学开发和调试。研发同学可以清晰地看到选中View所在的层级结构，工具会将该View的类名、宽高、内外边距属性、文本内容、字体字号、文字颜色、背景色等显示出来，并可以对View的属性进行动态的调整，实时调整实时生效。另外，工具还提供了View拖动功能，帮助研发同学对View的位置进行调整。

在一切完备后，研发同学可以操作工具上传实现图。上传成功后，在UI检测门户网站中的【UI检测功能】页中，通过图片对比算法可以自动计算处两图之间的差异，并将两者之间偏差较大的地方显著标记出来，让研发同学能快速检查出可能有问题的点。除此以外，研发同学还可在【UI检测功能】页中随意对实现图进行拖动，来对细节进行精确比对。对比图是将设计稿的UI图灰度化，然后UI图和实现图自动对比将对比出的不同的位置标上红色。通过对比图可以看出实现图和UI图在颜色方面的不同，比如区块颜色，文字颜色等，由于实现图和UI图中的文字字体可能不同，生成的对比图上的文字边缘可能会标上红色，如果对比图中的某个文字没有被红色完全覆盖，说明实现图和UI图对比位置的文字颜色相同。如图所示：

![[../../../_attachments/Pasted image 20260520155459.png]]

同时点击UI图、实现图、对比图的某个位置可以弹出像素图，通过像素图可以查看3张图片的具体位置的像素，比如点击对比图中"中"这个位置，如图所示：

![[../../../_attachments/Pasted image 20260520155508.png]]

像素图中的选框表示当前选中的像素，UI图、实现图、对比图会同时选中相同位置的像素以便进行像素比较，UI图的像素图、实现图的像素图下面会显示当前选中像素的颜色值，如果UI图、实现图在该像素处颜色不同，下面显示颜色值的区块会变红。点击像素图中的某个像素可以将该像素设为选中像素，另外像素图可以在鼠标左键按下同时移动鼠标时进行上下移动。使用洋葱研发自检工具来进行视觉还原自检，有效的减少了开发阶段的UI问题。

### UI自检平台

目标：UI自检平台是提供给设计同学使用的验收、标注、反馈的一体化平台。这个平台将验收工具和设计平台打通，通过图片对比简化验收过程，并实现客户端问题标注和一键反馈的功能。解决了设计同学验收时无有效验收工具、验收效率低、反馈路径多样的问题。

方案：UI对比和标注功能是专门为设计同学提供的验收、标注、反馈的一体化平台。

![[../../../_attachments/Pasted image 20260520155523.png]]

设计同学在移动设备上根据设计稿ID（设计稿ID可从[印迹平台](https://ingee.meituan.com/)获取）拉取设计稿后，就可以在移动设备上直接进行设计稿和实现效果的对比。如果实现效果达不到设计稿预期，设计同学可直接在移动设备上标记出问题。标记问题时，洋葱会将标记相对于整个屏幕的坐标记录下来。当进行标记问题上传时，洋葱会将实现效果图和标记问题列表一并传给印迹，印迹解析后标注到前端生成一个错误标注的链接，反馈给研发同学进行修改。

### 屏幕规格模拟

通过上面的介绍，可以看到研发自检方案和UI自检方案中都存在一个问题，即需要将设计稿和实现图进行重叠后的对比，但这样就会产生一个问题，特别是对于Android设备来说，不同分辨率所显示出来的1dp的大小换算到英寸显示的大小是不一样的。设计稿只有一张图，进行对比的时候是需要宽度填充满整个屏幕的。所以，如果分辨率不一致导致了客户端显示尺寸不一样，那么和设计稿进行对比是完全没有意义的。所以，我们需要想办法将各个 机型的分辨率进行统一，使得不同机型展示的UI效果尽可能地接近。

![[../../../_attachments/Pasted image 20260520155540.png]]

在美团的UE/UI的设计规范中，输出的设计稿尺寸是等效iPhone 7分辨率；经过换算，Android屏幕横向宽度就是375dp。以此为切入点，我们对手机的APP显示分辨率进行了换算和改造。

在洋葱中，我们可以修改APP的配置，对整个APP的屏幕像素密度重新计算，换算成iPhone 7大小，然后强制重刷页面，让页面宽度以375dp来显示。这样，在进行对比时，就可以严丝合缝地将设计稿和实现页对齐了。
