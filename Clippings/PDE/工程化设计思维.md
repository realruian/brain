# 工程化设计思维

想提升 AI Coding 能力的设计师

核心、通用 的概念模式

- **空间结构**的同构:  图层即 HTML ———— 「像素级」掌握样式和布局, **有效提高设计稿的代码还原度**;

- **复用逻辑**的同构:  组件化即封装逻辑 ———— 「组件化」思维转变, 学会并理解设计/代码中「封装重复」的逻辑, 具备复杂模块的改动资质;

提高代码判断力与工作效率, **掌握与前端协同时的 debug 思路**, 在生产环境稳定交付高质量前端代码, 实现全链路交付。

## 1 引入

设计师上手 AI Coding 不乏遇到以下问题:

- 工具方法迭代快、旧教程针对性不强 —— **学不完,** **靠踩坑碰运气缓慢积累经验;**

- 效果难以控制、结果难以判断 —— **控不准,** **难以投入生产环境;**

- 不知道原来做不了 / 原来可以这样做 —— **沟通Gap, 被技术限制打回**。

现状是, 简单任务, 可以通过多磨几轮对话解决;

但复杂任务, 当质量要求变高、流程变长、细节变多, 需要持续的迭代时, 如果还用“改一下、跑一下、看一下”的碰运气式调整, 会异常痛苦…

**不仅耗费更多时间、更多算力, 产出还不稳定、难以达到任务要求**。

### 1) 专业能力还重要吗?

“AI 生成又快又好, 人都要失业了, 现在还一点一点学那些专业知识, 有意义吗?”

**1.算力使用效率与专业能力**

…… 分享一个最近听到的概念: **[「算力效率比」(CER, Compute Efficiency Ratio) ](https://www.bilibili.com/video/BV1GDw6ztEY4/?vd_source=9b96597b33ef1e7f48e462fe1f876ebf)**

**定义：** 在同样的时间，用同样能力的AI模型，一个人需要消耗多少算力，才能把某个难度标准下的事情做完做好。

|  | 高效的人 (CER = 1) | 差距 | 低效的人 (CER = 2) |
| --- | --- | --- | --- |
| 简单任务(难度 = 10) | 算力消耗 = 1 × 10 = **10** | 10 | 算力消耗 = 2 × 10 = **20** |
| 复杂任务(难度 = 1000) | 算力消耗 = 1 × 1000 = **1000** | 1000 | 算力消耗 = 2 × 1000 = **2000** |

**启示：**

- CER 的差异, 会随着任务复杂度, 加倍放大;

- CER 值越低，代表一个人在 AI 时代的工作能力越强。

而 CER 要怎么提高? —— 要知道**如何准确高效地指导 AI 工作 = 你自己要有 KnowHow**。

可以说, AI 并没有降低专业能力的重要性, 反而是专业能力的放大器……而 **不用人类执行的表象, 会让人陷入“专业能力不重要”的陷阱**。

**2.意图密度与细节打磨**

…… 还想分享一个有所启发的概念, 叫[**「意图密度」 (Intention Density)**](https://www.xiaohongshu.com/explore/69a2b96d0000000026031628?xsec_token=ABh71UV4bEFfKneURBuL3dYVmiFAsMV17IsYbiT85jGZg=&xsec_source=pc_collect)

**定义**: 生成结果中, 有多少来自使用者的主动决策, 有多少来自“通常就是这样”的默认路径。

通常来说, 创作过程可以拆成2部分的循环: **意图(Intention)**, 你要做什么, 为什么; **执行(Execution)**, 你怎么把它做出来。

AI 正在不可阻挡地胜任 执行层面 的工作, 然而, 当人放任 AI 去替代 意图层面 的工作时, 内容就开始变得空洞、变得千篇一律……

正是因为 AI 生成的内容唾手可得, **有高意图密度的、能够和 AI slot 拉开差距的内容将会更为稀缺**。

放到设计领域, 大家在实践中不免发现, AI 生成的 UI 细节往往差点意思……

设计师 Coding 的价值在于, 把那些只看代码无法评判, 但实际用起来构成产品方方面面的体验和观感的细节揪出来、打磨好,

这些外行人说不出所以然、感知不出问题和差距的地方, **正是为产品赋予“品味”和“灵魂”的关键, 是设计师在 AI 时代的护城河**。

> **细节的执念(Craft Obsession),**是海外设计工程师JD中出现最密集、描述最具体的特质

**3.设计角色转型与工作方式改变**

专业能力决定你的 CER，意图密度决定你的产出质量，而[**「设计工程师」(Design Engineer)**](https://km.sankuai.com/collabpage/2752193814)是这两者都做好之后自然走向的角色定位。

综合 17 份有效的海外设计工程师招聘 JD 人才画像:

Design Engineer 是**用工程能力实现设计品质、用设计品味指导工程决策**的复合型人才，其核心价值在于消除设计与实现之间的损耗，**独立完成从问题定义到生产上线的全链路**。

**与传统设计师对比**

| 对比 | 传统产品设计师 | Design Engineer |
| --- | --- | --- |
| **核心交付物** | 设计稿、标注文档、原型 | 可运行的代码原型、设计系统 |
| **工具栈** | Figma / MasterGo、Ae / Rive… | Figma + Cursor / Claude Code + Github / Gitee |
| **与研发的关系** | 移交设计稿, 等待还原 | 直接参与实现, 减少理解损耗 |
| **用研方式** | 访谈、问卷、可用性测试 | AI 辅助分析 + 定量数据 + 传统研究 |
| **衡量成功的标准** | 设计方案被采纳 | 产品指标被改变 |
| **话语权来源** | 设计专业判断 | 设计判断 + 可验证的原型 |

这意味着, 善于利用 AI Coding 的设计师, 其定位不是止步于设计稿交付的角色, 而是能够且擅长**用工程能力, 解决设计问题**的人。

**可参考的能力要求分布**

| 能力维度                   | 提及频次  | 典型描述                                                       |
| ---------------------- | ----- | ---------------------------------------------------------- |
| 前端工程（React/TypeScript） | 17/17 | "proficient in React / NextJS"                             |
| 端到端产品交付                | 16/17 | "from concept to production"                               |
| 产品感知（Product Sense）    | 15/17 | "you know when something feels right"                      |
| 设计系统构建与维护              | 14/17 | "own the design system"                                    |
| 动效与交互设计                | 13/17 | "fluid animations, expressive gestures"                    |
| Figma 熟练使用             | 12/17 | "mastery of Figma"                                         |
| 用户研究与需求理解              | 8/17  | "work closely with users to deeply understand their needs" |

这意味着, 有意愿向 Design Engineer 转型的设计师, 为了胜任稳定的闭环交付能力, 有必要掌握一定的工程知识和方法, 而不是只依赖于工具能力或盲目地 AI 对话。

**4.工具和工作流会变, 但底层逻辑恒常**

为了“弥合设计与代码之间的差距”, 业界不乏产生了很多基于不同实现原理、针对不同环节的工具和产品:

- 低代码建站工具: Framer、Webflow…

- 设计工具: figma、MasterGo…

- 切图工具: 蓝湖、Figma Dev Mode…

- 设计转代码工具: Builder、Anima、Figma2Code插件、MasterGo 的 NoCode D2C插件…

- AI Coding 工具与模块: Cursor 的 Visual Edit、直接与 IDE 集成的画布 [Pencil.dev](http://Pencil.dev) 、[Paper.design](http://Paper.design)…

- 设计稿与 IDE 的连接方案: MCP、API、CLI …

然而, 当前的补丁式工具无法完好解决设计从生产到落地各个环节的问题, 要在补丁上打补丁。

一方面, 会导致使用过程更加繁琐而累赘; 一方面, 每个补丁也都需要投入研发资源、需要等待, 不能第一时间解决问题……

事实上, **理解一些代码知识就能绕过一个又一个补丁式的工具, 相比起来, 学习的性价比很高**……原理层面的东西, 不随工具变化而颠覆。

### 2) 那么, 要怎么学?

| 过去的学习 —— 遍历式 | AI 时代的学习 —— 渐进披露, 按需深入 |
| --- | --- |
| 完整的学科框架, 一门学完再一门 | 理解核心概念、熟悉流程 |
| 写大量代码/设计稿 | 积累经验、建立KnowHow |

反观我从开始接触25年初, 到如今熟练地将 AI Coding 纳入生产力[↗](https://km.sankuai.com/collabpage/2748864156)26年初[](https://km.sankuai.com/collabpage/2748864156), 经历过几个关键 Aha Moment:

1. 发现 **图层** 和 **html标签** 是对应的;

2. 发现 **设计组件** 和 **代码组件** 是对应的;

3. 发现 **交互逻辑** 和 **代码逻辑** 是共通的.

…… 之所以能够实现有效转变, 是因为设计流程本身有很多已经在画的图、已经在用的方法, 和工程思维是同构的!

所以能打通的前提是设计侧的基本功, 先要有才能与代码侧融会贯通

作为设计师, 要做的不是"学编程语法", 而是**把设计概念作为“认知锚点”, 随着实践不断补全代码侧的知识框架, 按需求渐进式掌握领域知识, 积累具体的经验**。

所以, 什么是工程化设计思维?

理解代码原理, 反哺设计习惯, 弥合设计思路与代码实现间的Gap —— 让设计即代码, 事半功倍

## 2 工程化设计思维

> 注: 本篇会讲很多“是什么”、“为什么”、“怎么做”, 关键意义在于习得「KnowHow」

### 1) 共通点1: 图层 即 HTML

> 静态代码样式、空间结构的同构

#### 对比: 图层、样式与布局约束

设计师用图层组织视觉层级，工程师用盒模型组织页面的 DOM [^3]结构，两者描述的是同一件事 —— 元素间的包含、排列、间距关系。

**1.图层与结构**

在设计稿里怎么分层，DOM 树就是什么结构，两者可以一一对照, 更多详见: [设计说【AI Coding系列】第一讲 —— AI 友好设计稿](https://km.sankuai.com/collabpage/2732066436)。

无论在改什么技术栈, 只要改的是页面元素的结构关系, 改的就是`<标签名称>`类的 HTML;

标签名称、注入的属性、包裹的内容, 决定了它是什么类型的“图层”。

| 设计 | 代码 |
| --- | --- |
| 图层 / Frame | HTML标签 (渲染后成为DOM节点)<br>编译前, 原码 HTML 标签 编译后, DOM 节点<br>普通图层<br>带自动布局的图层<br>带绝对定位的图层<br>文本类图层<br>组件类图层 |
| … | … |

**2.样式与CSS**

设计稿中选中图层后, 侧面板已设置的样式 = 每层 HTML 节点中的CSS参数:

无论在改什么技术栈, 只要改的是页面元素的样式与效果, 改的就是每层节点中`样式属性:参数`或`样式属性-参数`的 CSS。

|  | 设计字段 | CSS 属性 | TailwindCSS 原子化CSS | 备注 |
| --- | --- | --- | --- | --- |
| **示例** |  | 古典 CSS 写法, 有内嵌、CSS类封装[^4]等写法 | AI 友好的 CSS 写法, 更简短、原位生效、便于调试和维护 |  |
| **位置** | 元素对齐 | `display`, `justify-content`, `align-items` | `flex`, `justify-center`, `items-start` | 控制布局主轴与交叉轴的对齐方向 |
| 文本对齐 | `text-align`, `vertical-align` | `text-left`, `text-center`, `align-middle` | 文本内容的对齐方式 |  |
| 内边距 | `padding`, `padding-top`, `padding-left` | `p-1`, `pt-2`, `px-auto` | 元素的内间距 |  |
| 外边距 | `margin`, `margin-top`, `margin-left` | `m-4`, `mt-2`, `mx-auto` | 元素的外间距, 设计稿中不支持设置 |  |
| 旋转 | `transform: rotate(deg)` | `rotate-45`, `-rotate-6` | 支持任意角度旋转 (需启用 transform) |  |
| **布局** | 布局类型 | `display` | `flex`, `grid`, `block`, `inline-flex` | 控制主容器的布局方式 |
| 尺寸 | `width`, `height` | `w-64`, `h-32`, `w-full` | 固定或自适应尺寸 |  |
| 最大/最小值 | `max-width`, `min-height` | `max-w-md`, `min-h-screen` | 响应式或弹性布局时常用 |  |
| 固定比例 | `aspect-ratio` | `aspect-square`, `aspect-video` | 保持图片或组件的固定比例 |  |
| **样式** | 填充(背景色) | `background-color` | `bg-gray-200`, `bg-[#fafafa]` | 支持自定义或调色板颜色 |
| 效果(混合模式) | `mix-blend-mode`, `background-blend-mode` | `mix-blend-multiply`, `bg-blend-overlay` | 与背景图叠加效果 |  |
| 描边 | `border`, `border-color`, `border-width` | `border`, `border-2`, `border-gray-300` | 包括边框宽度和颜色 |  |
| 不透明度 | `opacity` | `opacity-50`, `opacity-80` | 元素整体透明度 |  |
| 圆角 | `border-radius` | `rounded`, `rounded-lg`, `rounded-full` | Tailwind 默认支持 8 种圆角等级 |  |
| **效果** | 投影 | `box-shadow` | `shadow`, `shadow-lg`, `shadow-inner` | 可自定义阴影层级或扩展主题 |
| 背景模糊 | `backdrop-filter: blur()` | `backdrop-blur`, `backdrop-blur-md` | 需浏览器支持`backdrop-filter` |  |
| **状态** | Hover | `:hover { background-color: #2563eb; }` | `hover:bg-blue-600` | 当鼠标悬浮时变换的样式/效果 |
| Active | `:active { transform: scale(0.97); }` | `active:scale-95` | 当控件激活/按下时变换的样式/效果 |  |
| Focus | `:focus { outline: 2px solid #60a5fa; }` | `focus:outline focus:outline-blue-400` | 当控件被聚焦时变换的样式/效果 |  |
| Disabled | `opacity: 0.5; cursor: not-allowed;` | `disabled:opacity-50 disabled:cursor-not-allowed` | 当控件禁用时变换的样式/效果 |  |
| **交互** | 鼠标动作 | `cursor: pointer;` | `cursor-pointer`, `select-none` | 当满足条件时, 鼠标执行的效果 |
| 动效过渡 | `transition: background 0.2s ease;` | `transition-colors duration-200` | 执行变换时遵循的动画曲线 |  |

**3. 布局约束与自适应**

> [演示跟练设计稿](https://www.figma.com/design/hBTtJPc3iUhDRBVJPcLvug/%E8%B7%9F%E7%BB%83%E7%B4%A0%E6%9D%90?node-id=3064-329&t=OH4y5GIJfAp0Tttr-1) —— 进入后, 标题下拉菜单 > Duplicate 复制为自己可编辑的设计稿

布局约束能让设计稿变"活"，元素之间有关系，有方向，有间距规则，能响应变化。这不只是效率问题，是设计稿能不能被工程实现的问题。

缺少“约束设计”的设计稿是"死的"—— 元素位置是手动摆的，一旦内容变化就全部乱掉。

练习/自查方式: 找一个需要代码 ready 的设计稿,  拖拽容器, 按照下方要点 一一检查每个层级的布局约束

善用原型预览、常常拖拽容器, 像实现设计稿效果一样, 设置并排查前端页面。

| 概念 | 设计 | 意义 | 代码 |
| --- | --- | --- | --- |
| **尺寸约束:**<br>固定<br>适应<br>填满 | 常见 | Figma: <br>MasterGo: | 在不同的宽高空间下、<br>在或多或少的内容下, <br>定义每个容器的“延展性”, <br>确保各种情况下展示协调<br>**无视会怎么样**: <br>D2C的容器都是“硬”的, 稍微变换尺寸就错位<br>难以发现、难以修复基础前端问题 |
| **尺寸极值**: <br>极大值 max<br>极小值 min | 按需 | 组合尺寸约束, 定义边界场景<br>**无视会怎么样**: <br>进阶的布局效果无法实现<br>难以发现、修复进阶的前端问题 |  |
| **溢出截断**: <br>内容溢出: <br>裁剪超出容器的内容<br>文本溢出: <br>省略超出容器的文本<br>图片溢出: <br>按什么方式填充图片 | 常见 | Figma: <br>MasterGo: | 内容裁切: <br>确定裁切边界, 平衡空间与留白<br>文本溢出: <br>为动态字段的展示空间预留裁切边界, 避免溢出问题<br>**无视会怎么样**: <br>页面导入真实数据后, 内容顶出页面, 导致变形/遮挡 |
| **绝对定位**: <br>固定相对方向<br>固定比例缩放<br>固定边缘距离 | 常见 | Figma: <br>MasterGo: | 异形 UI 切图实现<br>特殊位置 UI 实现: <br>角标、悬浮窗、遮罩<br>**无视会怎么样**: |
| **断点**: 多端响应式 | 按需 | 插件支持 | 一个前端页面适配多端展示尺寸 |

#### 作用: 样式与布局「像素级对齐」

本模块不是在要求“一定要按这种方式画稿”, 而是在指引一种“精细调节UI细节的有效思路”, 尤其是**距离上线最后5%的细节问题**。

| 费力度对比 | 设置设计意图 | 2.  验收与调试 |
| --- | --- | --- |
| 设计稿: 自由度更高、更快 | (在知道的前提下)在设计稿把参数设置清楚 = 点两下 | 随意拖拽、输入、清空、原型预览 |
| 代码: 自由度更低、更笨重 | 在代码把参数设置清楚 = 定位代码 + chat + check | 依赖开发者工具、Mock数据、定位代码才能调出特定状态 |

因此, 如果能前置在设计稿定义清楚, 可以节省代码环节的调试功夫; 如果在代码侧发现了问题, 可以映射在设计稿的设置上, 设计稿怎么修、代码怎么修。

终态: 如果在设计稿调试的经验很充足, 那甚至可以不依赖设计稿, 直接快速调试代码。

| Before: 越不会调, 越需要调 —— 事倍功半 | After: 越会调, 越不用调 —— 事半功倍 |
| --- | --- |
| 做设计稿: “为什么还要收拾图层, 好麻烦”<br>D2C后……“为什么还原度那么低?” | 做设计稿: **把对布局、样式的设计意图, 在设计稿组织清楚**; <br>D2C: 从源头收束还原问题, 提供清晰的定义。 |

#### 对比: 组件、属性与复用关系

设计师用组件库管理可复用的 UI 单元，工程师用 React/Vue 组件封装可复用的代码模块，两者描述的是同一件事 —— **把"重复出现的东西"抽象成一个有名字、有参数、可配置的单元**。

**1.组件与属性**

不同技术栈下, **组件/实例/Props 的概念模型一致**。

|  | **设计** | **代码** |
| --- | --- | --- |
| 示例 |  |  |
| 共通点 | **组件**(Component): 可被复用的 UI 单元 | `function Button() {}` `defineComponent({ name: 'Button' })` `struct Button: View {}` |
| **实例**(Instance): 子组件, 随母组件改动而改动 | `<Button />` 普通 html 标签 | `<Button />` 同 React `Button()` 直接构造 |
| 变体中的**交互状态**(default/hover/active/disable)<br>设计稿只能把状态(state)包在“变体”里 | CSS 伪类 + State：<br>`:hover` / `useState` | CSS 伪类 + 响应式：<br>`:hover` / `ref()` `.buttonStyle` + `@State var isPressed` |
| **变体**(Variant): 同一组件的不同形态 | Union Type Prop：`variant="primary"` | String Prop：`variant="primary"` enum Prop：`variant: .primary` |
| **布尔属性**(Boolean Property): 控制某部分展/隐藏 | Boolean Prop 布尔值：`showIcon={true}` | Boolean Prop：`:showIcon="true"` Bool Param：`showIcon: true` |
| **文本属性**(Text Property): 组件内可替换的字符内容 | String Prop 字符串：`label="提交"` | String Prop：`label="提交"` String Param：`label: "提交"` |

**2. 拆分边界的判断 —— 化整为零, 便于维护和拓展**

一个组件只做一件事。如果一个组件承担了太多职责(耦合度高)，它就很难被复用，也很难被修改。

原子化能力的封装组合, 才能构成可持续维护、拓展的复杂系统。

因此, 与研发的代码交付, 应该尽可能拆分、可验证、最小闭环。

|  | **设计侧** | **代码侧** |
| --- | --- | --- |
| **什么时候该封装?** | **复用性拆分**: 多个页面重复使用、一个页面重复出现多个<br>**预览性拆分**: 为了实现复杂变体的变换效果 | **复用性拆分**: 消除重复, "这个东西在多个地方用到了"<br>**结构性拆分**: “这个文件太长了, 每次定位问题滚动地很费劲”, 认知成本高 |
| **封装了有什么用?** | 类似场景可复用, 不用重复劳动<br>统一管理, 快速预览效果(布局&交互)<br>调试清楚后, 清晰地传递D2C指令 / 为代码 debug | 同类复用, 减少冗余代码, 避免迭代时漏改<br>模块拆分, 降低模块耦合度<br>便于维护和拓展<br>便于 debug 定位问题 |
| 好的体现 | 组件拆分与内嵌, 尽可能避免直接排列组合<br>便于定位和维护<br>足够小的原子组件有更好的通用性、可拓展性 | 模块拆分清晰, 便于定位代码、拓展、维护 |
| 坏的体现 | 按照组件特性的排列组合数量, 制作所有变体 | 代码臃肿, 难以定位和维护<br>所有短期需求只能补丁式改动, 重构耗时长且风险将越来越高 —— 代码架构带来的长期影响<br>展开图示: Good vs Bad<br>这就是为什么, 看起来长得一样的界面, <br>工程师更倾向于自己写, 而不是拼合设计师提供的页面级代码 |

**典型技术栈的前端文件结构**

更进一步, 你在设计稿里按 **视觉边界**拆组件 —— 这一块看起来是一个整体，就抽成一个组件。

但在代码的世界里, “万物皆可封装”:

- **常量与配置** —— 不会随运行时变化的值，统一抽出来集中管理。

> 比如 Design Token、中英文案、环境变量。改一处，全局生效；不抽出来，就会散落在各个文件里，改动时容易遗漏。

- **逻辑行为** —— 一段可以独立描述的业务逻辑，可以封装成函数或自定义 Hook。

> 比如"搜索防抖"、"滚动加载更多"、"权限判断"，这些逻辑和具体的界面无关，抽出来之后可以在多个组件里复用，也更容易单独测试。

封装的本质不是"减少代码量"，而是**给变化划定边界**—— 把"可能一起变"的东西放在一起，把"不会一起变"的东西隔离开。

从而确保一个系统的可持续维护和拓展, 支撑更复杂的业务。

**3. 组件做出来 ≠ 完成了 —— 契约思维**

设计组件和代码组件都有一个隐含的"契约"：**这个组件对外承诺了什么，对内封装了什么**。

**契约的三个维度：**

| **维度** | **设计侧** | **代码侧** |
| --- | --- | --- |
| **输入约束**（能传什么） | Property 的类型和可选值 | Props 的类型定义（TypeScript） |
| **边界状态**（极端情况） | 空状态、超长文本、超多数量 | 空数据、loading、error 状态的处理 |
| **行为约束**（怎么响应） | Hover/Active/Disabled 状态标注 | 事件回调（`onClick`, `onChange`）的定义 |

**无视契约会怎样：**

- 设计稿里的组件"看起来完整"，但导入真实数据后，各种边界情况没有设计，研发 / AI 只能自己猜；

- 代码组件没有类型约束，调用方传错参数，运行时才报错；

- 改一个地方，其他地方莫名其妙坏掉 —— 因为组件的边界不清晰，耦合太深。

#### 作用: 具备组件逻辑的「改动资质」

本模块不是在要求"一定要按这种方式建组件"，而是在指引一种“理解并判断复杂模块好不好, 该怎么优化”的思维方式，尤其是面对一个陌生的代码库时，能快速定位"这里改了会影响哪里"。

设计稿的组件结构能直接指导代码组件设计，和工程师对齐组件边界时有共同语言，用 AI 生成组件时能精准描述 props 和状态。

| Before: 看不到组件边界, 越改越坏 | After: 理解复用逻辑, 向可维护、可拓展的方向设计 |
| --- | --- |
| 设计稿：每个页面重新画，没有复用组件，研发也跟着复制代码 | **抽象出可复用组件，定义清楚 Variant 和 Property**；研发直接对应实现，一处改全局生效 |
| 改需求时："把所有页面的按钮颜色都改一下" —— **逐页修改，漏改，不一致** | **改主组件一处，所有实例自动更新**—— 设计侧和代码侧都是如此 |
| 提交后, 研发说改动了业务逻辑, 不方便采用 —— 因为**不知道改了影响其他地方的代码** | 理解组件的 Props 和边界后：**知道哪些是内部实现，哪些是对外接口**，改动有把握 |
| 给研发提需求："把这个卡片改一下，加个标签" <br>—— 研发："这个卡片在8个地方用了，你要改哪个？" | 提需求时：**明确是改主组件（影响所有实例）还是改某个页面的局部覆盖**，沟通零歧义 |

#### 练习: 拆解一个模块/页面的可复用组件

很多设计师做"组件"只是为了复制方便，这不是组件化思维。

真正的组件化是在问：这个东西的**稳定部分**是什么，**可变部分**是什么？稳定的做成结构，可变的做成参数（variant、size、state）。

一个 Button 有多少种状态，就说明你对这个组件的边界想得有多清楚。

关键心法

从“合并同类项”开始, 把重复出现的模块做成组件, 捋清页面的构成和调用关系

> 看到任何重复出现的元素，先问三个问题：
> 输入：有哪些 props/variant？
> 边界：空数据 / 极长文本 / 图缺失？
> 行为：hover/active/disabled 是否定义？
> 不变的是结构，变化的是参数，"谁来决定"是数据来源——这三个问题回答清楚，组件的边界就清楚了，无论是在设计稿还是在代码里。

**3) 共通点3: 系统行为**

很多设计师画稿只标注"正常态"，边界情况靠工程师自己猜。

但一个完整的交互设计必须回答：这个界面有哪些状态？什么事件触发状态转换？每个状态下用户能做什么、不能做什么？

——这就是有限状态机 (Finite State Machine) 的定义，你在画交互稿时其实一直在做这件事，只是没有意识到。

**对比: 状态、逻辑与边界情况**

交互设计本质上在定义: 一个系统在不同输入下的响应方式 —— 状态机、边界情况、信息流动路径

这和代码里的条件判断、状态管理、数据流向是同一套思维，只是交互设计师用流程图和标注表达，工程师用代码表达

| **设计** | **代码** |
| --- | --- |
| 状态标注（loading / empty / error / success） | 状态管理（useState / store） |
| 流程分支（如有权限 vs 无权限） | 条件判断（if / else） |
| 页面/模块跳转路径 | 路由逻辑（router） |
| 操作触发（点击、提交） | 事件处理（onClick / onSubmit） |
| 边界情况（网络断开、数据为空） | 异常处理（try/catch / fallback） |
| 用户操作流程图 | 业务逻辑函数 |

**作用:** **业务逻辑的「零歧义掌控」**

- 交互稿的状态覆盖越完整，代码里的 bug 越少。

- 用 AI 写逻辑时，能把交互稿直接翻译成精准的 Prompt。

- 看到代码里的状态管理逻辑，能直接用交互设计的语言去理解和 debug。

| 难度 | 场景 | 案例 |
| --- | --- | --- |
| 基线 | D2C | 状态完整，工程师 / AI 不需要猜边界情况，减少不合规范的自由发挥 / 返工 |
| 中线 | 自己改 | **自驱体验优化**: 能看懂代码里的状态管理逻辑，能用交互语言参与 debug |
| 高线 | AI 基建 | **设计知识 - 定义约束规则**: 能定义 AI 可稳定遵循的交互逻辑约束规则，作为 AI Coding 的上下文输入 |

**练习:**

> 看到任何交互场景，先问：**这里有几个状态？每个状态的触发条件是什么？状态之间能不能互相转换，转换的条件是什么？有没有我没考虑到的边界情况？**
> 把这些问题的答案写下来，就是一份完整的交互稿，也是一份可以直接喂给 AI 的逻辑描述。状态想得越清楚，代码就越少出 bug，AI 生成的结果就越接近你想要的。
