# 01 Ground 构建 Agent 的业务资产上下文


Ground 仓库怎么用

2026-05-06。这份不是规范文档，是跟你聊清楚两件事——**为什么我们决定 PDE 的 Ground 资料用 git 代码仓库**，以及**你怎么从零跑起来**。
看完你应该能：(1) 知道为啥，(2) 自己开一个仓库跑通一个真实需求。

---

起因—一份上下文资料放哪儿，组织怎么往 AI 范式进化

你今天接了个需求，比如「福利中心入口优化」。做的过程会产生一堆东西：

需求 intent（一句话：为谁、做什么）、边界 scope（什么做、什么不做）、历史PRD 文档、几版 设计稿
AB 实验设计、上线后的数据复盘

**这些资料信息怎么让 AI 知道，有了资料这些东西放哪儿？**

很多人没想过这个问题。但**资料放哪儿，决定了你的工作能不能复用、能不能给 AI 用、能不能跟同事共享**。

"CODE 不是只给程序员的吗？"

这是大多数人第一反应。**澄清一下**：

**CODE 仓库不是"只放代码"——它是"目录树 + 改动历史"的存储**。代码可以放，也可以放 markdown、HTML、图片、配置、数据。任何"会变化、需要协作、需要历史"的东西都适合。

PDE 的 Ground 资料就是这种东西：

业务术语（markdown）
PRD（markdown）
HTML 设计稿（HTML 文件）
数据基线（markdown 表格）
实验记录（markdown）

跟代码长得很像——都是文本文件、需要协作、需要版本。**用同一套工具，没毛病。**


---


仓库里的内容给谁看？

四类受众：

**AI Agent**——你让 Claude / Codex / 内部智能体读 ground 帮你写 PRD、出设计稿
**用 AI 协作的团队**—— AI 清楚每个人直接做了什么，当你要开是 XX 时，它会告诉你最近 xx 做了一个类似的，你可以和他交流
**人类**-按需让 AI 灵活提取你想要的，关键内容产出还是看学城文档

第 3 类**特别关键**——这是我们这一代 PDE 跟以前的核心差异。你工作时不再是"打开 doc 写"，而是"让 AI 拉 ground 一起写"。**AI 能不能干活，看你的 ground 长什么样、放哪儿。**

---

用之前先认 4 个词

git 概念多，但你先记 4 个动作就够了。**类比**：你跟同事一起改一份共享文件夹。

| 词 | 是啥 | 类比 |
| --- | --- | --- |
| `clone` | 把云端仓库下载到你电脑 | 把共享文件夹同步到本地（一次性） |
| `pull` | 拉别人的最新改动到本地 | 收一下同事的更新 |
| `commit` | 把你这次改的"登记"到本地历史 | 作业先写好（还没交） |
| `push` | 把本地的"登记"同步到云端 | 把作业交到老师那 |


口诀就一句：**改之前 pull，改完了 commit + push**。

或者：

每天上班 pull 一次、下班 push 一次，差不多就够了。现在也可以直接用 skill，说同步到仓库，更新下仓库即可。

第一步：先开你自己的测试仓库/建团队仓库

**别急着拉同事。先自己一个人跑通一个需求**——这是最重要的一步。

具体怎么做：

开始之前 · 装好这几样（10 分钟，一次性）

| 工具 | 干什么用 | 怎么装 |
| --- | --- | --- |
| **Claude Code + CatPaw CLI**（中转代理） | 跟 AI 对话的客户端，后面所有"跟 AI 说一句话"都在这里说 | [《安装 Claude Code 并使用 CatPaw CLI 中转代理》](https://km.sankuai.com/collabpage/2751430003) |

| `**pde-init**`** + **`**pde-ground**`** skill** | 建仓库 + 日常用要的两个 skill |  |


这三样装完才进下面三步。CatPaw Desk + 学城 skill 默认你已经有，没装的不在本文讨论范围。

---


关于这两个 skill · 到底什么时候用哪个

`pde-init` 和 `pde-ground` 干的事**完全不同，不冲突，不会互相替代**。装一次都装上，AI 听你说什么自己挑对的——你不用记 skill 名字。

| skill | 什么时候用 | 一辈子用几次 | 你跟 AI 说什么会触发它 |
| --- | --- | --- | --- |
| `**pde-init**` | 你这个业务**还没有 Ground 仓库**，要从零建一个 | **每个业务只用一次**（建完就退场） | "我要建 PDE 仓库" / "帮我搞个 XX 业务的 Ground" / "新业务起手" |
| `**pde-ground**` | 仓库**已经建好了**，每天进去干活 | **每天用** | "同步一下" / "我改完了" / "我想做 XX 需求" / "复盘 YY" / "广场 DAU 多少" |


**类比**：`pde-init` 是装修队（一次性，装完就走），`pde-ground` 是日常用的管家（每天叫）。


**所以你的路径是**：

装两个 skill（一次）
第一次接触这套 → 走 `pde-init`，建仓库（一次）
此后每天工作 → 走 `pde-ground`（无数次）

下面"三步走"严格按这个顺序——**第一步对应 **`**pde-init**`**，第二步以后都是 **`**pde-ground**`。

---


三步走

第一步 · 建你自己的仓库（用 `/pde-init`，一次性 30 分钟）

去 [https://dev.sankuai.com/code](https://dev.sankuai.com/code) 登录 → 右上角"创建仓库" → 命名 `~你的MIS/{业务}pde`，比如 `~zhangsan/livesquare-pde`。

命名建议：业务英文名 + `pde` 后缀。这样以后第二、三个业务的 ground 仓库一眼能认出。

打开 Claude Code，说"我要建 PDE 仓库" →发仓库链接给它  → skill `/pde-init` 全自动：命名、抄 mlivepde 骨架、改业务名、写你的偏好、第一次 push。你不打任何 git 命令——它替你打。

**资料从哪来？两条路（可混用）**

| 路径 | 适合什么场景 | 怎么做 |
| --- | --- | --- |
| **A. AI 从学城搜罗** | 你业务在学城有历史沉淀（PRD / 业务概述 / 数据复盘 / 组织架构） | 跟 AI 说"把 XX 业务的核心 PRD、数据、组织资料从学城拉到 ground/"——CatPaw Desk + 学城 skill 搜、拉、按目录归档一条龙 |
| **B. 自己整理 md 让 AI 注入** | 资料散在本地、OneDrive、笔记、聊天记录里 | 把原始资料堆到一个文件夹（粘贴成 .md 或导出都行），跟 AI 说"按 ground/ 四类目录帮我归档这些"——AI 读完分类整理 |


通常**先 A 把学城历史一锅端，再 B 补本地草稿和私下决策**。

2 · 日常用（用 `/pde-ground`，每天 5 分钟）

cd 到仓库，跟 AI 说人话："同步一下" / "我想做 XX 需求" / "复盘上次 YY" / "广场 DAU 多少" —— skill `/pde-ground` 帮你做 git 操作 + 按 AGENTS.md 的 7 类任务读对的资料。不装 skill 也能用，4 个 git 命令（pull / commit / push，偶尔 clone）足够。

3 · 梳理业务资产（继续用 `/pde-ground`，长期，真正花脑子的地方）

mlivepde 给了一份骨架可看（`ground/` 下分 domain / playbook / metrics / spec 四类），**但这只是"长什么样"的参考，不是模板**。

**关键原则**：

**强烈建议按你自己的业务信息和协作习惯，设计你自己的目录结构和索引**——你团队怎么分类知识、谁会经常翻什么、什么文件叫什么名，完全可以跟 mlivepde 不一样。直播业务的资产组织未必适合你
`**INDEX.md**`** 必须自己写**——它是你这个仓库的资产索引，直接反映你业务的资产长什么样；mlivepde 的 INDEX 不能挪用
**内容必须自己结合业务梳理**——mlivepde 的术语表全是直播术语，组织文件是直播 PM 名单，跟你业务和团队没关系。照抄等于没建
**第一版别求全**——填 3-5 个最核心术语 + 一份当前数据基线就够开工。剩下边用边长：每跑完一个任务发现 ground 缺啥，按 AGENTS.md §3 回流一次

这一步 AI 帮不了你写底层事实（它不知道你的业务），但能帮你**整理结构、对照模板补缺、检查新鲜度**。

后续怎么用

日常工作流

跑稳之后，日常就是这套：

| 时机 | 做什么 |
| --- | --- |
| 每天上班 | `git pull` 一次（拿同事昨晚的更新） |
| 改完一段东西 | `git commit -m "说清做了啥"` |
| 下班 / 中午 | `git push` 一次 |
| 来一个新任务 | 直接说，AI 按 7 类路由跑——大多不用人工选 |
| 开新需求 | 复制 `_template`，目录名带日期 |
| 改公共 ground/* | 群里说一声，避免别人也在改 |
| 上线后 1-4 周 | 填 `06-effect.md`，按 §3 反馈回流约定决定回流到哪 |
| ground/* 看着过期 | 检查 frontmatter 的 `last_updated`，刷新 |


7 类任务的日常对照（每类一句话）

| 任务 | 你常说的话 | AI 自动做 |
| --- | --- | --- |
| A 咨询 | "X 现在啥情况"、"Y 是什么" | 读 ground 直接答 |
| B 分析 | "分析 X"、"看下 Y 的数据" | 读 metrics + experience 输出 |
| C 找人 | "X 是谁"、"找 Y 的负责人" | 读 organization |
| D 汇报 | "做个 X 介绍"、"对外讲讲 Y" | 读 ground 全套输出文档 |
| E 复盘 | "复盘 X"、"总结 Y 项目" | 读 projects/已结 + decisions |
| F 防撞 | "有人在做 X 吗" | 列 projects/在推 |
| G 新需求 | "我想做 X"、"加个 Y 功能" | **反问你偏好** |


模糊时 AI 会反问"我识别为任务 X，对吗"——你确认一下就行。

ground 反馈回流

任何任务跑完都可能更新 ground/：

**显著正向效果** → 提炼新 pattern 到 `ground/playbook/experience.md`
**关键指标变化** → 更新 `ground/metrics/baseline.md`
**结构性决策** → 追加 `ground/playbook/decisions.md`
**重大变更**（仓库结构 / 规约 / 域知识大调）→ 写到 `ground/changelog.md`

回流后必须更新对应文件 frontmatter 的 `last_updated`。

**这是仓库越用越准的关键**——不回流就只是个静态资料库。

跟业务代码仓库的关系

很多人会问："我的代码也放这儿吗？" **不放。**

一个 PDE 工作场景里有**两个仓库**：

**Ground 仓库**（本仓库）：需求、PRD、设计稿、决策、经验、基线
**业务代码仓库**：代码、接口、数据库、构建脚本

两者**互相引用**，不合并：

Ground 仓库的 PRD 里链接代码仓库 PR：

```
实现链接：https://dev.sankuai.com/code/business-repo/pr/123

```
代码仓库 commit 信息里写需求 ID：

```
福利中心改造 (mlivepde:2026-04-08-福利中心入口优化)

```

**为什么要分开**：

Ground 不绑业务代码迭代——它是跨需求的长期资产，业务代码每个版本都在变
工程师天天 push 业务代码，你 push 几次 ground——分开后互不打扰
安全考虑：Ground 里有 PRD、设计稿，可能含商业机密——可放个人 namespace（`~你的MIS`）；业务代码归团队 namespace，权限不一样

推开到组内

拉 1 个同事一起跑

跑通自己之后，**别群发**。找你身边最愿意尝鲜的同事——可能是另一个 PM，可能是设计同学。

跟他说：

"我搞了个仓库，专门放需求资料的。你试试用，看顺不顺手——不顺手我们改。"

给他 clone 链接，让他选一个手头真实需求跑。两人项目互不干扰（每人自己 `projects/{需求}/` 目录），不用担心打架。

跑 1-2 周后两人复盘：

多人提交有没有冲突
AI 读 ground 是不是真省事
哪些规约要改

这一阶段叫"小组试点"。**重点不是用得多顺，而是发现哪些坑。**

慢慢推开到组内

小组试点跑通后，再往组内推。要做几件事：

**半小时 git 培训** — 基于仓库里的 [HOW-TO-USE.md](http://HOW-TO-USE.md) 给 PM / 设计同学讲基本 git 操作。**别让他们自己看文档（看不进去）。**
**起手命令一键化** — 写个 `bin/new-project.sh` 之类，一行命令开新需求。避免每个人都记 `cp -r projects/_template ...`。
**每周组内 review 一次回流** — 把上线的需求 [06-effect.md](http://06-effect.md) 拉出来过一遍，决定哪些要回写到 ground/* 沉淀。
**每个新人加自己的 PDE 偏好段** — [AGENTS.md](http://AGENTS.md) §4 是自治演进的入口；每个人写自己的工具/命名/语言偏好，互不干扰。

**核心原则**：不要"先写完美方案再推"。规约本来就是边用边改——每周用、每周改。

---

几条不要做的事

| 不要做 | 后果 |
| --- | --- |
| `git push --force` | 别人的工作被你覆盖，找不回 |
| `git reset --hard` | 你本地未保存的修改全没 |
| 删除别人的 `projects/{需求}/` 目录 | 删了就删了，找工程师恢复 |
| 在仓库存代码 / 大文件 / 视频 | 仓库会变臃肿，clone 越来越慢 |
| commit 信息只写"修改" | 三个月后查历史看不懂自己改了啥 |
| 在 [AGENTS.md](http://AGENTS.md) 里写死具体 skill / 工具名 | 违反元思维（地基不挑工具）；明天工具改名 [AGENTS.md](http://AGENTS.md) 也要改 |

几个常见问题

**Q: 我完全不会 git，这套用得起来吗？**
A: 你不需要"会 git"，你需要会 4 个命令（pull / commit / push / 偶尔 clone）。GUI 工具（VSCode 自带 git 面板 / SourceTree / GitHub Desktop）一键搞定。

**Q: git 命令出错了怎么办？**
A: 截图找你组里工程师，10 秒钟解决。**不要自己乱试 --force --hard 这些。**

**Q: 同事跟我同时改了同一个文件冲突了怎么办？**
A: 一般不会冲突——大家各自改自己的 `projects/{需求}/` 目录。只有改公共 `ground/*` 才可能冲突，git 大多数情况能自动合并；真冲突找工程师 5 秒解决。

**Q: AI 读这个仓库要怎么搞？**
A: 直接告诉 Claude / Codex / 内部智能体仓库地址或者 cd 到本地 clone 的位置，它们会自动读 [AGENTS.md](http://AGENTS.md)。**不需要你做任何集成。**

**Q: AGENTS.md v0.2.0 怎么知道我用什么工具走"做新需求"？**
A: 不知道——它会反问你（任务 G 的标准动作）。你回答 A/B/C 任一个；或者直接说"用 Forge skill"、"用 Codex"、"按我的 SOP"。AI 跟着走。第一次回答完之后，可以把答案写到 §4 PDE 偏好段，下次自动按这个走。

**Q: 我能不能改 AGENTS.md 里的任务路由？**
A: 当然能——这就是 §8 节"反馈回流到本 [AGENTS.md](http://AGENTS.md)"的意思。规约跟着实践跑。改完 commit 即可。

**Q: PDE 偏好段是干嘛的？**
A: 让你写自己的工具偏好，避免 AI 每次都反问。比如你定 `default_task_g_approach: 用 Forge skill`，下次做新需求时 AI 直接调 Forge 不再反问。每个 PDE 自己一段，互不影响。

**Q: 我能不能不用 mlivepde 骨架，自己设计目录？**
A: 当然可以。骨架只是参考——**你跑得顺更重要**。但建议先用一段时间再改，避免提前优化。

**Q: 仓库里能放图吗？**
A: 小图（设计稿截图、流程图）OK，单张控制在 500KB 内。视频 / 大图 / 多张高清图走 OSS 或 CDN 链接。

**Q: 我没装 Forge skill 也能用这个仓库吗？**
A: **完全可以**。[AGENTS.md](http://AGENTS.md) v0.2.0 不依赖任何客户端 skill。任务 G 反问你时你说"按通用引导走"，AI 不调任何外部 skill 也能跟你协作。

**Q: 我有了 Ground 仓库，还需要装 Forge skill 吗？**
A: 不必。Forge skill 是给"还没建仓库"的场景兜底——你已经有仓库了，[AGENTS.md](http://AGENTS.md) 就够。只在你想跑一次性 demo / 跨业务实验时才用 Forge。
